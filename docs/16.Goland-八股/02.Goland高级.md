## Sync面试题

### 三色标记法是什么？

三色标记法是Go垃圾回收器使用的核心算法

**三色定义**：

- **白色**：未被访问的对象，垃圾回收结束后白色对象会被清理
- **灰色**：已被访问但其引用对象还未完全扫描的对象，是待处理队列
- **黑色**：已被访问且其所有引用对象都已扫描完成的对象，确认存活

**标记流程**：GC开始时所有对象都是白色，从GC Root（全局变量、栈变量等）开始将直接可达对象标记为灰色。然后不断从灰色队列中取出对象，扫描其引用的对象：如果引用对象是白色就标记为灰色，当前对象所有引用扫描完成后标记为黑色。重复这个过程直到灰色队列为空。

### Go 语言的 GC 使用的是什么？

Go 的 GC 目前使用的是无分代（对象没有代际之分）、不整理（回收过程中不对对象进行移动与整理）、并发（与用户代码并发执行）的三色标记清扫算法。

### Go语言GC的根对象到底是什么？

根对象在垃圾回收的术语中又叫做根集合，它是垃圾回收器在标记过程时最先检查的对象，包括：

1. 全局变量：程序在编译期就能确定的那些存在于程序整个生命周期的变量。
2. 执行栈：每个 goroutine 都包含自己的执行栈，这些执行栈上包含栈上的变量及指向分配的堆内存区块的指针。
3. 寄存器：寄存器的值可能表示一个指针，参与计算的这些指针可能指向某些赋值器分配的堆内存区块。









### 1.如果有一个全局变量怎么保证并发安全？

:::note

1. 使用`sync.Mutex`互斥锁，通过加锁保证同一时间只有一个 goroutine 可以修改变量。
2. 使用 `sync.RWMutex` 读写锁，在读多写少的场景下可以提高并发性能。
3. 使用 `atomic` 提供的原子操作，适用于计数器等简单变量操作。
4. 通过 `channel` 让一个 goroutine 专门管理这个变量，其他 goroutine 通过 channel 与它通信。

实际开发中最常用的是 `Mutex` 或 `atomic`。

:::

### 2.Go 语言是如何实现原子操作的？

:::note

Go 语言通过 sync/atomic 包实现原子操作。

原子操作是指一个操作在并发环境下不可被中断，要么全部执行，要么完全不执行。

Go 的 atomic 底层依赖 **CPU 提供的原子指令**，比如 x86 的 LOCK 指令，并通过 CAS，也就是 Compare And Swap 算法实现。

CAS 的核心思想是先比较当前值和期望值，如果相同就更新为新值，否则操作失败。相比 Mutex，atomic 不需要加锁和 goroutine 阻塞，因此性能更高，但它只适用于简单变量操作，比如计数器或状态标志。

:::

### 3.原子操作和锁的区别？

- 实现方式不同：原子操作通过**CPU 提供的原子指令**实现；**锁**是通过 **操作系统调度 + 线程阻塞机制** 实现的。
- 性能区别：原子操作不会阻塞，开销小并发性能高；锁可能会阻塞，开销较大并发性能相对低。
- 使用场景：原子操作适合简单变量操作比如计数器或状态标志；锁适合复杂数据结构或者多个变量需要同时修改的场景。

:::note

原子操作和锁都是用来保证并发安全的，但实现方式不同。

原子操作是通过 CPU 提供的原子指令实现的，比如 CAS 操作，它不会发生 goroutine 阻塞，因此性能比较高。

而锁是通过互斥机制保证同一时间只有一个 goroutine 访问共享资源，如果获取不到锁，goroutine 可能会被阻塞并进入调度。

在实际使用中，原子操作适合简单变量操作，比如计数器或状态标志；而锁适合复杂数据结构或者多个变量需要同时修改的场景。

:::

### 4.互斥锁mutex底层是怎么实现的？

:::note

Go 的 Mutex 底层主要由两个字段组成：`state` 和 `sema`。

state 用来表示锁的状态，比如是否已经被锁定以及等待的 goroutine 数量；sema 是一个信号量，用于实现 goroutine 的阻塞和唤醒。

当调用 `Lock` 时，Go 会先通过 CAS 尝试获取锁，如果成功就直接进入临界区；如果失败，会先进行几次自旋尝试获取锁，如果仍然失败，就会进入等待队列并阻塞。

当调用 `Unlock` 时，如果有 goroutine 在等待，runtime 会通过信号量唤醒其中一个 goroutine。

此外 Go 的 Mutex 还有正常模式和饥饿模式，当 goroutine 等待时间过长时会进入饥饿模式，从而避免锁饥饿问题。

:::

### 5.Mutex 有几种模式？

:::note

Go的`Mutex`主要有两种模式：**正常模式和饥饿模式**。

默认是**正常模式**，讲究的是性能。新请求锁的goroutine会和等待队列头部的goroutine竞争，新来的goroutine有几次"自旋"的机会，如果在此期间锁被释放，它就可以直接抢到锁。

这种方式吞吐量高，但可能会导致队列头部的goroutine等待很久，即"不公平"。

当一个goroutine在等待队列中等待超过`1ms`后，Mutex 会进入**饥饿模式**，讲究的是公平。锁的所有权会直接从解锁的goroutine移交给等待队列的头部，新来的goroutine不会自旋，必须排到队尾。这样可以确保队列中的等待者不会被"饿死"。

当等待队列为空，或者一个goroutine拿到锁时发现它的等待时间小于1ms，饥饿模式就会结束，切换回正常模式。

:::

### 6.在Mutex上自旋的goroutine 会占用太多资源吗

:::note

并不会占用太多资源，因为自旋是有严格限制的。

Go runtime 只有在多核 CPU、锁竞争不激烈以及锁可能很快释放的情况下才会允许 goroutine 自旋，而且自旋次数是有限的，一般只有几次。

如果自旋之后仍然无法获取锁，goroutine 就会进入阻塞状态并等待调度器唤醒。

:::

### 7.Go 的 Mutex 为什么会有“饥饿模式”？

:::note

Go 的 Mutex 引入饥饿模式是为了解决锁饥饿问题。在普通模式下，新来的 goroutine 可以直接尝试抢锁，这样虽然性能比较高，但可能导致等待队列中的 goroutine 长时间得不到锁。

为了解决这个问题，引入了“饥饿模式”，当一个 goroutine 等待锁超过 1ms 时，Mutex 就会进入饥饿模式，在这种模式下锁会直接交给等待队列中的 goroutine，而新的 goroutine 不能再抢锁。当等待队列减少或者竞争降低时，Mutex 又会回到 normal mode，从而在公平性和性能之间取得平衡。

:::

### 8.Mutex 已经被一个 Goroutine 获取了, 其它等待中的 Goroutine 们只能一直等待。那么等这个锁释放后，等待中的 Goroutine 中哪一个会优先获取 Mutex 呢?

:::note

取决于Mutex当前处于正常模式还是饥饿模式。

在**正常模式**下，锁的分配是"不公平"的。锁释放后会唤醒一个等待的 goroutine，但它不会直接获得锁，而是和新来的 goroutine 一起竞争锁，因此新来的 goroutine 可能抢到锁。

在**饥饿模式**下，锁的分配是"绝对公平"的。锁会直接交给等待队列头部的 goroutine，任何新来的goroutine都不会参与竞争。

:::

### 9.sync.Once 的作用是什么，讲讲它的底层实现原理？

:::note

`sync.Once`的作用是：在多个 goroutine 并发执行时，确保某个函数只执行一次。

它常用于单例对象的初始化或一些只需要执行一次的全局配置加载。

它的底层实现主要由一个 done 标志位和一个 Mutex 组成。调用 Do() 时会先通过 atomic.Load 判断 done 是否为 1，如果已经执行过就直接返回，这是一个快速路径。

如果 done 为 0，就进入慢路径，通过加锁保证只有一个 goroutine 执行函数，同时再做一次 done 检查，这是双重检查机制。函数执行完成后会通过 atomic.Store 将 done 设置为 1。

这样设计的好处是第一次执行需要加锁，而后续调用只需要一次原子读取，性能非常高。

:::

### 10.WaiGroup 是怎样实现协程等待？

:::note

WaitGroup 的作用是等待一组 goroutine 执行完成。它的底层实现主要依赖原子操作和 runtime 的信号量机制。

WaitGroup 内部维护一个计数器 counter 表示未完成的任务数量，还有一个 waiters 记录正在等待的 goroutine 数量，以及一个信号量 sema。

调用 Add() 时通过 atomic 操作增加计数器，Done() 实际上是执行 Add(-1)。当 counter 递减到 0 时，runtime 会通过信号量唤醒所有在 Wait() 中阻塞的 goroutine。

Wait() 的实现是先检查 counter 是否为 0，如果不为 0，就调用 runtime_Semacquire 进入阻塞状态，直到 Done() 将计数器减为 0 后通过 runtime_Semrelease 唤醒。

:::

### 11.sync.Map的底层原理？

:::note

sync.Map底层是由两个Map（`read`和`dirty`）和一个**命中计数器**组成。

`read map`是一个只读的`map`，提供无锁的并发读取。`dirty map`可读可写，读操作不加锁，写操作加锁。

读操作优先访问`read map`，无锁快速返回，如果未命中则访问 `dirty map`；写操作会加锁更新 `dirty map`，并在必要时将 `dirty map` 合并到 `read map`，通过原子操作更新可见性。

:::

### 12.sync.Map 适用的场景？

:::note

`sync.Map`适合读多写少的高并发场景。

:::

### 13.read map和dirty map之间有什么关联？

:::note

`read map` 是 `dirty map` 的快照。它只包含 **过去的 key**，并且是**不可变的**。

`dirty map` 是实际可写数据，包含所有的最新数据，新增或更新的 key 写入`dirty map`，积累到一定程度后，`dirty map`里的数据"晋升"并覆盖掉旧的`read` map。

:::

### 14.为什么要设计nil和expunged两种删除状态？

:::note

目的是解决**在`sync.Map`的"读写分离"架构下，如何高效、无锁地处理"删除"操作**。

当 key 被删除时，先标记为 nil，表示该 key 无效但可能被后续写入复活；如果 key 永久不再使用，则标记为 expunged，表示彻底删除，可在 read map 合并时直接忽略。

:::

## GMP面试题

### 1.什么是协程（Goroutine）？

:::note

协程是用户态轻量级线程，它是线程调度的基本单位。通常在函数前加上`go`关键字就能实现并发。一个Goroutine会以一个很小的栈启动2KB或4KB，当遇到栈空间不足时，栈会自动伸缩， 因此可以轻易实现成千上万个goroutine同时启动。

:::

### 2.进程、线程、协程有什么区别？

:::note

- 进程:进程是具有一定独立功能的程序，进程是系统资源分配和调度的最小单位。 每个进程都有自己的独立内存空间，不同进程通过进程间通信来通信。由于进程比较重量，占据独立的内存，所以上下文进程间的切换开销（栈、寄存器、虚拟内存、文件句柄等）比较大，但相对比较稳定安全。
- 线程:线程是进程的一个实体,线程是内核态,而且是 CPU 调度和分派的基本单位,它是比进程更小的能独立运行的基本单位。线程间通信主要通过共享内存，上下文切换很快，资源开销较少，但相比进程不够稳定容易丢失数据。
- 协程:协程是一种用户态的轻量级线程，协程的调度完全是由用户来控制的。协程拥有自己的寄存器上下文和栈。 协程调度切换时，将寄存器上下文和栈保存到其他地方，在切回来的时候，恢复先前保存的寄存器上下文和栈，直接操作栈则基本没有内核切换的开销，可以不加锁的访问全局变量，所以上下文的切换非常快。

:::

### 3.什么是GMP模型？

:::note

GMP是Go运行时的核心调度模型。G是goroutine协程；M是machine操作系统线程，负责实际执行 G；P是processor，逻辑处理器，它是G和M之间的桥梁。它负责调度G。

三者的关系：Goroutine 不能直接运行在 M 上，必须通过 P 调度。

调度逻辑是这样的，M必须绑定P才能执行G。每个P维护一个自己的本地G队列（长度256），M从P的本地队列取G执行。当本地队列空时，M会按优先级从**全局队列、网络轮询器、其他P队列**中窃取goroutine。

:::

### M寻找可运行G的过程是怎样的？

![image-20260324134452373](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260324134457919.png)

:::note

**M会优先检查本地队列（LRQ）**：从当前P的LRQ里`runqget`一个G。（无锁CAS），如果本地队列没有可运行G，**再次检查全局队列（GRQ）**去全局队列里`globrunqget`找。（需要加锁）；如果还没有，就**检查网络轮询器（netpoll），**就去`netpoll`里看看有没有因为网络IO就绪的G。（非阻塞模式），依然没有获取到可运行G，则会**从别的P偷（steal work），这个偷的过程是**随机找一个别的P，从它的LRQ里偷一半的G过来。

:::

### P和M在什么时候会被创建？

:::note

**P的创建时机**：P在调度器初始化时**一次性创建**。在`schedinit()`函数中会调用`procresize()`，根据`GOMAXPROCS`值创建对应数量的P对象，存储在全局的`allp`数组中。之后P的数量基本固定，只有在调用`runtime.GOMAXPROCS()`动态调整时才会重新分配P。

**M的创建时机**：M采用**按需创建**策略。初始只有m0存在，当出现以下情况时会创建新的M：

- 所有现有M都在执行阻塞的系统调用，但还有可运行的goroutine需要执行
- 通过`startm()`函数发现没有空闲M可以绑定P执行goroutine
- M的数量受`GOMAXTHREADS`限制，默认10000个

**创建流程**：新M通过`newm()`函数创建，它会调用`newosproc()`创建新的系统线程，并为这个M分配独立的g0。创建完成后，新M会进入`mstart()`开始调度循环。

:::

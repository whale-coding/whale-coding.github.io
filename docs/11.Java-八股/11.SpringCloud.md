## 分布式理论

### ⭐️1.什么是CAP定理？

:::note 参考答案

CAP定理：在分布式系统中不可能同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition tolerance）这三个特性，最多只能同时满足其中两个。

- **一致性**：指的是所有节点在同一时间看到的数据都是一致的。
- **可用性**：指的是系统在正常响应时间内返回合理的响应，即使部分节点出现故障，系统仍然可以继续提供服务。
- **分区容错性**：指的是当网络分区故障发生时，系统仍然能够继续运行。网络分区是分布式系统中不可避免的问题，比如机房之间的网络中断，这时候不同机房的节点就无法正常通信了。

对于分布式系统来说，P 是必须要有的，因为网络既然会存在分区问题（网络延迟、丢包、中断等），分区容错性也就成为了必然的选择。因此，对于分布式系统来说，要么是CP，要么是AP。

:::

>扩展知识点1：什么是网络分区？

分布式系统中，多个节点之间的网络本来是连通的，但是因为某些故障（比如部分节点网络出了问题）某些节点之间不连通了，整个网络就分成了几块区域，这就叫**网络分区**。

### 2.为什么 CAP 不可兼得？

:::note 参考答案

对于分布式系统来说，分区是必然存在的。网络分区指的是在分布式系统中，多个节点之间的网络本来是连通的，但是因为某些故障（比如部分节点网络出了问题）某些节点之间不连通了，整个网络就分成了几块区域。

因此，分区容错性（P）必须要满足，因为如果要牺牲分区容错性，就得把服务和资源放到一个机器，或者一个“同生共死”的集群，那就违背了分布式的初衷。

在分区容错的前提下，`一致性`和`可用性`是矛盾的，只能满足其一。

:::

### 3.什么是 BASE 理论？

:::note 参考答案

BASE（Basically Available、Soft state、Eventual consistency）理论是对CAP定理中 AP 方案的延伸，核心思想是即便不能达到强一致性，也可以根据应用特点采用适当的方式来达到最终一致性的效果。

BASE 的主要含义：

- **Basically Available（基本可用）**：系统出现故障时，允许损失部分可用性（如响应时间增加、功能降级）。
- **Soft State（软状态）**：允许系统存在中间状态，该状态不影响整体可用性。即允许系统在多个不同节点的数据副本存在数据延时。
- **Eventually Consistent（最终一致性）**：系统保证在一段时间内达到数据的最终一致。

:::

## 分布式事务

### 1.什么是分布式事务？

:::note 参考答案

分布式事务，本质上就是为了解决在微服务架构下，因业务操作跨越了多个独立的服务和数据库，而无法使用本地事务来保证数据一致性的问题。



分布式事务是指在分布式系统环境下，一次业务操作涉及多个服务、多个数据库节点或多个资源服务器，这些操作需要共同保证数据的一致性，即“要么全部成功，要么全部失败”。它解决了微服务架构下跨服务调用导致本地事务失效的问题。

:::

### 2PC的缺点？

- **同步阻塞**：所有参与者在prepare后锁定资源，直到收到commit/rollback（高并发下吞吐量骤降）
- **单点故障**：协调者宕机导致参与者永久阻塞
- **数据不一致**：网络分区时部分参与者可能提交成功

首先 2PC 是一个**同步阻塞协议**，像第一阶段协调者会等待所有参与者响应才会进行下一步操作，当然第一阶段的**协调者有超时机制**，假设因为网络原因没有收到某参与者的响应或某参与者挂了，那么超时后就会判断事务失败，向所有参与者发送回滚命令。

2PC（Two-phase commit protocol），中文叫二阶段提交。 **二阶段提交是一种强一致性设计**，2PC 引入一个事务协调者的角色来协调管理各参与者（也可称之为各本地资源）的提交和回滚，二阶段分别指的是准备（投票）和提交两个阶段。







### TCC？

该方案是最终一致性方案。

它是业务层面的2PC。

TCC将业务逻辑拆分为三个阶段：

- **Try**：预留资源（如冻结库存）
- **Confirm**：确认操作（正式扣减库存）
- **Cancel**：释放资源（解冻库存）

**执行流程**：

1. 主业务调用所有服务的try方法
2. 全部try成功则调用confirm；任一try失败则调用cancel

**优势**：

- **无全局锁**：只在try阶段锁定局部资源
- **高可用**：协调者可集群部署

二、工作原理：两阶段提交

AT模式将事务分为两个阶段：

- **一阶段（准备阶段）**：
  1. RM解析业务SQL，并在同数据库中生成该操作的**Before Image**（修改前快照）。
  2. 执行业务SQL。
  3. 生成**After Image**（修改后快照）。
  4. 将Before/After Image及业务SQL封装为`undo_log`，在本地事务中插入到`undo_log`表。
  5. 本地事务提交，释放数据库锁。
- **二阶段（提交或回滚）**：
  - **提交**：如果全局事务成功，TC会异步删除对应分支事务的`undo_log`。
  - **回滚**：如果全局事务失败，RM通过`undo_log`计算出逆向SQL（依据Before Image）并执行，从而将数据恢复到原样。

三、技术要点

- **非侵入式**：用户只需使用`@GlobalTransactional`注解，Seata通过数据源代理（DataSourceProxy）解析SQL，无需手动编写补偿逻辑。
- **读写隔离**：本地事务使用读已提交（Read Committed），而全局事务在执行完一阶段至二阶段提交前处于读未提交状态，Seata通过全局锁来维护原子性，写操作需检查全局锁。
- **数据一致性**：核心原理是“本地事务保证阶段一成功，用Undo Log保证阶段二可逆”。

### 1.谈谈你对面向对象的理解?

:::note 参考答案：

面向对象是一种编程思想，它通过把现实世界中的事物抽象成“类”，对象是类的具体实例。另一种思想是面向过程，面向过程以“步骤和过程”为核心，通过函数完成任务。面向对象是以对象为核心，通过对象交互完成任务。

拿洗衣机洗衣服为例，面向过程的做法是一步一步执行操作，比如“打开洗衣机->放衣服->加洗衣粉->启动清洗->烘干”；面相对象则关注“人”和“洗衣机”两个对象，人的职责是打开洗衣机、放衣服、加洗衣粉，洗衣机的职责是清洗和烘干。

从这个例子可以看出来，面向过程更加的直接和高效，而面向对象更加易于扩展、维护和复用。

面向对象有**封装、继承和多态**三大特性。

- **封装**是指可以利用权限修饰符把一个对象的属性私有化，同时可以提供getter、setter方法用来被外界访问，好处是可以保护对象的属性。
- **继承**是指子类可以继承父类的属性和方法。好处是可以提高代码的复用性，java中类只支持单继承。

- **多态**是指同一个方法在不同对象上可以表现出不同的行为，主要通过方法的重写实现。<span class="highlighted">多态的前置条件是子类继承父类、子类重写父类的方法、父类引用指向子类的对象。多态在编译时看左边，运行时看右边。</span>

总的来说，面向对象是一种编程思想，这种思想有利于系统的扩展、维护和复用，是现代主流编程语言的核心思想。

:::

### ⭐️2.接口和抽象类的区别?

:::note 参考答案：

- 两者设计目的不同，接口是行为的抽象，定义一种行为规范；抽象类是对类的抽象，是一种模版设计；
- 接口可以多实现和多继承，抽象类只能单继承；
- 接口中的成员变量只能是`public static final`类型的，不能被修改且必须有初始值；抽象类的成员变量可以有任意修饰符，可以在子类中被重新定义或者赋值；
- 抽象类可以有抽象方法和非抽象方法；接口在Java 8以前只能有抽象方法，从Java8开始可以有`default`方法和`static`方法，从Java9开始可以包含`private`方法；
- 抽象类可以有构造方法和静态代码块；接口不能有构造方法，也不能有静态代码块。

:::

>扩展1：抽象类可以实现一个接口吗？

:::danger 参考答案：

抽象类可以实现一个或多个接口。抽象类在实现接口时，可以选择性地重写接口中的抽象方法，甚至不重写任何方法，而是将这些方法的实现推迟到具体的非抽象子类中。 

:::

### 3.Java 中 final 的作用是什么？

:::note 参考答案：

 final 关键字可以修饰类、方法和变量：

- 当final修饰一个类时，这个类不能被继承，比如String 类；
- 当final修饰一个方法时，这个方法不能被重写；
- 当 final 修饰一个变量时，这个变量的值一旦被初始化就不能被修改。

:::

>扩展1：final修饰引用类型的变量时，引用变量可以修改吗？

```java
final Person p = new Person("Alice");
p.setName("Bob"); // 合法，可以修改对象的内部状态
p = new Person("Charlie"); // 编译错误，不能修改引用本身

final List<String> list = new ArrayList<>();
list.add("hello");  // 合法，可以往里面添加元素
list = new ArrayList<>();  // 编译错误，不能修改引用本身
```

:::danger 参考答案：

引用变量本身，也就是引用地址不能修改，一旦指向某个对象，不能再指向其他对象。

可以修改对象内部的状态，即对象的内容、属性完全可以修改。

:::

> 扩展2：Java中abstract和final能一起用吗？

:::danger 参考答案：

在Java中，`abstract`和`final`不能一起使用。它们的作用完全相反，逻辑上是冲突的。

`abstract`目的是被继承和实现（子类重写），而`final`目的是禁止继承或重写。

:::

### ⭐️4.重载和重写的区别？

:::note 参考答案：

- 重载是发生在同一个类中，**方法名相同**，但参数列表不同（个数、顺序或类型不同）；
- 重写是子类对**继承自父类的方法进行重新实现**，方法名、参数列表**必须完全相同**；
- 重载方法的返回值类型和访问修饰符可以不同；重写时，子类方法的返回值类型和抛出的异常要小于等于父类方法的返回值类型和异常，子类方法的访问权限要大于等于父类方法的访问权限。

:::

> 扩展1：参数列表不同，返回值不同是重载吗？

:::danger 参考答案：

重载是发生在同一个类中，**方法名相同，但参数列表不同**。<span class="highlighted">方法的重载和访问修饰符以及返回值类型无关。</span>

- 方法名相同，参数类型不同，算重载；
- 方法名相同，参数个数不同，算重载；
- 方法名相同，参数顺序不同，算重载；

因此，方法名相同，参数列表不同，返回值不同是重载。

:::

### ⭐️5.==和equals的区别？

:::note 参考答案：

- 对于基本数据类型来说，`==` 比较的是值，对于引用数据类型来说，`==` 比较的是对象的内存地址；
- `equals()`只能用于判断两个对象是否相等，如果类没有重写`equals()`方法，使用的是默认的 `Object`类的`equals()`方法，等同于`==`比较的是对象的内存地址，如果类重写了`equals()`方法，则比较的是对象的内容。

:::

### ⭐️6.为什么重写 equals 时必须重写 hashCode ⽅法？

:::note 参考答案：

<span class="highlighted">为了保证相等的对象具有相等的哈希值（Hash Code），从而在使用基于哈希的集合（如 HashMap、HashSet）时能够正确存储和查询。</span>如果仅重写 `equals`，两个逻辑相等（内容相同）的对象会因 `hashCode` 不同而被视为两个互不相关的独立对象，导致基于哈希的集合无法查找到相等元素。

比如`HashMap`，HashMap 通过对象的哈希码将其存储在不同的“桶”中，当查找对象时，它需要使用 `key` 的哈希码来确定对象在哪个桶中，然后再通过 `equals()` 方法找到对应的对象。如果重写了 `equals()`方法而没有重写 `hashCode()`方法，那么被认为相等的对象可能会有不同的哈希码，会导致相等的对象分布在不同的桶中，从而导致无法在 HashMap 中正确处理这些对象。

:::

### ⭐️7.String、StringBuffer、StringBuilder 的区别？

:::note 参考答案：

- `String` 是不可变的，也是线程安全的，比如你拼接一个字符串，它其实是新创建了一个对象；
- `StringBuffer` 是可变的而且是线程安全的，是用了 synchronized 修饰的方法；
- `StringBuilder` 和 StringBuffer 类似，但没有加锁，所以是线程不安全的，不过性能更好；

一般我们操作少量的数据使用String，单线程操作大量数据使用 StringBuilder，多线程操作大量数据使用 StringBuffer。

:::

### ⭐️8.String 为什么是不可变的?

```java
// String类
public final class String
    implements java.io.Serializable, Comparable<String>, CharSequence {
    @Stable
    private final byte[] value;  // 保存字符串的数组，Java8是char[],Java9之后是byte[]
    private final byte coder;
    private int hash;
}
```

:::note 参考答案：

在 Java 8 中，String 底层是使用`private final char[]`来保存字符串。

- 一方面保存字符串的数组被 `final` 修饰且为私有的，并且`String` 类没有提供修改这个字符串的方法；
- 另一方面`String` 类被 `final` 修饰导致其不能被继承，进而避免了子类破坏 `String`的不可变性。

这两点保证了String的不可变性。

在Java 9 之后，String的实现改用`byte[]`数组存储字符串。主要目的是为了**节省内存空间，提高内存利用率**。

:::

### String类为什么要设计成不可变的？

https://zhuanlan.zhihu.com/p/1949147385998606736

https://www.cnblogs.com/wkfvawl/p/11693260.html

:::note 参考答案：

主要是为了**安全性、高效的内存管理（字符串常量池）、缓存哈希值以及多线程安全性**。

- 安全性：字符串广泛用于网络连接（URL）、文件路径（Path）和数据库URL。如果字符串是可变的，在执行操作时它们可能会被恶意代码修改，导致安全漏洞。由于不可变，这些核心参数在传输过程中始终保持一致，保证了系统的安全性。
- 高效的内存管理：JVM中维护着一个特殊的内存区域——字符串常量池。如果String是可变的，当多个变量指向同一个字符串时，修改其中一个变量会影响到其他变量。不可变性允许JVM共享相同的字符串字面量，这极大地减少了内存的消耗，避免了为相同内容的字符串重复创建对象。 
- 缓存哈希值：String作为HashMap或HashSet的键（Key）非常常见。由于String的内容不可变，它的哈希值计算一次后就可以被缓存（cached），不需要每次使用都重新计算。这使得String在Map中作为Key的性能优于其他对象。
- 线程安全：不可变对象不仅可以被多个线程安全地共享，而且在多线程环境下不需要额外的同步控制（如synchronized），避免了并发修改导致的死锁或数据不一致问题。

:::

### 9.Object 类有哪些常见方法？

:::note 参考答案：

- `hashCode()`：返回对象的哈希码；
- `equals()`：用于比较 2 个对象的内存地址是否相等；
- `toString()`：返回对象的字符串表示；
- `clone()`：返回对象的一个副本，默认是浅拷贝；
- `wait()`：让当前线程等待；
- `notify()`：唤醒单个线程；
- `notifyAll()`：唤醒所有线程；
- `getClass()`：获取对象的类信息；
- `finalize()`：当垃圾回收器决定回收对象占用的内存时调用此方法，用于清理资源。

:::

### 10.Java中的深拷贝和浅拷贝有什么区别？

![深拷贝和浅拷贝](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223132959001.png)

::: center
深拷贝和浅拷贝示意图
:::

:::note 参考答案：

深拷贝和浅拷贝都会创建一个新对象。对于基本数据类型的属性，两者都复制值；对于引用数据类型，两者的区别在于**是否复制引用的对象本身**：

- **浅拷贝**：复制引用地址，新旧对象共享同一个内部对象。通过实现 `Cloneable` 接口并重写 `clone()` 方法实现。
- **深拷贝**：递归复制所有引用对象，新旧对象完全独立。可通过手动复制引用字段、**拷贝构造器**或**序列化/反序列化**实现。

:::

>补充知识点：什么是拷贝构造器？

**拷贝构造器（Copy Constructor）** 是一种特殊的构造方法，**用已有对象来创建新对象**，实现深拷贝。

```java
public class Person {
    private String name;
    private List<String> hobbies;  // 引用类型
    
    // 普通构造器
    public Person(String name, List<String> hobbies) {
        this.name = name;
        this.hobbies = hobbies;
    }
    
    // ✅ 拷贝构造器：关键在这里！
    public Person(Person other) {
        this.name = other.name;                    // String不可变，直接赋值
        this.hobbies = new ArrayList<>(other.hobbies);  // 新建List，深拷贝！
    }
}

// 原对象
Person p1 = new Person("Alice", new ArrayList<>(List.of("读书", "游泳")));

// 用拷贝构造器创建深拷贝
Person p2 = new Person(p1);  

// 验证独立性
p2.getHobbies().add("跑步");

System.out.println(p1.getHobbies()); // [读书, 游泳] ✅ p1不受影响！
System.out.println(p2.getHobbies()); // [读书, 游泳, 跑步]
```

### 11.说一下Java的异常体系？

![Java异常体系](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223134447831.png)

::: center
Java异常体系
:::

:::note 参考答案：

Java中的异常包括`Error` 和 `Exception`，它们都继承自`Throwable`类。

- `Error`类代表那些严重的错误，这类错误通常是程序无法处理的。比如，`OutOfMemoryError` 表示内存不足，`StackOverflowError` 表示栈溢出。这些错误通常与 JVM 的运行状态有关，一旦发生，应用程序通常无法恢复。
- `Exception` 类代表程序可以处理的异常。它分为编译时异常（Checked Exception）和运行时异常（Runtime Exception）。
  - `编译时异常`在编译时必须被显式处理，如果不捕获(try-catch)或声明抛出(throws)，那么编译就无法通过。例如：`IOException`、`SQLException` 等。
  - `运行时异常`是在运行时抛出的，我们可以不去处理，通常是由程序逻辑错误导致的。例如 `NullPointerException`、`IndexOutOfBoundsException` 等。

异常的处理方式通常有两种：

- 不处理异常，直接通过`throw` 和 `throws` 抛出异常，交给上层调用者处理。
- 使用 `try-catch` 捕获异常，处理异常。

:::

> 扩展1：try-catch-finally抛出异常，catch和finally的异常可以同时抛出吗？

```java
public static void test() {
    try {
        throw new RuntimeException("try异常");  // 先抛出
    } catch (Exception e) {
        throw new RuntimeException("catch异常"); // 再抛出
    } finally {
        throw new RuntimeException("finally异常"); // 最后抛出，获胜！
    }
}

// 调用结果：只抛出 "finally异常"
// catch异常 被丢失了！❌

// 输出：Exception in thread "main" java.lang.RuntimeException: finally异常  
```

:::danger 参考答案：

`catch` 和 `finally` 都发生异常，最终抛出的是 `finally` 的异常，`finally` 会**压制(Suppressed)**`catch` 的异常，`catch` 中的异常不会抛出。

如果需要保留两个异常，通常可以使用使用 `try-with-resources`或者手动保留`catch`中的异常。

`try-with-resources`会自动确保资源关闭，消除了传统的 `finally` 块来手动关闭资源，会正确的处理异常压制情况。如果 `try` 块内的代码和 `close()` 方法都抛出异常，`try-with-resources` 会将 `close()` 产生的异常“抑制”，并抛出最上层的业务异常。

手动保留`catch`中的异常，常见的做法是使用一个变量临时存储 `catch` 中的异常，然后在 `finally` 中抛出 `finally`异常的同时附带 `catch`异常 作为抑制异常。

:::

```java
// 代码辅助理解保留两个异常
// 以前的写法 (try-finally)
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    return br.readLine();
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (br != null) {
        try {
            br.close(); // 需要手动处理 close 的异常
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

// 现在的写法 (try-with-resources)
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    return br.readLine();
} catch (IOException e) {
    e.printStackTrace();
} // 自动关闭，即使异常了


// 手动保留异常demo
public static void test() {
    RuntimeException catchException = null;
    
    try {
        throw new RuntimeException("try异常");
    } catch (Exception e) {
        catchException = new RuntimeException("catch异常");
        throw catchException;
    } finally {
        RuntimeException finallyException = new RuntimeException("finally异常");
        if (catchException != null) {
            finallyException.addSuppressed(catchException);  // 关联异常
        }
        throw finallyException;
    }
}

// 结果：抛出 finally异常，但附带 catch异常 作为抑制异常
// 输出：
Exception in thread "main" java.lang.RuntimeException: finally异常
    Suppressed: java.lang.RuntimeException: catch异常
```

>扩展2： try-catch-finally的执行顺序？

```java
// 情况1 ：无异常
try {
    System.out.println("1. try");
} catch (Exception e) {
    System.out.println("2. catch");  // 不执行
} finally {
    System.out.println("3. finally");
}
// 输出：1. try → 3. finally


// 情况2 ：有异常且捕获
try {
    System.out.println("1. try");
    throw new RuntimeException("异常");
} catch (Exception e) {
    System.out.println("2. catch");  // ✅ 执行
} finally {
    System.out.println("3. finally");
}
// 输出：1. try → 2. catch → 3. finally


// 情况3 ：有异常但未捕获
try {
    System.out.println("1. try");
    throw new RuntimeException("异常");
} catch (NullPointerException e) {  // ❌ 类型不匹配
    System.out.println("2. catch");  // 不执行
} finally {
    System.out.println("3. finally"); // ✅ 仍然执行！
}
// 输出：1. try → 3. finally → 抛出异常
```

:::danger 参考答案：

核心要点：

- **finally 始终执行**：无论 try/catch 发生什么，finally 都会执行；
- **唯一不执行 finally 的情况**：`System.exit()`、JVM 崩溃、线程被中断；
- **return 不影响 finally**：即使 try/catch 中有 return，finally 仍会先执行；
- **finally 中的 return 会覆盖**：会覆盖 try/catch 的返回值。

以下是几种情况：

- 无return，无异常：`try ->finally`
- 无return，有异常：`try(未出现异常的前半段) -> catch ->finally`
- try或catch中有return，无异常：`try -> finally ->return(try)`
- try或catch中有return，有异常：`try(未出现异常的前半段) -> catch ->finally->return(catch)`
- 只要是finally中有return的情况，不论有没有异常,try或catch中有没有return：`try/catch->return(finally)` 当finally中有return的时候，相当于此代码肯定会返回该值。

:::

### 12.BIO、NIO、AIO 之间的区别？

![Java IO](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223145417070.png)

::: center
BIO、NIO 和 AIO的区别
:::

:::note 参考答案：

Java中的IO包括：BIO、NIO 和 AIO，它们的区别如下：

- `BIO`：采用同步阻塞式 I/O 模型，线程在执行 I/O 操作时被阻塞，无法处理其他任务；适用于连接数较少且稳定的场景。
- `NIO`：使用同步非阻塞 I/O 模型，线程在等待 I/O 时可执行其他任务，通过 `Selector` 监控多个 `Channel` 上的事件，提高性能；适用于连接数多但连接时间短的场景。
- `AIO`：采用异步 I/O 模型，线程发起 I/O 请求后立即返回，当 I/O 操作完成时通过`回调函数`通知线程，进一步提高了并发处理能力；适用于连接数多且连接时间长的高吞吐量场景。

一个烧水的例子来理解三者的区别：

`BIO`：我们的工作模式是先来到厨房，开始烧水，并坐在水壶面前一直等着水烧开。

`NIO`：我们的工作模式是先来到厨房，开始烧水，但是我们不一直坐在水壶前面等，而是回到客厅看电视，然后每隔几分钟到厨房看一下水有没有烧开。

`AIO`：我们的工作模式是先来到厨房，开始烧水，我们不一直坐在水壶前面等，也不隔一段时间去看一下，而是在客厅看电视，水壶上面有个开关，水烧开之后他会通知我。

阻塞 VS 非阻塞：人是否坐在水壶前面一直等。

同步 VS 异步：水壶是不是在水烧开之后主动通知人。

:::

>扩展1：NIO和AIO的区别？

:::danger 参考答案：

### 13.final、finally、finalize 的区别？

:::note 参考答案：

- `final` 是一个修饰符，可以修饰类、方法和变量。当 final 修饰一个类时，这个类不能被继承；当 final 修饰一个方法时，这个方法不能被重写；当 final 修饰一个变量时，这个变量是个常量，一旦赋值后，就不能再被修改了。
- `finally` 是 Java 中异常处理的一部分，用来创建 try 块后面的 finally 块。无论 try 块中的代码是否抛出异常，finally 块中的代码总是会被执行。通常，finally 块被用来释放资源，如关闭文件、数据库连接等。
- `finalize` 是Object 类的一个方法，用于在垃圾回收器将对象从内存中清除出去之前做一些必要的清理工作。

:::

### 15.new 子类的时候，子类和父类静态代码块，构造代码块、构造方法的执行顺序？

:::note 参考答案：

Java中静态代码块、构造代码块和构造方法的执行顺序是：**静态代码块 → 构造代码块 → 构造方法**。

静态代码块在类加载时执行一次，而构造代码块和构造方法则在创建对象时执行，其中构造代码块在构造方法之前运行。

对于继承关系，则是按照**父类静态代码块 → 子类静态代码块 → 父类构造代码块 → 父类构造方法 → 子类构造代码块 → 子类构造方法**的顺序执行。 

:::

### 16.String s = new String("abc")会创建几个对象？

:::note 参考答案：

会创建一个或两个对象。

- 字符串常量池中如果之前已经有abc，则不再创建新的，直接引用；如果没有，则创建一个。
- 堆中肯定有一个，因为只要使用了 new 关键字，肯定会在堆中创建一个。

:::

### 17.String str1 = new String("abc") 和 String str2 = "abc" 的区别？

:::note 参考答案：

`String str2 = "abc"`为字符串变量赋值，会先检查字符串常量池中是否已经存在相同内容的字符串，如果存在则直接引用字符串常量池中的字符串，如果不存在就创建一个新的字符串放入字符串常量池中并引用它。

使用`new String("abc")`的方式创建字符串时，实际分为两步：

- 第一步，先检查字符串字面量 "abc" 是否在字符串常量池中，如果没有则创建一个；如果已经存在，则引用它。
- 第二步，在堆中再创建一个新的字符串对象，并将其初始化为字符串常量池中 "abc" 的一个副本。

`String str2 = "abc"`会创建0个/1个对象；`new String("abc")`会创建1个/2个对象。

:::

### 18.说一下你对泛型的理解？ 

:::note 参考答案：

泛型主要用于提高代码的类型安全，它允许在定义`类、接口和方法`时使用类型参数，这样可以在编译时检查类型一致性，避免不必要的类型转换和类型错误。

如果没有泛型，像List这样的集合存储的是Object类型的数据，我们从集合中读取数据的时候必须进行强制类型转换，否则会引发类型转换常。

:::

>扩展1：什么是泛型擦除？

:::danger 参考答案：

简单来说，泛型擦除就是 Java 编译器在编译时，会把所有泛型信息`“擦除”`掉，替换成它们的上界`Object`。泛型规定的类型只在编译期存在，在运行时就不在了。

目的是为了兼容旧版本。这样做的好处是，泛型代码可以和旧版本的非泛型代码一起运行。

:::

注解本质上是一个`标记`。注解可以标记在类、方法和属性上等。有了标记之后，我们就可以在编译或者运行阶段去识别这些标记，然后去做一些事情。

例如我们常见的 AOP，使用注解作为切点就是运行期注解的应用，比如 lombok，就是注解在编译期的运行。

### 20. 反射及其应用场景？

:::note 参考答案：

反射是一种允许在程序运行时动态获取类的信息并操作类的能力。通过反射，可以动态地获取类的字段、方法、构造方法等信息，并在运行时调用方法或访问字段。

反射有很多的应用场景，比如：

- Spring 框架使用反射来动态加载和管理 Bean；
- Java 的动态代理机制使用反射来创建代理类；
- JUnit测试框架使用反射机制来发现和执行测试方法。

:::

>扩展1：反射调用私有方法和直接调用有什么区别？

:::danger 参考答案：

**访问控制**：直接调用，严格遵守`private`限制，类外部无法直接调用，保障了封装性；反射调用，暴力突破访问检查，实现外部对私有方法的访问。

**性能上**：直接调用在编译时即确定目标地址，速度快，性能好；反射调用需要在运行时进行类型检查和方法解析，存在性能开销，性能相对较差。

**类型安全**：直接调用支持编译时类型检查，参数错误会在编译期报错；反射调用，编译期无法检查方法名和参数正确性，一旦出错会抛出运行时异常。

**适用场景**：直接调用，适用于常规业务逻辑；反射调用，常用于框架开发、动态代理等需要操作私有成员的特殊场景。

:::

>扩展2：假如我现在有一个第三方的jar包，里面有一个私有的方法，我该如何调用这个方法？

:::danger 参考答案：

使用Java的反射机制：

1. **获取Class对象**：获取包含该私有方法的类的实例或Class对象；
2. **获取Method对象**：使用`getDeclaredMethod("方法名", 参数类型...)`获取私有方法；
3. **设置访问权限**：调用`method.setAccessible(true)`打破private限制；
4. **调用方法**：使用`method.invoke(实例对象, 参数...)`执行。

```java
// 示例代码：
// 假设第三方类为 com.example.ThirdPartyClass，私有方法名为 privateMethod.
import java.lang.reflect.Method;

public class TestPrivateMethod {
    public static void main(String[] args) throws Exception {
        // 1. 获取目标类实例
        com.example.ThirdPartyClass instance = new com.example.ThirdPartyClass();

        // 2. 获取该类的Class对象
        Class<?> clazz = instance.getClass();

        // 3. 获取私有方法：getMethodName, parameterTypes
        Method privateMethod = clazz.getDeclaredMethod("privateMethod", String.class);

        // 4. 关键：设置强制访问
        privateMethod.setAccessible(true);

        // 5. 执行方法：instance, args
        String result = (String) privateMethod.invoke(instance, "参数");
        System.out.println(result);
    }
}
```

:::

### 21.Java SPI 机制？

![img](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223154237206.jpg)

:::note 参考答案：

SPI 即 Service Provider Interface ，是JDK内置的一种`服务提供发现机制`，用于实现**接口与实现的解耦**，让第三方可以插拔式地扩展功能，主要是被框架的开发人员使用。

SPI有三个关键的组件：

- Service Interface：定义标准接口，位于调用方/标准方提供；
- Service Provider：具体实现类，位于第三方 Jar 包；
- ServiceLoader：加载实现的工具类，JDK 内置 `java.util.ServiceLoader`。

当服务的提供者提供了一种接口的实现之后，需要在classpath下的`META-INF/services/`目录里创建一个以服务接口命名的文件，这个文件里的内容就是这个接口的具体的实现类。当其他的程序需要这个服务的时候，就可以通过查找这个jar包的`META-INF/services/`中的配置文件，配置文件中有接口的具体实现类名，可以根据这个类名进行加载实例化，从而使用该服务。JDK中查找服务的实现的工具类是ServiceLoader。

比如JDBC加载驱动就是使用的SPI。在java中定义了`Driver`接口，具体的实现由不同厂商来提供，比如mysql、postgresql。再使用SPI来获取驱动的实现。

:::

>扩展1：SPI 和 API 有什么区别？

:::danger 参考答案：

- SPI：“接口”位于“调用方”所在的“包”中；
- API：“接口”位于“实现方”所在的“包”中。

### 抽象类和普通类的区别？

:::note 参考答案：

- 普通类没有abstract关键字修饰，抽象类是被abstract修饰的；
- 普通类可 `new` 对象，抽象类不能；
- 普通类不能有抽象方法，抽象类可以有抽象方法，也可以没有；

- 普通类常用作类的完整实现，抽象类常作为基类提供通用模板，定义规范。

### Java中序列化和反序列化是什么？使用场景？

:::note 参考答案：

Java中序列化是将对象的状态（数据）转换为可存储或传输的二进制字节流的过程，反序列化则是将其还原为原始Java对象的过程。

- 序列化是指将对象转换为字节流的过程，以便能够将该对象进行存储或传输；
- 反序列化就是将字节流转换回对象的过程，以便构建原始对象。

序列化和反序列化通常需要实现`java.io.Serializable`接口。

使用场景：①持久化存储对象；②RPC 调用中传输参数和返回值。

>扩展1：说说Lambda 表达式的底层原理？

:::danger 参考答案：

Lambda 底层运行原理如下：

- 在程序运行时，会在类中生成一个匿名内部类，匿名内部类会实现接口，并重写接口中的抽象方法。
- 类中会生成一个静态方法，静态方法中的代码就是 Lambda 表达式中的代码。
- 匿名内部类重写的抽象方法，会调用上一步的静态方法，从而实现 Lambda 代码的执行。

简单来说，Lambda 表达式其实是匿名内部类的语法糖，这个语法糖在程序执行时会进行兑现，也就是生成匿名内部类并进行任务执行。

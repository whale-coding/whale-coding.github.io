### 1.什么是消息队列，为什么要使用消息队列，消息队列的使用场景

:::note

**消息队列**，可以理解为一个使用队列来通信的消息中间件，广泛应用于分布式系统中。

使用消息中间件的目的主要有3个：**异步**、**解耦**、**削峰**。

**异步**：系统可以将那些耗时的任务放在消息队列中异步处理，从而快速响应用户的请求。比如说，用户下单后，系统可以先返回一个下单成功的消息，然后将订单信息放入消息队列中，后台系统再去处理订单信息。

**解耦**：生产者将消息放入队列，消费者从队列中取出消息，这样一来，生产者和消费者之间就不需要直接通信，生产者只管生产消息，消费者只管消费消息，这样就实现了解耦。

**削峰**：高并发下，大量请求到达系统后，由生产者接收请求并将其转化为消息，发送到 RocketMQ 队列中。队列用来充当缓冲区，然后系统再慢慢按自己的最大消费能力去消费这些消息，这样就避免直接把后端服务打垮掉。

:::

### 2.为什么选择使用RabbitMQ? 而不是Kafka、RocketMQ？

就说自己的服务器资源有限，虽然kafka和Rocket性能更好，但是CPU和内存占用也大，自己的项目也不需要太多吞吐量，所以综合来说Rabbit是最合适的.

:::note

一方面是自己的服务器资源有限，虽然kafka和RocketMQ性能更好，但是CPU和内存占用也大，自己的项目也不需要太多吞吐量。另一方面，RabbitMQ提供了一个可视化的界面，方便我们的使用。

### 3.说一说RabbitMQ的核心组件和工作原理

![77299e96-fb5f-4e26-aa73-312002cc2956](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304214829605.png)

:::note

RabbitMQ 整体上是一个生产者与消费者模型。RabbitMQ的核心组件包括：生产者、消费者、Broker、交换机和消息队列。

生产者生产消息，消费者消费消息，消息队列用来保存消息，Broker可以看做是RabbitMQ 服务节点或者是RabbitMQ 服务实例。交换机负责根据指定路由把生产者的消息分配到对应的消息队列。这是因为在 RabbitMQ 中消息必须经过交换机这一层分配到对应的消息队列。

**RabbitMQ的工作原理**：

生产者通过TCP连接到RabbitMQ Broker 服务器，建立一个Channel (信道)。通过该信道，生产者将消息发送给交换机（Exchange）。交换机根据类型和消息的路由键（Routing Key）和其绑定的队列（Queue）关系，将消息路由到指定的队列。消息被路由到队列后，会存储在队列中，直到被消费者消费。消费者也通过TCP连接到RabbitMQ Broker 服务器，建立一个Channel。通过该信道，消费者可以主动从队列中拉取消息（Pull）或者由服务器推送消息（Push）。消费者处理完消息后，需要向Broker 服务器确认消息已消费。

:::

### 4.RabbitMQ有哪些交换机？

:::note

RabbitMQ支持**fanout**、**direct**、**topic** 和 **headers** 四种交换机类型。

- **fanout交换机，**是将消息投递到所有绑定的队列，不需要规则，不需要匹配，相当于广播；
- **direct交换机**，是将消息路由到 BindingKey 和 RoutingKey 相匹配的队列中；
- **topic交换机**，是根据通配符匹配BindingKey 和 RoutingKey 然后投递到相匹配的队列中；
- **headers交换机**，是根据发送的消息内容中的 headers 属性进行匹配相应的队列。

:::

### 5.如何保证消息可靠投递不丢失？

![d24e27a2-182c-4851-afdc-38609f8f614a](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304215041967.png)

:::note

消息可能在生产者生产消息到MQ的过程中丢失、MQ自身搞丢了、MQ到消费者的过程中丢失了。

生产者到RabbitMQ：**开启Confirm确认机制。**

RabbitMQ自身：消息的持久化、MQ的高可用集群。

RabbitMQ 到消费者：**手动Ack**。

:::

### 6.如何保证消息的幂等性？消息的重复消费问题是如何解决的？

:::note

幂等性：多次操作，对系统不会造成影响。比如消息重复发送多次，消费者重复消费多次。

保证MQ的幂等性，只要保证消费者不会重复消费相同的消息即可，生产者发送多条消息影响并不大。

- 生产者发送消息时，为每条消息设置一个全局唯一的id。
- 消费者消费消息时，使用redis的setnx命令：SET id 1 NX，若返回OK，说明该消息之前没有消费过，正常消费；若返回nil，说明该消息之前已经消费过，那就不用处理。

:::

### 7.如何保证消息的顺序性？/RabbitMQ是怎么实现顺序消费的？

:::note

顺序消费，即保证消息的发送顺序和消息的消费顺序保持一致，RabbitMQ消息发送到队列中，队列本身就是有序的，所以RabbitMQ保证消息的顺序性，只用考虑消费者即可。



出现消息乱序的情况：

- 一个队列，有多个消费者去消费，这样就会造成顺序的错误，消费者从MQ里面读取数据是有序的，但是每个消费者的执行时间是不固定的，无法保证先读到消息的消费者一定先完成操作，这样就会出现消息并没有按照顺序执行。
- 一个队列对应一个消费者，但是消费者里面进行了多线程消费，这样也会造成消息消费顺序错误。

解决方案：

- 队列与消费者一对一，一个队列设置一个消费者。
- 使用内存队列，一个队列只对应一个消费者，然后这个消费者内部用内存队列做排队，然后分发给底层不同的worker来处理。

:::

### 8.什么是死信队列？RabbitMQ死信队列是如何导致的？

![3570d82e-e19d-46d5-a43a-47f6272c18d6](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304215422991.png)

:::note

死信队列：当消息在一个队列中变成死信之后，它能被重新发送到另一个交换机中，这个交换机就是死信交换机，死信交换机绑定的队列就是死信队列。

死信队列本身也是一个普通的消息队列，可以设置一些参数将其设置为死信队列；

死信队列是一个用于存放无法被消费的消息的队列，这些消息被称为死信，死信队列可以避免消息一直被消费却无法消费成功的情况。

**导致死信的几种原因：**

1. 消息被拒绝，包括消息不确认和拒绝确认
2. 消息过期
3. 消息队列满了，丢弃的消息

:::

### 9.什么是延迟队列？RabbitMQ如何实现延迟队？

![c55684e5-c5f6-4abb-8dc5-8e89566aecbc](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304215514138.png)

:::note

存放“延迟消息”的队列就叫做延迟队列，“延迟消息”指的是消息被发送到队列以后，并不想让消费者立即拿到消息，而是等待指定的时间后，消费者才能拿到这个消息。

RabbitMQ本身没有延迟队列，可以使用插件来实现或者**使用TTL消息过期+死信队列**来实现。

:::

### 10.RabbitMQ的高可用机制有了解嘛？

![4da4134f-b40e-4015-befc-80f4ed5b1882](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304215559959.png)

:::note

RabbitMQ有三种模式：

- 单机模式
- 普通集群模式
- 镜像集群模式

1. 普通集群模式：在多台机器上启动多个 RabbitMQ 实例，创建的**队列只会放在一个RabbitMQ 实例上**，但是多个实例之间**会复制队列的元数据**，从而保持队列的元数据同步，**但是并不会复制队列中的消息**。元数据也就是队列的配置信息，通过元数据可以找到队列所在的实例。当消费的时候，实际连接到的RabbitMQ实例会通过元数据找到实际队列所在的实例，然后拉取数据。
2. 镜像集群模式：多个RabbitMQ实例之间会复制所有的内容，不仅包括队列的元数据，还包括队列中的消息，也就是说多个RabbitMQ实例的内容是完全相同的，所以叫做镜像模式。

普通集群模式，一但队列所在的RabbitMQ实例宕机了，就没有办法实现高可用，算不上真正的高可用，镜像集群模式是RabbitMQ 的高可用模式。

:::

### 11.如果有百万消息堆积在MQ中，如何解决？

:::note

出现消息堆积的原因：生产和消费失衡，当生产者发送消息的速度超过消费者处理消息的速度，就会导致队列中的消息堆积。

**应急处理**：

1. 增加消费者数量，提高消费速度。例如增加消费者实例，消费者开启多线程消费。
2. 扩大队列容积，提高消费堆积的上限。可以使用RabbitMQ的**惰性队列**，将来不及消费的消息保存到磁盘上，等高峰期过了之后再慢慢消费。

**长期优化：**

1. 从业务逻辑上优化消费者，减少单条消息处理的耗时。
2. 在生产端做**限流和削峰填谷**，避免流量瞬时冲击。
3. 对于延迟后无效的消息，设置 **TTL** **+ 死信队列**，避免无意义堆积。

:::

>补充知识点：惰性队列的特征
>
>- 接收到消息后直接存入磁盘而非内存
>- 消费者要消费消息时才会从磁盘中读取并加载到内存
>- 支持数百万条消息的存储

### 12.如何解决RabbitMQ中因为消息堆积而导致的消息过期失效的问题？

:::note

消息堆积，会导致大量消息存储在队列中得不到消费，而消息又设置了TTL过期时间，当达到过期时间时，消息会被过期丢弃。

解决办法：

- 让消息不要被堆积
- 不设置TTL过期时间或者增加过期时间的TTL值
- 设置死信队列，消息过期之后投递到死信队列中，之后再进行补偿处理。
- 编写临时程序补发消息。

:::

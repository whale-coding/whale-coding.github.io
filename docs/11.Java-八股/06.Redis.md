## Redis基础

### 1.Redis是什么？ 通常应用于哪些场景？

:::note

Redis 是一个基于键值对的 NoSQL 数据库。Redis的数据是以键值对的形式存放在内存中的，对数据的读写操作都是在内存中完成，因此读写速度非常快。

常用于缓存、分布式锁、排行榜、计数器等场景。

:::

>1.Redis 和 Memcached 有什么区别？

:::danger 答案

- Redis 支持的数据类型更丰富(String,Hash,List,Set,ZSet),而 Memcached 只支持最简单的key-value 数据类型;
- Redis 支持数据的持久化，可以将内存中的数据保持在磁盘中，重启的时候可以再次加载进行使用，而Memcached 没有持久化功能，数据全部存在内存之中，Memcached 重启或者挂掉后，数据就没了；
- Redis 原生支持集群模式，Memcached 没有原生的集群模式，需要依靠客户端来实现往集群中分片写入数据；
- Redis 支持发布订阅模型、Lua 脚本、事务等功能，而 Memcached 不支持。

:::

### 2.Redis有哪些数据类型？

:::note

Redis的常见数据类型有：String、List、Set、ZSet、Hash；

扩展数据类型有：Bitmap、HyperLogLog、GEO、Stream。

- String是最基本的数据类型，value是字符串，适合缓存单个对象，比如缓存验证码；
- List是一个有序的列表，允许在两端进行插入、删除元素，常用于消息队列或者是任务列表；
- Set是无序不可重复集合，支持交集、并集操作，常用于去重、共同好友等场景；
- ZSet是有序不可重复集合，集合中的元素按分数进行排序，支持范围查询，常用于排行榜；
- Hash的value是一个键值对集合，适合存储对象，可以对单个字段进行操作。常用于缓存对象；
- Bitmap是一串连续的二进制数组，常用于签到场景；
- HyperLogLog可以用来做基数统计，常用于网站UV统计；
- GEO用于存储和查询地理位置信息，常用于附近的人；
- Stream是5.0新增的数据结构，主要用于消息队列。

:::

>1.List实现的消息队列和Stream实现的消息队列有什么不同？

:::danger 答案

- List实现的消息队列，不支持消息确认机制，只支持简单的消息入队出队操作，适合简单的消息队列；
- List实现的消息队列，生产者需要自行实现全局唯一ID；Stream实现的消息队列自动生成全局唯一ID；
- Stream实现的消息队列，支持消息的**持久化**、**消息确认机制**以及**消费者组**等高级功能，更适合复杂的场景。

:::

### 3.Redis 为什么这么快？

:::note

- Redis 是基于内存的数据库，读写操作都发生在内存中，内存的读写速度远高于磁盘；
- Redis的数据结构是经过优化的，比较高效，比如String的底层用的并不是C语言中的字符串而是SDS（简单动态字符串）；
- 对于读写命令的执行，Redis是单线程的，避免了上下文切换和锁竞争问题；
- Redis采用I/O多路复用技术，可以处理多个并发连接而无需创建大量线程；

:::

### 4.什么是I/O多路复用技术？

:::note

IO 多路复用是一种同步IO模型，允许单个线程同时监听多个Socket，只在Socket可读/可写时

进行处理，避免阻塞等待。

IO多路复用有三种实现：select、poll和epoll。

select和poll都需要**遍历所有fd**来查找就绪的Socket，效率低。而epoll基于**事件驱动**，**只返回就绪的fd**，且没有fd数量限制。目前大多采用的**epoll**模式，效率高。

epoll支持**LT（水平触发）**和**ET（边缘触发）**两种模式，Redis使用的是LT模式。

Redis采用**单线程Reactor模型**，基于epoll实现I/O多路复用，配合连接应答处理器、命令请求处理器、命令回复处理器等事件处理器来处理多个客户端请求。单线程设计避免了锁竞争和上下文切换开销，性能更加的好。

:::

>1.IO多路复用的三种实现：select、poll、epoll有什么区别？

| 特性               | select                 | poll                     | epoll                                |
| :----------------- | :--------------------- | :----------------------- | :----------------------------------- |
| **fd数量限制**     | 有（默认1024/2048）    | 无限制（受限于系统内存） | 无限制                               |
| **数据结构**       | 位图（bitmap）         | 数组                     | 红黑树 + 就绪链表                    |
| **查找就绪fd方式** | **遍历所有fd**（O(n)） | **遍历所有fd**（O(n)）   | **直接返回就绪fd**（O(1)）           |
| **fd拷贝次数**     | 每次调用都拷贝到内核   | 每次调用都拷贝到内核     | **epoll\_ctl时拷贝一次**，后续不拷贝 |
| **触发模式**       | 水平触发               | 水平触发                 | **支持LT + ET**                      |
| **跨平台**         | POSIX标准，兼容性好    | POSIX标准                | **Linux特有**                        |

:::danger 答案

- select有fd数量限制，poll和epoll没有fd数量限制；
- select是基于位图，poll是基于数组，epoll是基于红黑树+就绪链表；
- select和poll是遍历所有fd，epoll是直接返回**就绪fd**；
- select和poll是水平触发，epoll支持水平触发和边缘触发。

:::

>2.epoll的两种触发模式：LT（水平触发）、ET（边缘触发）了解吗？Redis用的是哪种？

epoll的两种触发模式：

| 模式                              | 特点                                       | 适用场景                      |
| :-------------------------------- | :----------------------------------------- | :---------------------------- |
| **LT（Level Trigger，水平触发）** | 只要fd就绪，**一直通知**，直到处理完毕     | 默认模式，编程简单，Redis使用 |
| **ET（Edge Trigger，边缘触发）**  | 状态变化时**只通知一次**，必须一次性处理完 | 高性能场景，需配合非阻塞I/O   |

:::danger 答案

- LT触发：只要fd就绪，**一直通知**，直到处理完毕。
- ET触发：状态变化时**只通知一次**，必须一次性处理完。

Redis使用的是**LT触发模式**。

:::

### 5.Redis是单线程的吗？

:::note

Redis 6.0以前Redis是单线程的，**Redis 6.0 版本引入了多线程**。

对于网络 I/O 采用多线程来处理，目的是提高网络I/O的性能。**但是对于命令的执行，Redis 仍然使用单线程来处理**。

:::

### 6.为什么 Redis 设计为单线程？6.0 版本为何引入多线程？

:::note

Redis早期设计成单线程，主要是因为：

- 单线程比较简单，不存在竞争条件和死锁问题，方便开发和维护；
- Redis 是IO 密集型而非 CPU 密集型，主要受内存和网络 IO 限制，而非 CPU 的计算能力，单线程可以避免线程上下文切换的开销；
- 单线程可以保证命令执行的原子性和数据的一致性；
- Redis在单线程下通过IO多路复用技术，也可以高效地处理大量并发连接。

<br>

Redis6.0引入多线程，主要原因是：

- 提升 CPU 利用率。在多核 CPU 时代，单线程模型无法充分利用多核 CPU 的计算能力。引入多线程可以提升CPU的利用率，提高Redis的处理能力和吞吐量；
- 优化 I/O 操作。在高并发场景下，单线程模型难以高效处理大量的网络 I/O 请求。引入多线程后，Redis 可以将 I/O 操作分配到多个线程中并行处理，从而提升**网络I/O 处理能力**。

<span class="highlighted">

虽然引入了多线程，但是Redis 的读写命令的执行仍然使用单线程来处理，多线程主要用于优化网络I/O。

</span>

:::

## 缓存三兄弟

<span class="highlighted">

缓存击穿是单个热点数据失效导致的，缓存穿透是因为请求不存在的数据，而缓存雪崩是因为大范围的缓存失效。

</span>

### 1.什么是缓存穿透？解决方案？

:::note

**缓存穿透**是指大量请求的 key 是不合理的，数据根本不存在于缓存中，也不存在于数据库中，这会导致大量的请求打到数据库，导致数据库压力骤增，甚至宕机。这种情况大概率是遭到了恶意攻击，攻击者故意制造一些非法的 key 发起大量请求。

常见的解决方案有两种：

1. **缓存空数据**：如果缓存和数据库都查不到某个 key 的数据，就将该key和空值写到 Redis 中去并设置过期时间，这样下次相同的查询就能直接从缓存返回，而不再访问数据库。这种方法的优点是实现起来简单，但是如果黑客恶意攻击，每次构建不同的请求 key，就会导致 Redis 中缓存大量无效key。这种方案适合查询key比较集中的场景。
2. **使用布隆过滤器**：布隆过滤器是基于bitmap位图实现的，可以高效的判断一个元素是否存在于集合中，我们可以将所有可能存在的key存储到布隆过滤器中，查询时先检查布隆过滤器，如果布隆过滤器认为该key不存在，就直接返回空；否则再去查询缓存，这样就可以避免无效的缓存查询。

:::

### 2.什么是缓存击穿？解决方案？

:::note

**缓存击穿**是指如果缓存中的**某个热点数据过期**了，大量请求并发访问该热点key，由于数据存在于数据库中，但不存在于缓存中，大量请求就会直接打到数据库，导致数据库压力骤增，甚至宕机。

常见的解决方案有两种：

1. **使用分布式锁**：当缓存失效时，第一个访问的线程先获取锁并负责重建缓存，其他线程等待或重试。
2. **逻辑过期**：缓存本身不设置过期时间，也就是永不过期，但在缓存值中维护一个逻辑过期时间。当缓存逻辑上过期时，返回旧值的同时，异步启动一个线程去更新缓存。

如果优先考虑数据的强一致，可以使用互斥锁的方案，但是性能没有那么高；如果优先考虑可用性和性能，可以使用逻辑过期方案。

:::

### 3.什么是缓存雪崩？解决方案？

:::note

**缓存雪崩**是指在某一时间段，大量的缓存同时失效或者Redis服务突然宕机了，导致大量的请求都直接落到了数据库上，对数据库造成巨大的压力。

针对大量缓存同时失效的情况解决方案：

- **将缓存失效时间随机打散**：可以在原有的失效时间基础上增加一个随机值，这样每个缓存的过期时间都不重复了，也就降低了缓存集体失效的概率。
- **设置缓存不过期**： 我们可以通过后台服务来更新缓存数据，从而避免因为缓存失效造成的缓存雪崩，也可以在一定程度上避免缓存并发问题。

针对Redis服务宕机的情况解决方案：

- 使用高可用的缓存集群，避免单机出现问题整个缓存服务都没办法使用。
- 使用多级缓存，例如本地缓存+Redis 缓存的二级缓存组合，当 Redis 缓存出现问题时，还可以从本地缓存中获取到部分数据。

:::

### 4.缓存击穿和缓存雪崩的区别？

:::note

缓存击穿是单个热点数据的缓存失效，导致大量并发请求直接击穿到数据库；

而缓存雪崩是大量数据的缓存同时失效，引发数据库承受巨大的压力。

主要区别在于失效的范围：击穿是“一个点”，雪崩是“一片面”。

:::

### 5.布隆过滤器的原理？

:::note

布隆过滤器主要是用于检索一个元素是否存在于一个集合中。

它的底层是基于`bitmap`，先去初始化一个比较大数组，里面存放的二进制0或1。在一开始都是0，当一个key来了之后经过多次hash计算，比如3次，模于数组长度找到数据的下标然后把数组中原来的0改为1，这样的话，三个数组的位置就能标明一个key的存在。查找的过程也是一样的。

布隆过滤器绝不会漏判，即如果布隆过滤器认为某个数据不存在，那它一定不存在。但是存在误判，即可能会认为某个数据存在，但实际上并不存在。

:::

### 6.常见的缓存更新策略有哪些？

:::note

常见的缓存更新策略共有3种： 

- Cache Aside（旁路缓存）策略； 
- Read/Write Through（读穿 / 写穿）策略；
-  Write Back（写回）策略；

:::

## Key相关问题

### 1.什么是hot key（热点key）? 有什么危害？

:::note

热点key，指的是在很短时间内被频繁访问的key。比如参与秒杀的商品。

:::

### 2.hot key有什么危害？

:::note

由于 Redis 是单线程模型，大量请求集中到同一个key会导致该 Redis 节点的 CPU 使用率飙升，响应时间变长，如果突然访问 hotkey 的请求超出了 Redis 的处理能力，Redis 就会直接宕机。这种情况下，大量请求将落到后面的数据库上，可能会导致数据库崩溃。

在 Redis 集群环境下，热Key 还会导致数据分布不均衡，某个节点承受的压力过大而其他节点相对空闲。

因此，hotkey 很可能成为系统性能的瓶颈点，需要单独对其进行优化，以确保系统的高可用性和稳定性。

:::

### 3.如何发现/监控 hotkey ？

:::note

可以使用redis自带的 `--hotkeys`参数来查看Redis 中的热 Key。也可以借助京东的hotkey等开源项目去发现热key。

:::

### 4.如何解决 hotkey？

:::note

1. 增加本地缓存，将热 Key 缓存到本地内存中，这样请求就不需要访问 Redis 了；
2. 使用redis集群或者是读写分离。
3. 对于一些特别热的 Key，可以将其拆分成多个子 Key，然后随机分布到不同的 Redis 节点上。

比如将 `hot_product:12345` 拆分成 `hot_product:12345:1`、`hot_product:12345:2` 等多个副本，读取时随机选择其中一个。

:::

### 5.什么是big key（大key）？

:::note

如果一个 key 对应的 value 所占用的内存比较大，那这个 key 就可以看作是 bigkey。

比如说String类型的value超过`1MB`，集合类型比如List、ZSET的value 包含的元素超过`5000` 个。我们就可以认为其是大key。

:::

### 6.bigkey 是怎么产生的？有什么危害？

:::note

产生大key 通常是由于程序设计不当或数据规模预估不足造成的。比如使用String类型存储大文件，集合类型由于业务的增长包含大量的元素，又或者是有大量冗余的无用key。

大key的危害：

- 第一，会阻塞请求，由于 Redis 执行命令是单线程处理，在操作大 key 时会比较耗时，那么就可能会阻塞后续的请求处理；
- 第二，会导致内存增大，在内存有限的情况下，大key可能导致 Redis 内存不足；
- 第三，会阻塞网络，读取大Key会占用较大的带宽，可能会引发网络阻塞。
- 第四，影响主从同步，大 Key 会导致主从复制延迟、全量同步压力大

:::

### 7.如何发现 bigkey？

:::note

可以使用redis自带的 `--bigkeys`参数来查看Redis 中的大 Key。或者是借助一些开源的项目去分析RDB文件来找出bigkey。

:::

### 8.如何处理bigkey？

:::note

第一，可以对大Key进行拆分，将大Key拆分成多个小 Key 存储。比如将一个包含大量用户信息的 Hash 拆分成多个小 Hash；

第二，可以对大Key进行清理，可以使用`UNLINK`命令异步删除大Key，该命令会以非阻塞的方式缓慢逐步的清理传入的Key；

第三，可以对value进行压缩，减少开销；

第四，限制集合大小，在写入时做限制，例如一个 Hash 控制字段数量，List 限制长度；

第五，可以在业务层进行优化，不要使用Redis存储大对象，尽量避免大key的产生以及定期清理失效的key，对Redis的内存、带宽等进行监控。

:::

### 9.Redis的过期 key 删除策略有哪些？

:::note

Redis中提供了`惰性删除`和`定期删除`两种过期删除策略。

**惰性删除**：不主动删除过期的key，每次访问key时，都检查该key是否过期，如果过期则删除该key；

**定期删除**：每隔一段时间随机抽取一定数量的key进行检查，并删除其中过期的key。检查时间的间隔默认是`每秒10次`，可以通过修改配置文件中的`hz`参数来调整，每次随机抽取的数量是`20`。

Redis的过期删除策略是惰性删除和定期删除两种策略配合使用的。

:::

### 10.Redis的内存淘汰策略有哪些？

:::note

Redis支持8种数据淘汰策略：

- 默认策略是不淘汰任何key，但是内存满时不允许写入新数据；

- 对设置了TTL的key，淘汰策略有：TTL越小的优先淘汰、随机淘汰、LRU算法淘汰和LFU算法淘汰；

- 对全体key，淘汰策略有：随机淘汰、LRU算法淘汰和LFU算法淘汰：

  - LRU是最近最少使用，用当前时间减去最后一次访问时间，值越大淘汰优先级越高；

  - LFU是最少频率使用，会统计每个key的访问频率，值越小淘汰优先级越高。

平时开发过程中使用比较多的是`allkeys-lru`。

:::

>1. LRU 算法和 LFU 算法有什么区别？

:::danger 答案

**LRU** 是 **最近最少使用**，会选择淘汰最近最少使用的数据。

LFULFU

:::

## 持久化

### 1.Redis的持久化方式有哪些？

:::note

Redis提供了`RDB`、`AOF`和`混合持久化`三种持久化方式。

- **RDB**是快照的方式，将某一时刻的内存数据，以二进制的方式写入磁盘。当redis实例宕机恢复数据的时候，可以从RDB的快照文件中恢复数据；
- **AOF** 是日志追加的方式，redis每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。当redis实例宕机恢复数据的时候，会从这个文件中再次执行一遍命令来恢复数据。
- **混合持久化**：集成了 AOF 和 RDB 的优点，通常我们会使用混合持久化方式。

:::

### 2.RDB和AOF两种方式有什么区别？

:::note

- RDB文件的体积小，恢复速度快；AOF文件的体积相对大，恢复速度慢。
- RDB的安全性比较低，容易丢数据；AOF的安全性较高，不易丢失数据。

:::

### 3.RDB的原理？

:::note

RDB是快照的方式，将某一时刻的内存数据，以二进制的方式写入磁盘。

RDB 持久化可以通过 save 和 bgsave 命令手动触发，也可以通过配置文件中的 save 指令自动触发。

`save`命令会阻塞 Redis 进程，直到 RDB 文件创建完成；

`bgsave`命令会在后台 fork 一个子进程来执行 RDB 持久化操作，主进程不会被阻塞。

:::

> 1.RDB 做快照时会阻塞线程吗？

:::danger 答案

`save`命令会阻塞线程，`bgsave`命令不会阻塞线程。

:::

>2.RDB 在执行快照的时候，数据能修改吗？

:::danger 答案

可以的，执行`bgsave`过程中，Redis依然可以继续处理操作命令，也就是数据是能被修改的。

因为有<span class="highlighted">写时复制技术（Copy-On-Write, COW）</span>。

执行`bgsave`命令的时候，会通过`fork()`创建子进程，此时子进程和父进程是共享同一片内存数据的，因为创建子进程的时候，会复制父进程的页表，但是页表指向的物理内存还是同一个，此时如果主线程执行读操作，则主线程和`bgsave`子进程互相不影响。

如果主线程执行写操作，则被修改的数据会复制一份副本，然后`bgsave`子进程会把该副本数据写入`RDB`文件，在这个过程中，主线程仍然可以直接修改原来的数据。

:::

### 4.AOF的原理？

:::note

AOF 是日志追加的方式，redis每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里。

当 Redis 执行写操作时，会将写命令追加到 AOF 缓冲区；Redis 会根据刷盘策略将缓冲区的数据写入到 AOF 文件。

当 AOF 文件过大时，Redis 会自动进行 AOF 重写，剔除多余的命令，比如说多次对同一个 key 的 set 和 del，生成一个新的 AOF 文件；当 Redis 重启时，读取 AOF 文件中的命令并重新执行，以恢复数据。

:::

### 5.AOF写回策略有哪几种？

![img](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304145231670.png)

:::note

Redis 提供了 3 种写回硬盘（刷盘）的策略，在 Redis.conf 配置文件中的 appendfsync 配置项可以有以下 3 种参数可填：

- `Always`：每次写命令执行完，**同步**将 AOF 日志数据写回硬盘；
- `Everysec`：每次写命令执行完，先将命令写入到 AOF 缓冲区，然后每隔一秒将缓冲区里的内容写回到硬盘；
- `No`：由操作系统决定什么时候写回硬盘。

一般我们的项目中会设置为`everysec`。

:::

### 6.什么是AOF重写？

:::note

AOF 日志是一个文件，随着执行的写操作命令越来越多，文件的大小会越来越大。如果当 AOF 日志文件过大就会带来性能问题。

所以，Redis 为了避免 AOF 文件越写越大，提供了<span class="highlighted"> AOF 重写机制</span>，当 AOF 文件的大小超过所设定的阈值后，Redis 就会启用 AOF 重写机制，来压缩 AOF 文件。

AOF 重写机制是在重写时，读取当前数据库中的所有键值对，然后将每一个键值对用一条命令记录到「新的 AOF 文件」，等到全部记录完后，就将新的 AOF 文件替换掉现有的 AOF 文件。

比如，前后执行了`set name zhangsan`和`set name lisi`，在AOF重写时，会读取name最新的value，然后用一条`set name lisi`命令记录到新的AOF文件，这样就相当于对AOF文件进行了压缩。

:::

### 7.重写AOF日志的过程？

![img](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304151127852.png)

:::note

<span class="highlighted">

Redis 的重写 AOF 过程是由后台子进程 `bgrewriteaof` 来完成的。

</span>

触发重写机制后，主进程就会创建重写 AOF 的子进程，此时父子进程共享物理内存，重写子进程只会对这个内存进行只读，重写 AOF 子进程会读取数据库里的所有数据，并逐一把内存数据的键值对转换成一条命令，再将命令记录到重写日志（新的 AOF 文件）。

重写过程中，主进程依然可以正常处理命令。在重写 AOF 期间，当 Redis 执行完一个写命令之后，它会同时将这个写命令写入到 「AOF 缓冲区」和 「AOF 重写缓冲区」。

当子进程完成 AOF 重写工作后，会向主进程发送一条信号。主进程收到该信号后，将 AOF 重写缓冲区中的所有内容追加到新的 AOF 的文件中，并将新的 AOF 的文件进行改名，覆盖现有的 AOF 文件。

:::

### 8.为什么会有混合持久化？

:::note

RDB 优点是数据恢复速度快，但是快照的频率不好把握。AOF 优点是丢失数据少，但是数据恢复不快。混合持久化是集成了两者的优点，既保证了 Redis 重启速度，又降低数据丢失风险。

混合持久化工作在 **AOF 日志重写过程**，当开启了混合持久化时，在 AOF 重写日志时，fork 出来的重写子进程会先将与主线程共享的内存数据以 RDB 方式写入到 AOF 文件，然后主线程处理的操作命令会被记录在重写缓冲区里，重写缓冲区里的增量命令会以 AOF 方式写入到 AOF 文件，写入完成后通知主进程将新的含有 RDB 格式和 AOF 格式的 AOF 文件替换旧的的 AOF 文件。

也就是说，<span class="highlighted">使用了混合持久化，AOF 文件的前半部分是 RDB 格式的全量数据，后半部分是 AOF 格式的增量数据。</span>

:::

## Redis事务

### 1.Redis支持事务吗？

:::note

Redis 支持简单的事务。Redis 事务的本质是一组命令的集合。事务支持一次执行多个命令，一个事务中所有命令都会被序列化。在事务执行过程，会按照顺序串行化执行队列中的命令，其他客户端提交的命令请求不会插入到事务执行命令序列中。

基本流程是用 `multi`开启事务，然后执行一系列命令，这些命令会被放入队列，最后再用 `exec` 提交时批量执行。`discard`命令用于取消一个事务，它会清空事务队列；`watch` 命令用于监视一个或者多个 key，如果这个 key 在事务执行之前被其他命令改动，那么事务将会被打断。

:::

### 2.Redis 事务支持原子性吗？

:::note

Redis具备一定的原子性，但不支持回滚。

Redis 事务可以一次执行多个命令，但是Redis 中并没有提供回滚机制。虽然 Redis 提供了 `DISCARD` 命令。但是，这个命令只能用来主动放弃事务执行，把暂存的命令队列清空，起不到回滚的效果。

:::

### 3.Redis 事务支持ACID吗？

:::note

**Redis 的事务机制可以保证一致性和隔离性，但是无法保证原子性和持久性。**

- 原子性：Redis不支持事务的回滚，因此不能满足标志的原子性；
- 一致性：Redis 的事务执行过程符合一致性要求，即无论命令是否执行成功，都不会破坏内存数据的一致性。
- 隔离性：由于 Redis 执行命令是单线程的，保证了隔离性。
- 持久性：不保证，Redis 事务数据保存在内存中。即使开启 RDB 或 AOF 持久化，由于在事务执行时可能尚未将数据写入磁盘，无法保证数据完全不丢失。

:::

### 4.Redis 事务和MySQL事务有什么区别？

:::note

- MySQL事务具有强原子性，事务中的所有操作要么全部成功，要么全部回滚。Redis的事务原子性不完整，如果事务中某条命令执行失败，其后的命令仍会继续执行，且不支持回滚。
- MySQL支持事务自动回滚；Redis不支持事务回滚。
- MySQL基于 MVCC 和锁机制实现多种隔离级别，保证并发时数据一致；Redis基于队列处理，整体没有复杂的隔离级别概念。

:::

### 5.Redis 事务的改进方案？

:::note

可以使用 Lua 脚本来替代事务，脚本运行期间，Redis 不会处理其他命令，并且我们可以在脚本中处理整个业务逻辑，包括条件检查和错误处理，保证要么执行成功，要么保持最初的状态，不会出现一个命令执行失败、其他命令执行成功的情况。

:::

## 高可用

### 1.Redis的单机模式和集群模式有什么区别？

:::note

- 单机模式所有数据和Redis服务都运行在同一个服务器节点上，比较简单，适合小规模应用，但是没有办法保证高可用以及受到单机内存限制。
- 集群模式则通过多节点实现数据分散，可以实现高可用，适合大规模高并发应用。Redis中的集群包括：主从模式、哨兵模式、分片集群。

:::

### 2.Redis集群有哪些方案？

:::note

在Redis中提供的集群方案总共有三种：

- 主从复制
- 哨兵模式
- 分片集群

主从复制：一般是一主多从，主节点负责写数据，从节点负责读数据，从而实现读写分离，主节点写入数据之后，需要把数据同步到从节点中。

哨兵模式：增加哨兵节点，监控主从服务器，并且提供**主从节点故障转移的功能。**

分片集群：集群中有多个主节点，每个主节点设置多个从节点。

:::

### 3.Redis 主从复制的工作原理？

:::note

Redis 的主从复制是指通过**异步**复制将主节点的数据变更同步到从节点，从而实现数据备份和读写分离。

主从同步分为两个阶段，一个是**全量同步**，一个是**增量同步**。

**全量同步**是指从节点第一次与主节点建立连接的时候使用全量同步，流程是这样的：

1. 从节点请求主节点同步数据，其中从节点会携带自己的`replication id`和`offset`偏移量。
2. 主节点判断是否是第一次请求，主要判断的依据是主节点和从节点是否是同一个`replication id`，如果不是，就说明是第一次同步，那主节点就会把自己的`replication id`和`offset`发送给从节点，让从节点与主节点的信息保持一致。
3. 在同时主节点会执行`bgsave`命令，生成`RDB`文件后，发送给从节点去执行，从节点先把自己的数据清空，然后执行主节点发送过来的`RDB`文件，这样就保持了数据的一致性。

当然，如果在RDB生成执行期间，依然有请求到了主节点，主节点会以命令的方式记录到缓冲区，缓冲区是一个日志文件，最后把这个日志文件发送给从节点，这样就能保证主节点与从节点完全一致了，后期再同步数据的时候，都是依赖这个日志文件，这个就是全量同步。

**增量同步**指的是，当从节点服务重启之后，数据就不一致了，这个时候，从节点会请求主节点同步数据，主节点还是判断是否是第一次请求，不是第一次就获取从节点的`offset`值，然后主节点从命令日志中获取offset值之后的数据，发送给从节点进行数据同步。

:::

>1.主从复制有什么缺点？

:::danger 答案

主从复制保证不了Redis集群的高可用,当主节点宕机之后，集群就丧失了写数据的能力。

为了解决这个问题，Redis提供了哨兵机制来实现主从集群的自动故障恢复。

:::

>2.主从同步有什么风险？

:::danger 答案

Redis主从同步的主要风险包括：

**数据不一致**（因异步复制导致主节点宕机时丢失数据）

**复制风暴**（全量同步失败后反复重试消耗资源）

**数据延迟**（主从同步有网络和处理延迟）

:::

### 5.Redis哨兵机制了解吗？

:::note

Redis的哨兵机制主要是用来实现主从复制集群的自动故障恢复。

哨兵（Sentinel）用于监控主从集群的运行状态，并在主节点故障时自动进行故障转移。

哨兵的核心功能包括**监控、通知和自动故障转移**。哨兵会定期检查主从节点是否按预期工作，当检测到主节点故障时，就在从节点中选举出一个新的主节点，并通知客户端连接到新的主节点。

:::

>简记：哨兵节点主要负责三件事情：监控、选主、通知。

### 6.Redis哨兵的工作原理知道吗？

:::note

哨兵的工作原理可以概括为 3 个关键步骤：<span class="highlighted">定时监控与下线、领导者选举和故障转移。</span>

**定时监控与下线：**

- 首先，哨兵通过心跳机制会定期向主节点和从节点发送 `PING` 命令，以检查它们是否正常工作。
- 如果一个节点在指定时间内没有响应，哨兵会将其标记为“`主观下线`”。
- 多个哨兵会相互交换对节点的判断，如果超过一定数量的哨兵认为某个节点是“`主观下线`”，则会将该节点标记为“`客观下线`”。

**领导者选举：**

- 当主节点被标记为“`客观下线`”后，哨兵集群会选举一个领导者（Leader）来执行故障转移。
- 每个哨兵节点都可以竞选领导者，通过投票机制选出胜者。
- 哨兵节点会向其他哨兵节点发送投票请求，并根据接收到的同意票数来确定是否成为领导者。
- 获得半数以上的同意票且票数不小于配置的法定人数（`quorum`）值的哨兵节点将成为领导者。

**故障转移：**

- 一旦选举出领导者，领导者哨兵将负责执行故障转移操作。
- 领导者会从所有从节点中选择一个合适的节点作为新的主节点。
- 选择新主节点时会考虑：节点的优先级、节点的复制偏移量、节点的运行ID等因素。
- 确定新主节点后，领导者会向新的主节点发送`SLAVEOF NO ONE`命令，使其成为新的主节点。
- 领导者还会向其他从节点发送 `SLAVEOF`命令，使其成为新的主节点的从节点。

最后通过发布/订阅机制通知客户端主节点已经发生变化。

:::

### 8.什么是Redis集群脑裂？该怎么解决？

:::note

Redis集群脑裂指的是：在哨兵模式的Redis集群下，主节点与哨兵、从节点之间的网络发生故障了，但与客户端的连接是正常的。哨兵认为主节点已经下线了，于是会将一个从节点选举为新的主节点。这样就会出现两个“主节点”同时对外提供服务，就像大脑分裂了一样，这样会导致客户端还在旧主节点那里写入数据，新主节点无法同步数据，当网络恢复后，哨兵会将旧主节点降级为从节点，在降级过程中，它需要与新主节点进行全量同步，此时旧主节点的数据会被清空。这就导致了旧主节点中的大量数据丢失。

解决办法：

可以在redis的配置中进行设置：

- 第一，可以设置最少的从节点个数。比如设置至少要有一个从节点才能同步数据；
- 第二，可以设置主从数据复制和同步的延迟时间，达不到要求就拒绝请求，就可以避免大量的数据丢失。

:::

### 9.Redis 分片集群的实现原理是什么？

:::note

Redis分片集群（Redis Cluster ）主要解决的是海量数据存储的问题，集群中有多个主节点，每个主节点保存不同的数据，并且还可以给每个主节点设置多个从节点，就可以继续增大集群的高并发能力。同时每个主节点之间通过`ping`命令监测彼此的健康状态，就类似于哨兵模式。客户端请求可以访问集群中的任意节点，最终都会被转发到正确的节点。

Redis分片集群中数据的存储和读取，是采用哈希槽，<span class="highlighted">一个切片集群共有 16384 个哈希槽</span>，集群中每个主节点绑定了一定范围的哈希槽范围，key通过CRC16校验后对 16384取模来决定放置在哪个槽，通过槽找到对应的节点进行存储。读取也是类似的。

## 分布式锁

### 1.Redis 如何实现分布式锁？

:::note

可以使用 Redis 的 `SETNX` 命令实现简单的分布式锁。比如 `SET key value NX EX 3000` 就创建了一个锁名为 `key` 的分布式锁，锁的持有者为 `value`。NX 保证只有在 key 不存在时才能创建成功，EX 设置过期时间用以防止死锁。

:::

### 2.SETNX有什么问题，如何解决？

:::note

使用 SETNX 创建分布式锁时，虽然可以通过设置过期时间来避免死锁，但会**误删锁**。比如线程 A 获取锁后，业务执行时间比较长，锁过期了。这时线程 B 获取到锁，但线程 A 执行完业务逻辑后，会尝试删除锁，这时候删掉的其实是线程 B 的锁。

解决方案：

可以通过锁的自动续期机制来解决**锁过期的问题**，比如 Redisson 的看门狗机制，在后台启动一个定时任务，每隔一段时间就检查锁是否还被当前线程持有，如果是就自动延长过期时间。这样既避免了死锁，又防止了锁被提前释放。

:::

### 3.基于Redis setnx实现的分布式锁存在的问题？

:::note

基于 setnx 实现的分布式锁存在以下问题：

- 不可重入：同一个线程无法多次获取同一把锁；
- 不可重试：获取锁只尝试一次就返回 false ，没有重试机制；
- 超时释放：锁超时释放虽然可以避免死锁，但如果是业务执行耗时较长，也会导致锁释放，存在安全隐患。
- 主从一致性：如果 Redis 提供了主从集群，主从同步延迟，当主机宕机时，如果未来得及同步到其他机器上，则就会出现多线程获取锁成功情况，从而导致线程安全问题。

:::

### 4.说说 Redisson 分布式锁的原理?

Redisson 分布式锁的核心原理在于利用Redis 的单线程特性和原子操作来保证锁的获取和释放的原子性，以及通过看门狗机制和Lua 脚本来处理锁的续期和释放问题。简单来说，Redisson 通过在Redis 中设置一个key 来表示锁，当一个客户端尝试获取锁时，会使用 `SETNX` 命令（如果key 不存在则设置）来尝试创建该key。如果创建成功，则表示获取到锁。为了防止锁被长时间持有，Redisson 还会设置一个过期时间，并在锁被持有期间通过看门狗机制定时续期，确保锁不会因为持有线程的意外终止而无法释放。释放锁时，Redisson 使用Lua 脚本来保证释放锁操作的原子性，避免了并发问题。

以下是Redisson 分布式锁的详细原理：

1. 获取锁:

- **SETNX 命令:**
- Redisson 使用Redis 的 `SETNX` 命令（SET if Not Exists）来尝试获取锁。如果指定的key 不存在，`SETNX` 命令会设置该key 并返回1，表示获取锁成功。如果key 已经存在，`SETNX` 命令不会执行任何操作并返回0，表示获取锁失败。
- **过期时间:**
- 为了防止锁被长时间持有，Redisson 会为获取到的锁设置一个过期时间。这个过期时间是一个预设的时间值，用于在线程没有正确释放锁的情况下，自动释放锁，避免死锁。
- **看门狗机制(续期):**
- 如果没有设置过期时间，或者锁的过期时间快到了，Redisson 的看门狗机制会启动。看门狗是一个后台线程，会定期检查锁是否还被持有。如果锁仍然被持有，看门狗会向Redis 发送续期请求，延长锁的过期时间，防止锁在持有期间过期。

2. 释放锁:

- **Lua 脚本:**Redisson 使用Lua 脚本来释放锁，确保了释放锁操作的原子性。Lua 脚本可以保证在Redis 服务器上执行一系列操作，而不会被其他操作中断。释放锁的Lua 脚本会先判断当前持有锁的线程是否是释放锁的线程，如果是，则删除锁对应的key，释放锁。

3. 可重入锁:

- **重入计数:**Redisson 支持可重入锁。一个线程可以多次获取同一个锁，而不会被阻塞。为了实现可重入性，Redisson 在Redis 中使用一个hash 数据结构来存储锁的持有信息，每个线程持有锁的次数都会被记录。每次成功获取锁，计数器加1；每次释放锁，计数器减1。只有当计数器为0 时，才会真正释放锁。

4. 死锁检测与恢复:

- **过期时间:**
- Redisson 使用锁的过期时间来避免死锁。当线程获取锁后，如果线程发生异常，或者没有及时释放锁，锁会在过期时间到达后自动释放，避免了死锁的发生。
- **异步续期:**
- Redisson 还会使用异步续期机制，在获取锁成功后，会使用后台线程定期自动续期锁的过期时间，减少因为网络延迟、GC 暂停等问题而造成的误解锁。

总结:

Redisson 分布式锁通过Redis 的 `SETNX` 命令和Lua 脚本，结合看门狗机制和过期时间，实现了分布式环境下锁的获取、释放和续期，保证了锁的正确性和高可用性。同时，可重入锁和死锁检测与恢复机制进一步提升了Redisson 分布式锁的可靠性。

:::note

Redisson 分布式锁的核心原理在于利用Redis 的单线程特性和原子操作来保证锁的获取和释放的原子性，以及通过**看门狗机制**和**Lua 脚本**来处理锁的续期和释放问题，并且支持可重入锁。

我们一般使用Redisson的`getLock`方法获取RLock锁对象，使用`tryLock`方法尝试获取锁，使用`unlock`方法释放锁。

### 5.Redisson 看门狗（watch dog）机制了解吗？

:::note

Redisson 的看门狗机制是一种**自动续期**机制，用于解决分布式锁的过期问题。

看门狗机制可以让我们在获取锁的时候省去手动设置过期时间的步骤，它在内部封装了一个定时任务，每隔 `10 秒`会检查一次，如果当前线程还持有锁就自动续期 `30 秒`。

**基本原理**：当调用 `lock()` 方法加锁时，如果没有显式设置过期时间，Redisson 会默认给锁加一个 30 秒的过期时间，同时启用一个名为“看门狗”的定时任务，每隔 10 秒（默认是过期时间的 1/3），去检查一次锁是否还被当前线程持有，如果是，就自动续期，将过期时间延长到 30 秒。

续期的 Lua 脚本会检查锁的 value 是否匹配当前线程，如果匹配就延长过期时间。这样就能保证只有锁的真正持有者才能续期。

当调用 `unlock()` 方法时，看门狗任务会被取消。或者如果业务逻辑执行完但忘记 unlock 了，看门狗也会帮我们自动检查锁，如果锁已经不属于当前线程了，也会自动停止续期。

这样我们就不用担心业务执行时间过长导致锁被提前释放，也避免了手动估算过期时间的麻烦，同时也解决了分布式环境下的死锁问题。

### 1.Redis 的 ZSet 底层实现原理？

:::note

Zset的底层数据结构是由<span class="highlighted">压缩列表或跳表</span>实现的。

- 如果Zset的元素个数小于`128`个，并且每个元素的值小于 `64` 字节时，Redis 会使用压缩列表作为底层数据结构；
- 否则，Zset会使用跳表作为底层数据结构。
- 在Redis7.0中，压缩列表被废弃，改用`listpack`数据结构来实现。

选择压缩列表作为底层实现时，每个元素会使用两个紧挨在一起的节点来保存：第一个节点保存元素的成员，第二个节点保存元素的Score值。

:::

### 2.跳表了解吗？跳表是怎么实现的？

:::note

跳表是一种数据结构，它是在有序链表的基础上建立了多层索引，最底层包含所有数据，每往上一层，节点数量就减少一半。它的核心思想是"用空间换时间"，通过给链表建立多级索引，跳过大量节点，从而提高查找效率。

每个节点有 50% 的概率只在第 1 层出现，25% 的概率在第 2 层出现，依此类推。查找的时候从最高层开始水平移动，当下一个节点值大于目标时，就向下跳一层，直到找到目标节点。

跳表的查找、插入、删除的平均期望时间复杂度均为O(logN) 。

跳表节点包含分值 score、成员对象 obj、一个后退指针 backward，以及一个层级数组 level。每个层级包含 forward 前进指针和 span 跨度信息。

:::

### 3.跳表在创建节点时候，怎么确定的层数？跳表中为节点确定层数的方法?

:::note

**Redis 跳表在创建节点的时候，随机生成每个节点的层数**。

具体的做法是：

<span class="highlighted">

跳表在创建节点时候，会生成范围为[0-1]的一个随机数，如果这个随机数小于0.25（相当于概率25%），那么层数就增加1层，然后继续生成下一个随机数，直到随机数的结果大于0.25+束，最终确定该节点的层数。

</span>

这样的做法，相当于每增加一层的概率不超过25%，层数越高，概率越低，层高最大限制是64。

:::

### 4.怎么往跳表插入节点？

:::note

首先是找到插入位置，从最高层的头节点开始，在每一层都找到应该插入位置的前驱节点，用一个 update 数组把这些前驱节点记录下来。这个查找过程和普通查找一样，在每层向右移动直到下个节点的值大于要插入的值，然后下降到下一层。

接下来随机生成新节点的层数。通常用一个循环，每次有 50% 的概率继续往上，直到随机失败或达到最大层数限制。

创建新节点后，从底层开始到新节点的最高层，在每一层都进行标准的链表插入操作。这一步要利用之前记录的 update 数组，将新节点插入到正确位置，然后更新前后指针的连接关系。

### 5.ZSet为什么使用跳表不使用B+树？或者红黑树？

:::note

ZSET 选择跳表而非B+树或红黑树是因为跳表在内存中的实现更简单、维护成本低，同时具备良好的范围查找性能，这符合Redis作为内存数据库的设计哲学。

B+树更适合磁盘I/O，而在内存中实现会引入不必要的复杂性；红黑树在范围查找方面不如跳表直观，且实现相对复杂，不如跳表简洁。

<br>

1. 为什么不使用B+树？

- **B+ 树是为磁盘 I/O 优化的**， B+ 树的设计核心是降低树的高度，从而减少磁盘 I/O 次数；而**Redis 是纯内存操作**，在内存中，没有“磁盘 I/O”的瓶颈。
- 跳表本质上是“多层链表”，实现简单；B+树的分裂和合并保持平衡的操作，实现更复杂，增加了开发和维护成本。
- zset 的核心操作（增删改查、范围查询），跳表能以更低的复杂度实现同等甚至更优的性能。
- 跳表更适合内存场景，B+树更适合磁盘场景。

<br>

为什么不使用红黑树？

- 跳表在范围查询时，可以利用其多层链表结构快速定位范围，而红黑树的范围查询需要更复杂的遍历。

- 跳表的插入和删除操作比红黑树的平衡调整（旋转）更简单，需要维护的节点更少。

:::

>1.zset为什么要使用跳表呢？

:::danger 答案

- 跳表天然就是有序的数据结构，查找、插入和删除都能保持 `O(log n)` 的时间复杂度。
- 跳表支持范围查询，找到起始位置后可以直接沿着底层链表顺序遍历，满足Zset的各种操作。

>2.为什么跳表的范围查询效率比字典高？

:::danger 答案

字典是通过哈希函数将键值对分散存储的，元素在内存中是无序分布的，没有任何顺序关系。而跳表本身就是有序的数据结构，所有元素按照分值从小到大排列。

当需要进行范围查询时，字典必须遍历所有元素，逐个检查每个元素是否在指定范围内；而跳表的范围查询就高效的多。

:::

### 6.压缩列表是怎么实现的？

:::note

`压缩列表（ziplist）`是**由连续内存块组成的顺序型数据结构**，有点类似于数组。

整个结构包含头部信息，如总的字节数、尾部偏移量、节点数量，以及连续的节点数据。

压缩列表节点包含三个部分：前一个节点的长度、编码类型和实际的数据。

前一个节点的长度是为了支持从后往前遍历；编码类型会根据数据的实际情况选择最紧凑的存储方式。

压缩列表的缺点是会发生**连锁更新**的问题，当插入或删除节点导致某个节点长度发生变化时，可能会影响后续所有节点存储的“前一个节点长度”字段。因此连锁更新一旦发生，就会导致压缩列表占用的内存空间要多次重新分配，这就会直接影响到压缩列表的访问性能。

因此，压缩列表只会用于保存的节点数量不多的场景，只要节点数量足够小，即使发生连锁更新，也是能接受的。

:::

### 7.quicklist了解吗？

:::note

`quicklist`是List 的底层实现，是一个混合型数据结构，结合了压缩列表和双向链表的优点。主要目的是解决压缩列表的**连锁更新**的问题。

`quicklist`是一个双向链表，链表中的每个节点都是一个 `ziplist`，通过结合链表的高效增删和压缩列表的内存节省特性，有效减少了`ziplist`节点过多导致的连锁更新问题。

:::

### 8.listpack了解吗？

:::note

`listpack`是Redis 7.0中用于替代 `ziplist`（压缩列表）的一种紧凑、有序的内存数据结构，目的是解决  `ziplist`的“连锁更新”问题。

它最大的特点是每个元素只记录自己的长度，不记录前一个元素的长度，通过去掉前一个元素长度的字段，彻底解决了连锁更新的问题和更高效的内存使用。

:::

### 9.哪些原因可能会导致Redis阻塞？

:::note

- **save命令创建RDB快照**：执行`save`命令创建RDB快照是在主线程进行的，可能导致阻塞。 
- **AOF写入文件阻塞**：主线程write操作将AOF缓冲区写入AOF文件可能占用大量I/O，导致阻塞。
- **AOF同步刷盘阻塞**：如果设置为每次都同步刷盘，会导致频繁IO，引发阻塞。
- **AOF重写阻塞**：AOF重写过程中，Redis会将现有数据重新写入AOF文件，这个过程会消耗大量CPU和I/O，可能导致阻塞。
- **bigKey**：如果一个key的数据量特别大，在对其进行查找和删除操作时，Redis可能需要较长时间完成，造成阻塞。
- **清空数据库**：执行清空数据库时，Redis需要删除所有数据，这可能导致长时间阻塞。
- **集群扩容**：在Redis集群扩容时，节点之间的数据迁移会占用大量网络和cpu，导致阻塞。 
- **内存交换**：如果物理内存不足，Redis数据被交换到磁盘，会极大地降低性能，引发阻塞。
- **CPU竞争**：cpu压力较大时，Redis可能与其他进程争夺CPU资源，导致执行效率下降，从而产生阻塞。 

:::

### 10.线上Redis获取缓存的耗时突然变高了，你觉得可能有哪些原因？怎么排查？

**由浅入深分析：**

Redis 是单线程的，因此任何需要长时间处理的命令都会阻塞所有请求。

缓存变慢的常见深层原因通常是：

1. big key 或复杂命令导致事件循环卡住。
2. 主从复制时的内存拷贝也可能造成短暂停顿。
3. 连接池被打爆后客户端排队。
4. CPU 上升导致事件循环处理变慢。
5. 热key 访问量过大，导致 Redis QPS 对单核极限产生压力。

特别是 big key，很容易被忽视，但在真实线上常常是罪魁祸首。

**面试加分点：**

1. 提到 SLOWLOG+ MONITOR 配合排查。
2. 能说出拆分 bigkey 的方案，比如 hash 拆成多个 bucket。
3. 提到线上 Redis 的延迟绝大多数都是外部因素而不是Redis 本身。



:::note

我会把 Redis 缓存变慢的原因分三类：**客户端原因**、**网络原因**、**Redis 服务端原因**，然后按照从外到内的方式排查。

第一，客户端层面可能存在连接池耗尽、请求超时、序列化耗时增加。例如在高峰期如果连接池配置过小，会导致连接等待，整体变慢。

第二，网络层面也很常见，比如带宽不足、RTT增大、跨机房访问或者 Redis 节点迁移后距离变远，这类可以通过监控 RTT 或 traceroute 很快定位。

第三，服务端层面的问题通常最多。我会重点关注以下几类：

- Redis 实例 CPU 飙高，可能是 bigkey、热 key、复杂命令（sort、zrange、keys）导致。
- 慢查询，通过 slowlog 获取耗时命令，看是不是zset 的大范围扫描等。
- big key 访问阻塞：比如一次 HGETALL一个几MB 的hash，会阻塞单线程。
- 内存碎片率过高或内存 nearing maxmemory造成频繁淘汰。
- AOF rewrite 或 RDB save 时会导致短时间卡顿。

<br>

实际排查时我会按照以下步骤进行：

1. 检查 RTT；
2. 检查客户端连接池；
3. Redis CPU, slowlog, bigkey, keyspace hit/miss;
4. 检查是否发生 bgsave 或主从同步。

这样既能快速定位，也能保证间题排查有明确路径。

:::

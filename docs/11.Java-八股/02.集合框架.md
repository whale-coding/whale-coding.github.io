### Java 中有哪些集合类？请简单介绍

![二哥的 Java 进阶之路：Java集合主要关系](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223165456813.png)

:::note 参考答案：

在java中提供了两大类的集合框架，第一大类是Collection接口下的，主要由 List、Set、Queue 组成。

- List 表示有序的集合，存储的元素是有序可重复的，开发中用的比较多的实现类包括 ArrayList和LinkedList；
- Set 表示不可重复的集合，存储的元素是不可重复的，常用的实现类有 HashSet、LinkedHashSet和TreeSet；
- Queue 表示用于保持元素队列的集合，存储的元素是有序可重复的，实现类包括 PriorityQueue、ArrayDeque等；

第二大类是Map接口下的，Map表示键值对的集合，一个键映射到一个值，key 是无序不可重复的，value 是无序可重复的。开发中常用的实现类有HashMap、LinkedHashMap和TreeMap。

:::

## ArrayList

### ArrayList 和 Array（数组）的区别？

:::note 参考答案：

- ArrayList是基于动态数组实现的，会根据实际存储的元素动态地扩容或缩容；Array是静态数组，被创建之后长度不可变。
- ArrayList 允许使用泛型来确保类型安全，而Array 不可以使用泛型。
- ArrayList中只能存储对象，对于基本数据类型需要使用对应的包装类；Array可以直接存储基本类型数据，也可以存储对象。
- ArrayList创建时不需要指定大小，而Array创建时必须指定大小。
- Array只支持下标访问，不支持动态插入删除；ArrayList更灵活，提供丰富的API操作，使用起来更方便。

:::

### ArrayList 和LinkedList 的区别？

:::note 参考答案：

两者的主要区别体现在底层数据结构和使用场景：

- ArrayList 底层是**动态数组**，支持随机访问；LinkedList底层是**双向链表**，不支持随机访问；
- ArrayList只需要存数据，内存占用较少；LinkedList需要额外存两个指针，内存占用相对多；
- ArrayList 更利于查找，LinkedList 更利于增删，因此ArrayList适合查询多，插入删除少的场景；LinkedList适合插入删除频繁、数据量大的场景，比如消息队列，LinkedList 更合适或者作为双端队列或栈使用。

我们在项目中一般很少会使用到 `LinkedList` 的，需要用到 `LinkedList` 的场景几乎都可以使用 `ArrayList` 来代替。

另外，两者都不是线程安全的，如果要在多线程环境下使用，可以使用Collections类的`synchronizedList()`方法将ArrayList包装成线程安全的List或者是使用`CopyOnWriteArrayList`代替。

:::

>扩展面试题1：ArrayList 和LinkedList 哪个更快，为什么？

:::danger 参考答案：

在大多数场景下，ArrayList 比 LinkedList 更快。ArrayList 基于数组，支持O(1)时间的随机访问，且连续内存对 CPU 缓存友好，遍历速度极快。只有在频繁从列表头部插入或删除元素时，LinkedList（基于链表）才表现更佳，因为此时不需要移动后续元素。

:::

>扩展面试题2：ArrayList 插入和删除元素的时间复杂度

:::danger 参考答案：

-  在 ArrayList 的末尾插入元素：O(1) 
-  在 ArrayList 的中间或开头插入元素：O(n) 
- 删除指定位置的元素：O(n)         

:::

>扩展面试题3：ArrayList如何遍历时删除？

:::danger 参考答案：

ArrayList 遍历过程中删除元素，使用 `Iterator.remove()` 或

Java 8 引入的 `Collection.removeIf()`，内部本质也是用迭代器实现的。


<br>

ps：直接在 for-each 循环中删除会引发 `ConcurrentModificationException` 异常，而直接使用索引循环删除则容易导致索引错位而漏删或数组越界。 

:::

### 说一下 ArrayList 的扩容机制？

![三分恶面渣逆袭：ArrayList扩容](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223170448650.png)

当往 ArrayList 中添加元素时，会先检查是否需要扩容，如果当前容量+1 超过数组长度，就会进行扩容。扩容后的新数组长度是原来的 1.5 倍，然后再把原数组的值拷贝到新数组中。

:::note 参考答案：

`ArrayList` 底层是动态数组，**JDK8**以后的版本使用默认构造方法创建的`ArrayList` 是**延迟初始化**，当首次add时，才分配`初始容量10`；指定容量则立即分配。

当 `size` 等于数组长度时触发扩容，**新容量为原容量 1.5 倍**，通过 `Arrays.copyOf` 复制元素。

:::

### 为什么ArrayList的扩容系数是1.5倍？

:::note 参考答案：

**1.5 倍是内存占用与扩容频率的平衡点**。比 2 倍省内存，比 1.2 倍扩容次数少，减少数组拷贝开销。

:::

### 往ArrayList里面添加1w条数据，会发生什么？怎么解决？

:::note 参考答案：

存1万个数据到`ArrayList`中，最主要会发生**多次自动扩容**，导致内存重分配，效率略低。

 `ArrayList` 初始容量默认为 10。存入第11个数据时，容量会增加到原来的1.5倍，也就是15，并创建一个新数组。存入1万个数据，`ArrayList` 会自动扩容多次，涉及旧数据向新数组的复制，会产生短暂的垃圾回收压力。比较耗时，效率低下。

解决办法：建议初始化时设定容量（如 `new ArrayList<>(10000)`）以避免频繁扩容，从而优化性能。 

:::

### ArrayList是线程安全的吗？哪一步会导致线程不安全？

:::note 参考答案：

**ArrayList不是线程安全的**。

多线程环境下，并发修改（特别是同时执行`add()`或`remove()`操作）会导致数据覆盖、丢失或索引越界等问题。

:::

### 有哪几种实现 ArrayList 线程安全的方法？

```java
// Collections.synchronizedList() 方法
SynchronizedList list = Collections.synchronizedList(new ArrayList());

// CopyOnWriteArrayList
CopyOnWriteArrayList list = new CopyOnWriteArrayList();
```

:::note 参考答案：

第一种：使用`Collections`类的`synchronizedList()`方法将ArrayList包装成线程安全的List，内部是通过`synchronized`关键字加锁来实现的。

第一种：使用`CopyOnWriteArrayList`代替ArrayList 。

:::

### ArrayList 和 Vector 的区别？

:::note 参考答案：

- ArrayList不是线程安全的，Vector是线程安全的；
- ArrayList的性能比Vector要高，因为不需要加锁；
- ArrayList的扩容系数是1.5倍，Vector的扩容系数是2倍；
- ArrayList的默认容量在JDK8+是延迟分配10，Vector是直接创建10。

:::

## HashMap

### 1.HashMap的底层原理？

![三分恶面渣逆袭：JDK 8 HashMap 数据结构示意图](https://cdn.paicoding.com/tobebetterjavaer/images/sidebar/sanfene/collection-8.png)

:::note 参考答案：

在JDK1.8之前，HashMap的底层数据结构是`数组+链表`，数组用来存储键值对，链表用来解决哈希冲突。HashMap 通过 key 的`hashcode`经过`hash()`方法计算得到hash值，然后通过`位运算`判断当前元素在数组中存放的位置，如果当前位置存在元素的话，就判断该元素与要存入的元素的hash值和key是否都相等，如果相等的话，直接覆盖，不相等就通过拉链法解决hash冲突。

在JDK1.8及之后，HashMap的底层数据结构是`数组+链表+红黑树`，解决hash冲突时，<span class="highlighted">当链表长度大于8且数组长度大于等于64则会从链表转化为红黑树</span>，目的是提高查询效率。当然，在将链表转换成红黑树前会判断，如果当前数组的长度小于 64，那么会选择先进行数组扩容，扩容后的数组大小是原来的 2 倍，而不是转换为红黑树，目的是减少哈希冲突的发生概率。

:::

>扩展面试题1：解决哈希冲突有哪些方法？

:::danger 参考答案：

我知道的有 3 种，再哈希法、开放地址法和拉链法。

:::

>JDK1.8的HashMap为什么要引入红黑树？

:::danger

JDK 1.8 的 HashMap 引入红黑树的主要原因是**优化哈希冲突严重时的查询性能**。在链表长度超过阈值（默认8）且数组长度达到64时，将冲突的链表转为红黑树，将最坏情况下的时间复杂度从O(n)降低至O(log n) ，有效防止了哈希碰撞攻击导致的系统性能卡死

:::

>为什么不直接用红黑树，还要保留链表？

:::danger

保留链表是因为红黑树节点占用空间是链表的两倍，链表在元素数量较少时查找速度快且空间占用更少。

:::

>HashMap树化阈值为什么是8和64这两个数字？

:::danger

HashMap源码注释中有给出数学依据：

在负载因子0.75的情况下，单个桶里出现8个以上元素的概率只有千万分之六，概率极低。

:::

### 2.HashMap底层为什么使用红黑树而不是二叉树？平衡二叉树？

:::note 参考答案：

红黑树是一种自平衡的二叉查找树，查找、插入、删除的时间复杂度都是 `O(log n)`。

二叉树每个节点最多有两个子节点，容易出现极端情况，比如插入的数据是有序的，那么二叉树就会退化成链表，查询效率就会变成 O(n)。

平衡二叉树比红黑树的要求更高，每个节点的左右子树的高度最多相差 1，这种高度的平衡保证了极佳的查找效率，但在进行插入和删除操作时，可能需要频繁地进行旋转来维持树的平衡，维护成本更高。

:::

>扩展1：为什么HashMap采用红黑树？

:::danger 参考答案：

链表的查找时间复杂度是 `O(n)`，当链表长度较长时，查找性能会下降。红黑树是一种折中的方案，查找、插入、删除的时间复杂度都是 `O(log n)`。

:::

>扩展2：红黑树的特点？

:::danger 参考答案：

红黑树是一种自平衡的二叉查找树：

- 每个节点要么是红色，要么是黑色；
- 根节点永远是黑色；
- 所有的叶子节点都是是黑色的；
- 红色节点的子节点一定是黑色的；
- 从任一节点到其每个叶子的所有简单路径都包含相同数目的黑色节点。

:::

### 3.HashMap 的长度为什么是 2 的幂次方？

:::note 参考答案：

主要目的是为了快速定位元素在底层数组中的下标。

HashMap 是通过 `hash & (n-1)` 来定位元素下标的，n 为数组的大小。

数组长度-1 正好相当于一个“低位掩码”，掩码的低位最好全是 1，这样 & 运算才有意义，否则结果一定是 0。

2 幂次方刚好是偶数，-1 是奇数，这样公式的最后一位就有可能是0/1，取决于 hash 值，这样可以保证哈希值的均匀分布。

:::

  - **JDK 8+**：利用数组容量为2的幂次特性，元素在重新计算索引时，要么在原位置，要么在“原位置+旧容量”的位置。通过低位链表（低位不变）和高位链表（增加旧容量）将数据迁移，保持了链表原有的插入顺序，避免了死循环。

  - 扩容时，如果红黑树中的元素数量较少（

扩容后的数组大小是原来的 `2 倍`，然后把原来的元素重新计算哈希值，放到新的数组中。

:::note 参考答案：

当HashMap中的元素总数超过阈值`容量 * 负载因子`时，触发扩容。HashMap默认初始容量为`16`，负载因子为`0.75`，因此默认在元素数量超过12时开始第一次扩容。扩容规则：新容量为旧容量的`2倍`，且保持容量始终为2的n次幂。

但是在JDK1.7和JDK1.8中，扩容时的数据转移有所区别：

- JDK1.7采用头插法进行数据转移，先扩容再插入数据，在转移过程中，需要为每个元素重新计算 `hashcode` 和索引。
- JDK1.8采用尾插法进行数据转移，先插入数据再扩容，元素新位置的计算只通过新容量下的位运算即可确定是保持原位置还是移到 `原位置 + 旧容量`，效率更高。

:::

>扩展2：JDK7和JDK8的HashMap扩容有什么区别？

- JDK1.7采用**头插法**进行数据转移，先扩容再插入数据，在转移过程中，需要为每个元素重新计算 `hashcode` 和索引。
- JDK1.8采用**尾插法**进行数据转移，先插入数据再扩容，元素新位置的计算只通过新容量下的位运算即可确定是保持原位置还是移到 `原位置 + 旧容量`，效率更高。

:::

### 5.HashMap扩容负载因子为什么是0.75？

HashMap扩容时选择0.75的负载因子是为了在**时间效率**（哈希冲突概率）和**空间效率**（内存占用）之间取得平衡。

0.75是一个经验值，它既能保证哈希冲突的概率较低，提高操作性能，又能避免过度占用内存空间，同时还能与HashMap容量的乘积（用于判断是否扩容的阈值）大概率地保持为整数，提升扩容效率。

:::note 参考答案：

0.75 是 Java 官方在时间与空间成本之间权衡后的经验值，核心目标是**减少哈希冲突**的同时**避免频繁扩容**。

这个数值基于泊松分布统计，能有效减少哈希冲突（链表/红黑树过长），同时避免频繁扩容导致的空间浪费和性能下降。

如果负载因子太高，比如1.0，空间利用率高但会导致大量哈希冲突，导致查询效率低；

如果负载因子太低，比如0.5，数组空闲位置多，冲突极少，但内存利用率太低，而且频繁扩容会损耗性能。

:::

### 6.说一下HashMap的put方法的具体流程？

哈希寻址 → 处理哈希冲突（链表还是红黑树）→ 判断是否需要扩容 → 插入/覆盖节点。

![三分恶面渣逆袭：HashMap插入数据流程图](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223183424548.jpg)

:::note 参考答案：

以JDK8为例，

1. 计算key的hash值并通过hash()方法进一步扰动哈希值，以减少哈希冲突;
2. 判断数组是否初始化，如果没有初始化就进行扩容初始化；
3. 使用哈希值和数组长度进行位运算，确定数组索引；
4. 然后会检查这个位置是否已经有元素：
   1. 如果当前位置为空，直接将键值对插入该位置；
   2. 否则判断当前位置的第一个节点是否与新节点的 key 相同：
      1. 如果相同直接覆盖 value
      2. 如果不同，说明发生哈希冲突
5. 如果是链表，将新节点添加到链表的尾部；如果链表长度大于等于 8，则将链表转换为红黑树；
6. 每插入一个元素，都会检查当前的元素个数是否超过了容量 * 负载因子（默认是 0.75）；
7. 如果超过了阈值，就会进行扩容，扩容为原来的两倍，同时重新计算每个元素的位置并放入新的数组中。

:::

>扩展1：只重写元素的 equals 方法没重写hashCode，put 的时候会发生什么？

:::danger 参考答案：

如果只重写 equals 方法，没有重写 hashCode 方法，那么会导致 equals 相等的两个对象，hashCode 不相等，这样的话，两个对象会被 put 到数组中不同的位置，导致 get 的时候，无法获取到正确的值。

:::

### 8.JDK 8 对 HashMap 做了哪些优化？

:::note 参考答案：

- 底层数据结构由数组 + 链表改成了数组 + 链表+红黑树的结构。
- 链表的插入方式由头插法改为了尾插法。头插法在扩容后容易改变原来链表的顺序。
- 扩容的时机由插入时判断改为插入后判断，这样可以避免在每次插入时都进行不必要的扩容检查，因为有可能插入后仍然不需要扩容。
- 哈希扰动算法也进行了优化。JDK 7 是通过多次移位和异或运算来实现的。JDK 8 让 hash 值的高 16 位和低 16 位进行了异或运算。

:::

### 9.HashMap 是线程安全的吗？

:::note 参考答案： 

HashMap 不是线程安全的。

- 多线程在进行 put 元素的时候，可能会导致元素丢失。因为计算出来的位置可能会被其他线程覆盖掉。
- put 和 get 并发时，可能导致 get 为 null。线程 1 执行 put 时，因为元素个数超出阈值而扩容，线程 2 此时执行 get，就有可能出现这个问题。

:::

### 10.怎么解决 HashMap 线程不安全的问题呢？

:::note 参考答案：

解决办法主要有三种：

- 使用`Hashtable`。Hashtable的每个方法都使用了synchronized加锁，保证了线程安全。
- 通过 `Collections.synchronizedMap` 方法返回一个线程安全的 Map，内部是通过 synchronized 对象锁来保证线程安全的。
- 使用并发工具包下的`ConcurrentHashMap`，它使用CAS+ synchronized 关键字来保证线程安全。

:::

>扩展面试题1：不使用ConcurrentHashMap的情况下，HashMap怎么实现线程安全？

:::danger 参考答案：

首先创建一个普通的HashMap，然后通过`Collections.synchronizedMap()`方法返回一个线程安全的Map，内部是通过 synchronized 对象锁来保证线程安全的。

```java
Map<String, Integer> nonThreadSafeMap = new HashMap<>();
Map<String, Integer> synchronizedMap = Collections.synchronizedMap(nonThreadSafeMap);
synchronizedMap.put("key2", 2);
synchronizedMap.get("key1");
```

:::

## Map

### 1.TreeMap 怎么实现有序的？

:::note 参考答案：

TreeMap是基于红黑树实现的，红黑树是自平衡的二叉查找树，由于二叉查找树的特性，中序遍历红黑树可以得到有序的 Key 序列。

如果 Key 实现了 `Comparable` 接口，TreeMap 默认按 Key 升序排列。

也可以自定义排序，在构造时传入 `Comparator` 对象，按用户指定的逻辑进行排序。

:::

### 2.HashMap和TreeMap区别？

:::note 参考答案：

- HashMap 是基于数组+链表+红黑树实现的，put 元素的时候会先计算 key 的哈希值，然后通过哈希值计算出元素在数组中的存放下标，然后将元素插入到指定的位置，如果发生哈希冲突，会使用链表来解决，如果链表长度大于 8，会转换为红黑树。
- TreeMap 是基于红黑树实现的，put 元素的时候会先判断根节点是否为空，如果为空，直接插入到根节点，如果不为空，会通过 key 的比较器来判断元素应该插入到左子树还是右子树。

:::

### 3.HashMap和LinkedHashMap的区别？

:::note 参考答案：

- LinkedHashMap是HashMap的子类，HashMapd的底层是数组+链表+红黑树，LinkedHashMap在HashMap 的基础上，增加了一条**双向链表**，用来维护元素的插入顺序。
- HashMap 不保证元素的顺序，插入顺序可能和遍历顺序不一致；LinkedHashMap 会按照元素的**插入顺序**或者**访问顺序**来保持遍历时的一致性。

:::

>扩展面试题1：Java 的 Map接口，我想根据插入顺序排序用什么Map实现类比较好？

:::danger 参考答案：

希望 `Map` 能够**按照插入顺序保存和遍历元素**，应该使用 `LinkedHashMap` 。

`LinkedHashMap` 是 `HashMap` 的子类，它内部维护了一个双向链表，用于记录键值对的插入顺序（默认行为）。遍历时，元素会按照插入的顺序返回。

:::

### 4.HashMap和HashTable的区别？

:::note 参考答案：

- Hashtable是线程安全的，HashMap是线程不安全的；
- 由于 Hashtable 的操作都是加了synchronized锁，性能会比 HashMap 差；
- HashMap 允许一个 key 为 null，多个 value 为 null，Hashtable 不允许 key 或 value 为 null，否则会抛 NullPointerException；
- Java8之后，HashMap使用链表和红黑树结合的方式优化冲突处理，而HashTable仅使用链表。
- Hashtable已经是过时的类，不推荐使用。

:::

### 5.HashSet 的底层实现？

:::note 参考答案：

HashSet 是基于 HashMap 实现的，只不过value由一个固定的 Object 对象填充，而key用于操作。

HashSet 会自动去重，因为它是用 HashMap 实现的，HashMap 的key是唯一的，相同key会覆盖掉原来的key，从而实现去重。

:::

### 6.HashSet 怎么判断元素重复？

:::note 参考答案：

HashSet 通过对象的 `hashCode()` 方法和 `equals()` 方法来判断元素是否重复。它首先根据 `hashCode()` 方法的值来查找潜在的存储位置，如果该位置上已存在其他元素，则会调用 `equals()` 方法进行比较；如果 `equals()` 方法返回 `true`，则认为元素重复；如果 `equals()` 返回 `false`，则会将新元素添加到该位置的链中。

:::

## 线程安全集合

### 1.CopyOnWriteArrayList 的实现原理？

![a80a2a0e-eb5e-49e9-8616-056b22c53949](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223185928579.png)

:::note 参考答案：

CopyOnWriteArrayList 是一个**线程安全**的 ArrayList 实现，原理是**写时复制**。

CopyOnWriteArrayList容器**允许并发读**，读操作是无锁的。写操作是**需要加锁**的，CopyOnWriteArrayList 内部有一个`ReentrantLock`锁，写操作时会先**复制**一份原始数组，然后**在新数组上进行修改**，修改完成后，再用新数组替换掉旧数组。

这就确保了在写操作时，其他线程能够读取到旧数组，而不会被阻塞。

CopyOnWriteArrayList 非常适合那种**读操作远多于写操作**的场景。

<br>

简记：**读操作不用加锁，写操作的时候复制一份新数组，改完后再替换老数组**。

:::

### 2.ConcurrentHashMap的实现原理？

:::note 参考答案：

在JDK 1.7中，ConcurrentHashMap采用 `Segment` 分段锁来保证线程安全。它将整个 Map 拆分成了多个 `Segment`，每个 `Segment` 本质上是一个小的 `HashTable`，并且每个 `Segment` 维护一把锁。这样在多线程环境下，只要不同线程操作的是不同的 `Segment`，就可以并发执行。JDK 1.7 最大并发度是 `Segment` 的个数，默认是 `16`。

在JDK 1.8中，**取消了 Segment 分段锁结构，改用了** **CAS** **+ synchronized 来保证并发安全**。

- 数据结构方面，底层变成了 **数组 +** **链表** **+ 红黑树**。当链表长度超过阈值（默认8）并且数组长度足够时，会把链表转为红黑树，提升查询效率。
- 并发控制方面，JDK 1.8 利用 CAS 来实现无锁插入，只有在必要时才使用 synchronized，而且锁的粒度更细，synchronized只锁定链表或红黑树的头节点，进一步减少了锁竞争。所以 1.8 的并发性能比 1.7 更好。

 总结一下就是：

- 1.7 用的是分段锁，锁的是 Segment；
- 1.8 用的是 CAS + synchronized，锁得更细，结构也更优。

:::

### 3.ConcurrentHashMap为什么 key 和 value 不能为null？

:::note 参考答案：

这是因为源码中ConcurrentHashMap的putVal()方法设计的就是当key或value为null时就会抛出NullPointerException。

这样设计的主要原因是，在并发环境下，如果我们允许 ConcurrentHashMap 的 key 或者是 value 为 null 的情况下，就会存在经典的“`二义性问题`”。也就是这个null是值本身设置的是null还是表示没有的意思。

:::

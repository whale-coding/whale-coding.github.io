## 内存管理

### 1.说一下 JVM 的内存区域有哪些？

![JVM内存区域](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306102818289.PNG)

:::note 参考答案： 

 JVM 的内存区域，我们可以把它分成**线程私有**和**线程共享**两大块。

线程私有区域，包括：**程序计数器、 虚拟机栈和本地方法栈**。

- **程序计数器**，也叫**PC寄存器**里面存放的是当前线程正在执行的字节码指令的地址。当多线程切换时，程序计数器能够帮助 JVM 恢复到正确的执行位置。
-  **虚拟机栈**，每个方法在执行时，都会在 Java 栈中创建一个**栈帧**，用于存储**局部变量**表、**操作数栈**、动态链接、方法出口等信息。
- **本地方法栈**，它和虚拟机栈差不多，只不过是给 **native 方法** 服务的。比如调用 C/C++ 的本地方法。

线程共享区域，包括：**堆和方法区。**

- **堆，用来存放对象实例和数组，**我们平时 new 出来的对象，基本上都在这里分配。堆可以细分为**新生代**和**老年代，堆也是垃圾回收的主要区域。**
- **方法区，是一个逻辑概念，**用于存储已被虚拟机加载的**类信息、常量、静态变量、即时编译器（JIT）编译后的代码等。JDK 1.7 以前是永久代，在 JDK 1.8 以后是元空间。**

:::

>1.JVM哪些区域可能发生内存溢出？

:::danger 参考答案： 

除了**程序计数器不会发生内存溢出**以外，其他区域都有可能发生内存溢出。

:::

>1.JVM哪些区域可能发生内存泄露？

:::danger 参考答案： 

堆和元空间可能发生内存泄露。

:::



>2.Java堆内存和栈内存的区别？

:::danger 参考答案： 

- 堆属于线程共享的内存区域，生命周期不由单个方法调用所决定，可以在方法调用结束后继续存在，直到不再被任何变量引用，最后被垃圾收集器回收。
- 栈属于线程私有的内存区域，通常随着方法调用的结束而自动释放，不需要垃圾收集器处理。

:::

>3.变量存在堆栈的什么位置？

:::danger 参考答案： 

局部变量存储在当前方法栈帧中的局部变量表中。

静态变量存储在方法区中，在 Java 7 中是永久代，在 Java8 及以后是元空间。

:::

### 2.说一下 JDK 1.6、1.7、1.8 内存区域的变化？

:::note 参考答案：

- JDK 1.6 使用永久代来实现方法区。
- JDK 1.7 时仍然是永久代，但发生了一些细微变化，比如将字符串常量池、静态变量存放到了堆上。
- 在 JDK 1.8 时，直接在内存中划出了一块区域，叫元空间，来取代永久代，并将运行时常量池、类常量池都移动到了元空间。

:::

>1.为什么使用元空间替代永久代？

:::danger 参考答案： 

永久代会导致 Java 应用程序更容易出现内存溢出的问题，因为它要受到 JVM 内存大小的限制。

元空间的大小不再受到 JVM 内存的限制，只要系统内存足够，就可以一直用。

:::

### 3.Java对象创建的过程？

对象的创建过程，可以用如下图来简要概括：

![1280X1280](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306103528543.PNG)

参考博客：https://www.51cto.com/article/781736.html

:::note 参考答案： 

Java对象创建的过程大概分为5个步骤：<span class="highlighted">类加载检查、分配内存、对象内存初始化、设置对象头、执行`<init>`方法</span>。

1. **类加载检查**：当需要创建一个类的实例对象时，比如通过new xxx()方式，虚拟机首先会去检查这个类是否在常量池中能定位到一个类的符号引用，然后检查这个符号引用代表的类是否已被加载、解析和初始化。如果没有，就先执行类加载流程。
2. **分配内存**：类加载成功后，虚拟机就能够确定对象的大小，此时虚拟机会在堆内存中划分一块对象大小的内存空间出来，分配给新生对象。
3. **对象内存初始化**：JVM 会对分配给对象的这一块内存初始化，也就是给实例对象的成员变量赋上初始值，比如 int 类型赋值为 0，布尔类型赋值为false，引用类型为null等。
4. **设置对象头**：内存初始化之后，虚拟机就会对对象进行必要的设置，比如这个对象是哪个类的实例、如何才能找到类的元数据信息、对象的哈希码、对象的 GC 分代年龄等信息，这些信息都会存放在对象头（Mark Word）中。
5. **执行`<init>`方法**：最后，JVM会执行`<init>`构造方法完成赋值操作，将成员变量赋值为预期的值，比如`int age = 18`，这样一个对象就创建完成了。

### 4.堆内存是如何分配的？什么是指针碰撞？什么是空闲列表？

![1b8f0274-0beb-4a28-a9ce-fa0b6254bca4](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306104044445.png)

:::note 参考答案： 

在堆中为对象分配内存时，主要使用两种方式：<span class="highlighted">指针碰撞法和空闲列表法</span>。

1. **指针碰撞法**：简单来说就是所有用过的内存在一边，空闲的内存在另一边，中间有一个分界指针，分配内存时，只需要将指针向空闲一方移动对象内存大小，如果没有发生碰撞，就将这段内存分配给对象实例。
2. **空闲列表**：虚拟机会维护一个列表，该列表中会记录堆中哪些内存块是可用的，包括内存块大小和地址信息，分配内存时，JVM会遍历空闲列表，找到一块能容纳对象大小的空间，划分给对象实例，并更新空闲列表上的内容。

- 指针碰撞法适用于堆内存规整（即没有内存碎片）的情况，空闲列表法适用于堆内存不规整的情况。

具体选择两种方式中的哪一种，取决于 Java 堆内存是否规整。而 Java 堆内存是否规整，取决于 GC 收集器的算法是"标记-清除"，还是"标记-整理"/"标记-复制"。Serial GC收集器使用的是指针碰撞法，CMS使用的是空闲列表法，G1在年轻代采用复制算法，因此使用指针碰撞法分配内存，在老年代存在内存碎片时会使用空闲列表法。

:::

### 5.JVM 里 new 对象时，堆会发生抢占吗？JVM是怎么设计来保证线程安全的？

:::note 参考答案： 

虚拟机是支持多个线程同时分配内存的，**堆会发生抢占，存在线程安全问题**。

比如用指针碰撞法时，虚拟机正在给对象 A 分配内存，但指针还没来及修改，此时又有一个线程给对象 B 分配内存，同时使用了原来的指针来分配，最后的结果就是这个区域只分配来一个对象，另一个对象被覆盖了。

针对内存分配时存在的线程安全问题，虚拟机采用两种方式来保证线程安全：

- **CAS+失败重试**：CAS 是乐观锁的一种实现方式，JVM采用 CAS 加上失败重试的方式保证更新操作的原子性。以指针碰撞法为例，通过 CAS 操作移动指针，只有一个线程可以移动成功，移动失败的线程重试，直到成功为止。
- **TLAB**：也称为本地线程分配缓冲。JVM会为每一个线程预先在 Eden 区分配一块儿内存，JVM 在给线程中的对象分配内存时，首先在 TLAB 分配，当对象大于 TLAB 中的剩余内存或 TLAB 的内存已用尽时，再采用CAS+失败重试进行内存分配。这样就不会有并发问题了。

:::

### 6.对象一定分配在堆中吗？什么是内存逃逸？

>1.对象一定分配在堆中吗？

:::note 参考答案： 

不一定。默认情况下，Java 对象是在堆中分配的，但 JVM 会进行**逃逸分析**，如果某些方法中的对象引用没有被返回或者没有在方法体外使用，也就是**未逃逸出去**，那么对象可以直接**在栈上分配内存**。

:::

>2.什么是逃逸分析？

:::note 参考答案： 

逃逸分析是一种 **JVM** **优化技术**，用来分析对象的作用域和生命周期，判断对象是否**逃逸出方法或线程**。

可以通过分析对象的**引用流向，判断对象是否被方法返回**、**赋值到全局变量**、**传递到其他线程**等，来确定对象是否逃逸。

**如果对象没有逃逸，就可以进行栈上分配**。

:::

### 7.能说一下对象的内存布局吗？

![eb49b419-4a08-42e0-a00f-5703f62c1713](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306104641484.png)

:::note 参考答案： 

对象的内存布局是由 Java 虚拟机规范定义的，但具体的实现细节各有不同。在 Hotspot 虚拟机中，对象在内存中的布局可以分为 3 块区域：<span class="highlighted">对象头、实例数据和对齐填充</span>。

- **对象头**：对象头是对象存储在内存中的元信息，包含了**Mark Word**、**类型指针**等信息。
  - Mark Word 存储了对象的运行时状态信息，包括锁、哈希值、GC 标记等；【在 64 位操作系统下占 8 个字节】
  - 类型指针指向对象所属类的元数据，也就是 Class 对象。【在 64 位操作系统下占 4个字节】
- **实例数据**：实例数据是对象实际的字段值，也就是成员变量的值。
- **对齐填充**：Hotspot 虚拟机要求对象的起始地址必须是 8 字节的整数倍，如果对象头和实例数据的总长度不是 8 的倍数，JVM 会通过填充额外的字节来对齐。

:::

>1.类型指针会被压缩吗？

:::danger 参考答案： 

类型指针可能会被压缩，以节省内存空间。比如说在开启压缩指针的情况下占 4 个字节，否则占 8 个字节。在 JDK 8 中，压缩指针默认是开启的。

:::

>2.为什么非要进行 8 字节对齐呢？

:::danger 参考答案： 

因为 CPU 进行内存访问时，一次寻址的指针大小是 8 字节，正好是 L1 缓存行的大小。如果不进行内存对齐，则可能出现跨缓存行访问，导致额外的缓存行加载，CPU 的访问效率就会降低。

也就说，8 字节对齐，是为了效率的提高，以空间换时间的一种方案。

:::

>3.new Object() 对象的内存大小是多少？

:::danger 参考答案： 

一般来说，目前的操作系统都是 64 位的，并且 JDK 8 中的压缩指针是默认开启的，因此在 64 位的 JVM 上，`new Object()`的大小是 16 字节（12 字节的对象头 + 4 字节的对齐填充）。

:::

### 8.访问对象的访问方式有哪些？句柄？直接指针？

![图片22222](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306105512394.png)

:::note 参考答案： 

对象访问方式是由虚拟机实现而定的，主流的访问方式主要有<span class="highlighted">使用句柄和直接指针</span>两种。

- **句柄**：如果使用句柄访问， Java 堆中将会划分出一块内存来作为句柄池，reference 中存储的就是对象的句柄地址，而句柄中包含了**对象实例数据**与**对象类型数据**各自的具体地址信息。
- **直接指针**：如果使用直接指针访问，reference 中存储的直接就是对象的地址。

使用句柄来访问的最大好处是 reference 中存储的是稳定的句柄地址，在对象被移动时只会改变句柄中的实例数据指针，而 reference 本身不需要修改。

使用直接指针访问方式最大的好处就是速度快，它节省了一次指针定位的时间开销。

HotSpot虚拟机主要使用直接指针来进行对象访问。

:::

### 9.对象有哪几种引用？各自的使用场景？

![f55d9a8f-33e8-4d9c-a0fc-c0c7db0079a0](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306105704258.png)

:::note 参考答案： 

- 使用 new 关键字赋值的引用就是强引用，只要强引用关联着对象，垃圾收集器就不会回收这部分对象，即使内存不足。
- 软引用通过 SoftReference 类实现。软引用的对象在内存不足时会被回收。
- 弱引用通过 WeakReference 类实现的。弱引用的对象会在下一次垃圾回收时会被回收，不论内存是否充足。
- 虚引用通过 PhantomReference 类实现。虚引用的对象在任何时候都可能被回收。

:::

### 10.JVM中堆的内存分区？

:::note 参考答案： 

在 JDK 7 及之前版本，堆内存被通常分为下面三部分：

1. **新生代**(Young Generation)
   1. 新生代又被划分为 Eden 空间和两个 Survivor 空间（Survivor0 和 Survivor1）。
2. **老年代**(Old Generation)
3. **永久代**(Permanent Generation)

JDK 8 版本之后， **PermGen(永久代) 已被 Metaspace(元空间) 取代**，元空间使用的是本地内存。 

:::

>1.xxxxx

:::danger 参考答案： 

新创建的对象会被分配到新生代的 Eden 空间。当 Eden 区填满时，会触发一次 Minor GC，清除不再使用的对象。

存活下来的对象会从 Eden 区移动到 Survivor 区。

对象在新生代中经历多次 GC 后，如果仍然存活，会被移动到老年代。

当老年代内存不足时，会触发 Major GC，对整个堆进行垃圾回收。

### 11.对象什么时候会进入老年代？

![5c1c5a0f-913d-4fbb-9fb0-9c4433075bf7](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306110112744.png)

:::note 参考答案： 

对象通常会在年轻代中分配，对象主要在以下三种情况下进入老年代：

- **长期存活的对象**：当对象在每次经历Minor GC后仍然存活，它的年龄计数器就会加1。当对象的年龄达到默认的15岁时，它就会被转移到老年代。
- **动态年龄判断**：如果Survivor区域中一批对象的总大小达到该区域容量的50%，那么所有年龄大于或等于这批对象中最大年龄的对象会被直接移到老年代。
- **大对象直接分配**：如果一个对象很大，超过设置的阈值大小，创建时就直接分配到老年代，不会经过新生代。 

:::

### 12.什么是内存泄露？什么是内存溢出？原因？排查方式？

内存溢出是排队去蹲坑，发现没坑了；内存泄漏，就是有人占着茅坑不拉屎，导致坑位不够用。

![fed419d3-f2a5-43b9-a18d-543b2bc88668](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306110216193.png)

>1.什么是内存泄露？什么是内存溢出？

:::note 参考答案： 

- **内存溢出**，俗称 OOM，是指当程序请求分配内存时，由于没有足够的内存空间，从而抛出 `OutOfMemoryError`。
- **内存泄漏**是指程序在使用完内存后，未能及时释放，导致占用的内存无法再被使用。

:::

>2.内存溢出的原因？

:::note 参考答案： 

可能是因为堆、元空间、栈或直接内存不足导致的。

:::

>3.内存泄露的原因？

:::note 参考答案： 

内存泄漏通常是因为长期存活的对象持有短期存活对象的引用，又没有及时释放，从而导致短期存活对象无法被回收而导致的。

- 静态的集合中添加的对象越来越多，但却没有及时清理；
- 单例模式下对象持有的外部引用无法及时释放；
- 数据库、IO、Socket 等连接资源没有及时关闭；
- ThreadLocal 的引用未被清理，线程退出后仍然持有对象引用。

### 13.STW 了解吗？实现原理？

>1.什么是STW？

:::note 参考答案： 

JVM 进行垃圾回收的过程中，会涉及到对象的移动，为了保证对象引用在移动过程中不被修改，必须暂停所有的用户线程，像这样的停顿，我们称之为`Stop The World`。简称 STW。

:::

>2.STW中是如何暂停线程的？

:::note 参考答案： 

JVM 会使用一个名为<span class="highlighted">安全点（Safe Point）</span>的机制来确保线程能够被安全地暂停，其过程包括四个步骤：

- JVM 发出暂停信号；
- 线程执行到安全点后，挂起自身并等待垃圾收集完成；
- 垃圾回收器完成 GC 操作；
- 线程恢复执行。

:::

>3.什么是安全点？

:::note 参考答案： 

安全点是 JVM 的一种机制，常用于垃圾回收的 STW 操作，用于让线程在执行到某些特定位置时，可以被安全地暂停。

安全点通常位于方法调用、循环跳转、异常处理等位置，以保证线程暂停时数据的一致性。

:::

## 类加载机制

### 1.你了解Java的类加载机制吗？

:::note 参考答案： 

JVM 的操作对象是 Class 文件。Java的**类加载器**是JVM中用于动态加载类文件的组件。它将`.class`文件中的字节码加载到内存中，并将其转换为`Class`对象，供JVM执行。

类加载的过程包括：**加载** 、 **验证** 、**准备** 、 **解析**和**初始化**五个步骤。

类加载器在加载类的时候使用的是双亲委派机制，即先委托父加载器尝试加载，只有父加载器无法加载时，子加载器才会加载。

:::

### 2.类的加载（装载）过程？

![3a7398f0-92a4-4bad-85f4-c6dfbe8283b8](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306110912521.png)

:::note 参考答案： 

类加载的过程包括：**加载** 、 **验证** 、**准备** 、 **解析**和**初始化**五个步骤。

- **加载**：将类的二进制字节码加载到内存中。
-  **验证**：检查类文件格式是否符合 JVM 规范。
- **准备**：为类的静态变量分配内存并设置默认值。
- **解析**：将符号引用替换为直接引用。
- **初始化**：执行静态代码块和静态变量初始化。

:::

### 3.类加载器有哪些？

<img src="https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306111207479.png" alt="b901baac-0060-4fdc-8620-e0d1c0b08f6f" style="zoom:50%;" />

:::note 参考答案： 

主要有四种：<span class="highlighted">启动类加载器、扩展类加载器、应用程序类加载器、用户自定义类加载器</span>。

- **启动类加载器**：负责加载 JVM 的核心类库。
- **扩展类加载器**：负责加载Java扩展目录下的jar包和类库。
- **应用程序类加载器**：负责加载用户类路径上的指定类库，是我们平时编写Java程序时默认使用的类加载器。
- **用户自定义类加载器**：开发者根据需求定制类的加载过程，比如加载网络上的类。

:::

### 4.什么是双亲委派机制？为什么要用双亲委派模型？有什么好处？

<img src="https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306111452571.png" alt="c973dee1-b328-4baa-b6b7-caefd4239dcf" style="zoom:80%;" />

:::note 参考答案： 

双亲委派机制是指，<span class="highlighted">类加载器在加载类时，先委托父加载器尝试加载，只有父加载器无法加载时，子加载器才会加载</span>。

原因：

Java的类加载器有一套默认的层级结构，最顶层是启动类加载器，负责加载 JDK 核心类；往下是扩展类加载器，加载 JDK 扩展目录的类；再往下是应用程序类加载器，加载我们自己写的类和第三方 jar 包；我们也可以自定义类加载器，放在最下层。

双亲委派机制的好处：

- **避免类的重复加载，保证类的唯一性**；
- **保证核心类库的安全性**；
- **实现类的复用，核心类只需要被顶层加载器加载一次，所有子加载器都能共享这个类**。

:::

### 5.双亲委派机制可以破坏吗？怎么破坏？

:::note 参考答案： 

可以破坏。

可以重写 `ClassLoader` 的 `loadClass()` 方法来破坏。

比如一些热部署框架就会破坏双亲委派机制。

如果不想打破双亲委派模型，就重写 ClassLoader 类中的 `findClass()` 方法，那些无法被父类加载器加载的类最终会通过这个方法被加载。

:::

### 6.有哪些破坏双亲委派模型的例子？

:::note 参考答案： 

- 第一种：**SPI** **机制加载** **JDBC** **驱动**
  - SPI 是 Java 的一种扩展机制，用于加载和注册第三方类库，常见于 JDBC等框架。双亲委派模型会优先让父类加载器加载类，而 SPI 需要动态加载子类加载器中的实现。
- 第二种：**热部署框架**
  - 热部署是指在不重启服务器的情况下更新应用程序代码，需要替换旧版本的类，但旧版本的类可能由父加载器加载。比如Spring Boot 的 DevTools 通常会自定义类加载器，优先加载新的类版本。
- 第三种：**Tomcat**
  - Tomcat是web容器，那么一个web容器可能需要部署多个应用程序。不同的应用程序可能会依赖同一个第三方类库的不同版本，但是不同版本的类库中某一个类的全路径名可能是一样的。**默认的双亲委派机制，是无法加载多个相同的类，所以，Tomcat破坏双亲委派原则，提供隔离的机制，为每个web容器单独提供一个Web 应用自己定义的类加载器**。

:::

### 7.为什么要有自定义类加载器？打破双亲委派机制后，可能会带来哪些问题？

>1.为什么要使用自定义类加载器？

:::note 参考答案： 

自定义类加载器是为了实现灵活的类加载策略。

- **灵活性和可定制性**：允许开发人员根据具体需求控制类的加载过程，实现更精细的类加载策略。
- **实现类隔离**：在同一个 JVM 中运行多个隔离的应用程序，每个应用程序都可以拥有自己独立的类加载器，避免了类库之间的冲突。
- **动态加载和热部署**：在不重启服务器的情况下，能够加载、卸载和更新类，这在热部署框架中会有所应用。 

:::

>2.打破双亲委派机制的潜在问题？

:::note 参考答案： 

- **类冲突**：如果打破双亲委派机制，不同的类加载器可能加载同一个类的不同版本，导致类版本不一致和冲突。
- **安全性问题**：双亲委派机制通过优先委托给父类加载器来保护核心类库不被恶意覆盖或修改。打破该机制可能会导致恶意代码加载和执行，从而威胁系统安全。

## 垃圾回收

### 1.讲讲 JVM 的垃圾回收机制？

:::note 参考答案： 

垃圾回收就是对内存堆中已经死亡的或者长时间没有使用的对象进行清除或回收。

JVM 在做 GC 之前，会先搞清楚什么是垃圾，什么不是垃圾，通常会通过**可达性分析算法**来判断对象是否存活。

在确定了哪些垃圾可以被回收后，垃圾收集器（如 CMS、G1、ZGC）要做的事情就是进行垃圾回收，可以采用标记清除算法、复制算法、标记整理算法、分代收集算法等。

Java 的垃圾回收过程主要分为：**标记存活对象**、**清除无用对象**、以及**内存压缩/整理**三个阶段。

不同的垃圾回收器在执行这些步骤时会采用不同的策略和算法。

:::

### 2.如何判断对象仍然存活？

![04afe71b-73dc-43f2-9829-7761dacdd57f](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306163500721.png)

:::note 参考答案： 

Java 通过**可达性分析算法**来判断一个对象是否还存活。

通过一组名为 “GC Roots” 的根对象，进行递归扫描，无法从根对象到达的对象就是“垃圾”，可以被回收。

这是 G1、CMS 等主流垃圾收集器使用的主要算法。

:::

### 3.可作为 GC Roots 的引用有哪几种？

:::note 参考答案： 

所谓的 GC Roots，就是一组必须活跃的引用，它们是程序运行时的起点，是一切引用链的源头。

在 Java 中，GC Roots 包括以下几种：

- 虚拟机栈中引用的对象
- 本地方法栈中JNI 引用的对象
- 方法区中常量引用的对象（指常量池中直接引用的对象）
- 方法区中类静态属性引用的对象（指类变量（static变量）所引用的对象）

:::

### 4.对象可以被回收，就代表一定会被回收吗？

:::note 参考答案： 

不一定。

如果对象在进行**可达性分析**后，GC Roots不可达，这个对象将会被第一次标记，随后进行一次**筛选**。筛选的条件是对象是否有必要执行 `finalize()`方法，当对象没有覆盖 `finalize` 方法，或 `finalize` 方法已经被虚拟机调用过时，虚拟机将这两种情况视为没有必要执行，这样的对象才会一定被回收。

有必要执行`finalize()`方法的对象需要二次标记，如果一个对象在`finalize()`方法中"**自救**"了，也就是与引用链上的对象建立了关联，那么就不会被回收。

:::

>1.finalize()方法了解吗？

:::danger 参考答案： 

`finalize()`方法属于`Object` 类，它在对象被垃圾回收之前，由系统自动调用，目的是允许对象在被销毁前完成特定的资源回收工作。

:::

### 5.垃圾收集算法了解吗？

:::note 参考答案： 

垃圾回收（收集）算法主要有四种，分别是<span class="highlighted">标记-清除算法**、**标记-复制算法、标记-整理算法和分代收集算法</span>。

1. **标记-清除算法**包括两个阶段：**标记阶段**会标记所有需要回收的对象，**回收阶段**会回收所有被标记的对象。这个算法的优点是**实现简单**，缺点是**回收过程中会产生内存碎片**。
2. **标记-复制算法**将内存空间**划分为两块**，每次只使用其中一块。当这一块的内存用完了，就将还存活着的对象**复制**到另外一块上面，然后清理掉这一块。这个算法可以解决内存碎片问题，缺点是**浪费了一半的内存空间**。
3. **标记-整理算法**与标记-清除算法类似，标记完之后不是直接对可回收对象回收，而是将存活的对象向内存的一端移动，然后清理边界以外的内存。
4. **分代收集算法**是目前主流的垃圾收集算法，它根据对象存活周期的不同将内存划分为几块，一般分为**新生代和老年代**。新生代用**标记-复制算法**，因为大部分对象生命周期短。老年代用**标记-整理算法**，因为对象存活率较高。

:::

### 6.知道哪些垃圾收集器？

:::note 参考答案： 

JVM 的垃圾回收器主要分为两大类：**分代收集器**和**分区收集器**，分代收集器的代表是 **CMS**，分区收集器的代表是 **G1 和 ZGC**。

1. CMS是一种**以获取最短回收停顿时间为目标**的垃圾回收器，采用**标记-清除算法**，分为**初始标记**、**并发标记**、**重新标记**和**并发清除**四个阶段，优点是**并发收集，停顿时间短**，适合延迟敏感的应用，但容易产生内存碎片，可能触发 Full GC。CMS 垃圾回收器在 **Java 9** 时被标记弃用，**JDK 14** 时被移除。
2. G1是一种**面向大内存、高吞吐**场景的垃圾收集器，它将堆划分为多个小的区域，通过**标记-整理算法**，避免了内存碎片问题。优点是**停顿时间可控，适合大堆场景**，但调优较复杂。G1 在 **JDK 9** 时取代 CMS 成为默认的垃圾回收器。
3. ZGC 是 **JDK 11** 时引入的一款低延迟的垃圾收集器。最大特点是将垃圾回收的**停顿时间控制在** **ms** **级**，适用于**大内存低延迟**的场景。

:::

### 7.详细说一下 CMS 的垃圾收集过程？

:::note 参考答案： 

CMS 使用标记-清除算法进行垃圾收集，分 4 大步：

- **初始标记**：标记所有从 GC Roots 直接可达的对象，这个阶段需要 STW，但速度很快。
- **并发标记**：从初始标记的对象出发，遍历所有对象，标记所有可达的对象。这个阶段是并发进行的。
- **重新标记**：完成剩余的标记工作，包括处理并发阶段遗留下来的少量变动，这个阶段通常需要短暂的 STW 停顿。
- **并发清除**：清除未被标记的对象，回收它们占用的内存空间。

:::

### 8.标记是怎么做的？三色标记法？

![33a45633-9ec4-4280-8fcc-39c5a8af3209](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306164158767.png)

:::note 参考答案： 

三色标记法用于标记对象的存活状态，它将对象分为三类：

- **白色**：表示尚未访问的对象。垃圾回收结束后，仍然为白色的对象会被认为是不可达的对象，可以回收。
- **灰色**：表示已经访问到但未标记完其引用的对象。灰色对象是需要进一步处理的。
- **黑色**：表示已经访问到并且其所有引用对象都已经标记过的对象。黑色对象是完全处理过的，不需要再处理。

<span class="highlighted">三色标记法的工作流程</span>：

- **初始标记**：从 GC Roots 开始，标记所有直接可达的对象为灰色。
- **并发标记**：在此阶段，标记所有灰色对象引用的对象为灰色，然后将灰色对象自身标记为黑色。这个过程是并发的，和应用线程同时进行。
- **重新标记**：重新标记阶段的目标是处理并发标记阶段遗漏的引用变化。为了确保所有存活对象都被正确标记，remark 需要在 STW 暂停期间执行。
- **使用写屏障来捕捉并发标记阶段应用线程对对象引用的更新**：通过遍历这些更新的引用来修正标记状态，确保遗漏的对象不会被错误地回收。

:::

### 9.详细说一下G1 垃圾收集器？

:::note 参考答案： 

G1 是一种面向大内存、高吞吐场景的垃圾收集器，通过标记-整理算法，避免了内存碎片问题。

它的核心思想是：把 Java 堆划分为多个大小相等的独立区域 Region，每个区域都可以扮演新生代或老年代的角色。同时，G1 还有一个专门为大对象设计的 Region，叫 大对象（Humongous） 区。

这种区域化管理使得 G1 可以更灵活地进行垃圾收集，只回收部分区域而不是整个新生代或老年代。

<br>

G1 收集器的运行过程大致可划分为这几个步骤：

- **并发标记**：G1 通过并发标记的方式找出堆中的垃圾对象。并发标记阶段与应用线程同时执行，不会导致应用线程暂停。
- **混合收集**：在并发标记完成后，G1 会计算出哪些区域的回收价值最高（也就是包含最多垃圾的区域），然后优先回收这些区域。这种回收方式包括了部分新生代区域和老年代区域。选择回收成本低而收益高的区域进行回收，可以提高回收效率和减少停顿时间。
- **可预测的停顿**：G1 在垃圾回收期间仍然需要STW。不过，G1 在停顿时间上添加了预测机制，用户可以 JVM 启动时指定期望停顿时间，G1 会尽可能地在这个时间内完成垃圾回收。

:::

>1.G1的分块方式带来了哪些好处？

:::danger 参考答案： 

这种区域化管理使得 G1 可以更灵活地进行垃圾收集，只回收部分区域而不是整个新生代或老年代。

:::

>2.大对象的判定规则是什么？

:::danger 参考答案： 

如果一个大对象超过了一个 Region 大小的 50%，就会被放入 Humongous 中。

### 10.详细说一下ZGC垃圾收集器？

:::note 参考答案： 

ZGC 是 JDK 11 时引入的一款低延迟的垃圾收集器，最大特点是将垃圾收集的停顿时间控制在 10ms 以内，即使在 TB 级别的堆内存下也能保持较低的停顿时间。

它通过并发标记和重定位来避免大部分 Stop-The-World 停顿，主要依赖指针染色来管理对象状态。

适用于需要超低延迟的场景，比如金融交易系统、电商平台。

:::

### 11.CMS和G1的区别？

![b2df1391-0cbd-4752-86fc-97d58900d3f7](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260306164733341.png)

:::note 参考答案： 

- CMS使用标记-清除算法进行垃圾收集；G1 使用标记-整理算法进行垃圾回收。
- CMS 适用于对延迟敏感的应用场景，主要目标是减少停顿时间，但容易产生内存碎片。
- G1 则提供了更好的停顿时间预测和内存压缩能力，适用于大内存和多核处理器环境。

:::

### 12.垃圾收集器应该如何选择？

:::note 参考答案： 

- **低延迟优先**的应用程序，可以采用**CMS**，它会尽量减少停顿时间，但会消耗更多 CPU 资源，且可能产生内存碎片。
- 对**吞吐量要求高、大内存、多核**的场景下，可以采用**G1**，它能有效管理大堆内存并兼顾吞吐量和延迟。
- 对于**超大内存、超低延迟要求**的场景，可以采用**ZGC**。

:::

### 13.Minor GC、Major GC、Full GC 都是什么意思？

:::note 参考答案： 

- Minor GC 也称为 Young GC，是指发生在年轻代的垃圾收集。年轻代包含 Eden 区以及两个 Survivor 区。
- Major GC 也称为 Old GC，主要指的是发生在老年代的垃圾收集。是 CMS 的特有行为。
- Full GC 是最彻底的垃圾收集，涉及整个 Java 堆和方法区。它是最耗时的 GC，通常在 JVM 压力很大时发生。

:::

### 14.Full GC 和 Young GC的区别？

:::note 参考答案： 

- Young GC仅回收新生代中的对象；Full GC回收整个堆内存，包括新生代和老年代。
- Young GC在新生代内存不足时触发；Full GC在堆内存不足、显式调用 **`System.gc()`** 时触发。
- Young GC回收快速，频繁发生；Full GC 涉及整个堆的回收，通常会停顿较长时间。

:::

### 15.什么时候会触发Young GC ？什么时候会触发 Full GC？

>1.Young GC（Minor GC）的触发条件?

:::note 参考答案： 

- 新生代没有足够的空间分配新对象时。

:::

>2.Full GC的触发条件?

:::note 参考答案： 

- 老年代空间不足；
  - 当老年代没有足够的空间容纳由 Minor GC 晋升过来的对象时，会触发 Full GC。
- 显式调用 `System.gc()`；
  - 开发者可以显式地调用 `System.gc()` 方法来建议 JVM 进行垃圾回收，这可能会触发 Full GC。
- 元空间的空间不足；
  - 当元空间的空间不足以加载新的类元信息时，会触发 Full GC。
- 发生 Heap Dump。
  - 当为了排查问题需要生成堆转储文件（Heap Dump）时，也会触发 Full GC。 

:::

### 17.你们线上用的什么垃圾收集器？什么垃圾收集算法？

:::note 参考答案： 

我们生产环境中采用的 G1 垃圾收集器，因为它不仅能满足低停顿的要求，而且解决了 CMS 的内存碎片问题，非常适合大内存、多核处理器的环境。

G1 采用的是分区式标记-整理算法，将堆划分为多个区域，按需回收，适用于大内存和多核环境，能够同时考虑吞吐量和暂停时间。

:::

## JVM调优

### 1.JVM 的常见参数配置知道哪些？

:::note 参考答案： 

- `-Xms`：设置初始堆大小
- `-Xmx`：设置最大堆大小
- `-XX:NewSize=n`：设置年轻代大小
- `-XX:NewRatio=n`：设置年轻代和年老代的比值
- `-XX:SurvivorRatio=n`：年轻代中 Eden 区与两个 Survivor 区的比值
- `-XX:+UseG1GC`：使用 G1 垃圾收集器
- `-XX:MaxGCPauseMillis=n` 设置最大垃圾回收停顿时间
- `-XX:+PrintGC`：输出 GC 日志
- `-XX:+PrintGCDetails` 输出 GC 详细日志

:::

### 2.服务器整个CPU占用过高，如何排查？

:::note 参考答案： 

首先，使用 **top 命令查看 CPU 占用情况**，找到占用 CPU 较高的**进程 ID**；

接着，使用 **jstack** **命令**查看对应进程的**线程堆栈**信息并输出到文件中；

然后再使用 **top 命令**查看**进程中线程的占用情况**，找到占用 CPU 较高的**线程 ID**；

接着在 **jstack** **的输出文件**中搜索这个**十六进制的线程 ID**，找到对应的**堆栈信息**；

最后，根据堆栈信息**定位到具体的业务方法**，查看是否有死循环、频繁的垃圾回收、资源竞争导致的上下文频繁切换等问题。

:::

### 3.内存飙高问题怎么排查？

:::note 参考答案： 

内存飚高一般是因为创建了大量的 Java 对象导致的，如果持续飙高则说明垃圾回收跟不上对象创建的速度，或者内存泄漏导致对象无法回收。

排查的方法主要分为以下几步：

- 第一，先观察垃圾回收的情况，可以通过 `jstat -gc PID 1000` 查看 GC 次数和时间。或者使用 `jmap -histo PID | head -20` 查看堆内存占用空间最大的前 20 个对象类型。
- 第二步，通过 jmap 命令 dump 出堆内存信息。

第三步，使用可视化工具分析 dump 文件，比如说 VisualVM，找到占用内存高的对象，再找到创建该对象的业务代码位置，从代码和业务场景中定位具体问题。

:::

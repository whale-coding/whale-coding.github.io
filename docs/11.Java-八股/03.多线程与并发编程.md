## 多线程

### 1.进程和线程的区别？

:::note 参考答案： 

- 进程是资源分配的最小单位，线程是CPU调度的最小单位；
- 进程在执行过程中拥有独立的内存单元，而多个线程共享进程的内存；
- 一个线程只能属于一个进程，而一个进程可以有多个线程；
- 进程切换的开销远大于线程切换的开销；

:::

### 2.线程有几种创建方式？

:::note 参考答案： 

线程有四种创建方式，分别是继承 Thread 类、实现 Runnable 接口、实现 Callable 接口以及使用线程池。

- 继承Thread 类，需要重写Thread 的 `run()` 方法，并且调用 `start()` 方法启动线程。
- 实现 Runnable 接口，需要重写 Runnable 接口的 `run()` 方法，并将实现类的对象作为参数传递给 Thread 对象的构造方法，最后调用`start()`方法启动线程。
- 实现 Callable 接口，需要重写 Callable 接口的`call()`方法，然后创建 FutureTask 对象，参数为 Callable 实现类的对象；紧接着创建 Thread 对象，参数为 FutureTask 对象，最后调用`start()`方法启动线程。
- 使用线程池，使用 ExecutorService 来管理线程，可以复用线程，避免频繁地创建和销毁线程。

:::

### 3.实现Runnable接口和实现Callable接口两种方式有什么区别？

:::note 参考答案： 

实现 Callable 接口创建的线程可以获取线程的执行结果，实现Runnable 接口的不能获取线程的执行结果。

:::

### 4.启动一个线程是 run()还是 start()?

:::note 参考答案： 

启动一个线程应该使用 `start()` 方法，而不是直接调用 `run()` 方法。

`start()` 方法会启动一个新的线程，并在这个新线程中执行 `run()` 方法中的代码，实现多线程并发执行。

如果直接调用 `run()` 方法，那只是一个普通的同步方法调用，它会在当前线程中执行，不会创建新的线程。

:::

>扩展面试题1：启动一个 Java 程序，你能说说里面有哪些线程吗？

::: danger 参考答案： 

- main 线程，这是程序执行的入口;
- 垃圾回收线程，它是一个后台线程，负责回收不再使用的对象;
- 编译器线程，比如 JIT。

:::

### 5.run()方法和start()方法有什么区别？

:::note 参考答案： 

两个方法的主要区别在于：

当程序调用start()方法时，会创建一个新的线程，在run()方法中的代码将会在新线程上运行；

直接调用run()方法，程序并不会创建新线程，run()方法内部的代码将在当前线程上运行。

:::

### 6.sleep 和 wait 的区别？

:::note 参考答案： 

- sleep 会让当前线程休眠，不需要获取对象锁，属于 Thread 类的方法；
- wait 会让获得对象锁的线程等待，要提前获得对象锁，属于 Object 类的方法；
- `sleep()`方法可以在任何地方被调用；`wait()`方法必须在同步代码块或同步方法中被调用；
- 调用 sleep 方法后，线程会进入 TIMED_WAITING 状态，当指定的时间结束后，线程会自动恢复到 RUNNABLE 状态，等待 CPU 调度再次执行。调用 wait 方法后，线程会进入 WAITING 状态，直到有其他线程在同一对象上调用 notify 或 notifyAll 方法，线程才会从 WAITING 状态转变为 RUNNABLE 状态，准备再次获得 CPU 的执行权。

:::

### 7.Thread类里面run()、start()、join()方法分别有什么作用？

:::note 参考答案： 

- run()方法：是线程的任务方法，定义线程要执行的代码。但直接调用 run() 不会启动新线程，只是普通方法调用。

- start()方法：会启动线程，创建新的线程并执行run()方法。

- join()方法：会让当前线程等待目标线程执行完成，再继续往下执行。例如，在主线程中创建了一个子线程，并且在主线程的后续代码需要等待子线程完成后才能执行，就可以在主线程中调用子线程的 `join()` 方法。


:::

### 8.调用join()方法会有什么现象？

:::note 参考答案： 

我们在主线程里面调用子线程的join()方法，主线程会一直等待，直到子线程运行完毕之后，主线程才接着运行。

:::

### 9.wait方法和join方法有什么区别？

:::note 参考答案： 

- **使用目的不同**：`wait()` 用于线程间通信，需要配合 `notify()` 或 `notifyAll()` 唤醒，而 `join()` 用于线程的执行顺序，使一个线程等待另一个线程执行完毕。
- **所属类不同**：`wait()` 在 `Object` 类中声明，而 `join()` 在 `Thread` 类中声明。
- **锁的释放不同**：`wait()` 会释放当前线程持有的锁，而 `join()` 方法不释放任何锁。
- **唤醒方式不同**：`wait()` 状态的线程可通过 `notify()` 或 `notifyAll()` 被其他线程唤醒，而 `join()` 方法的等待只能通过被调用线程的执行结束或中断来解除。

:::

### 10.怎么停止一个线程，interrupt和 stop 区别？

:::note 参考答案： 

- 可以通过`interrupt()`方法来通知线程停止，但不会直接终止线程，需要线程自行处理中断标志。
- 通过`stop() `方法用来强制停止线程。

:::

>扩展面试题1：如何强制终止线程？

:::danger 参考答案： 

第一步，调用线程的 `interrupt()` 方法，请求终止线程；

第二步，在线程的 `run()` 方法中检查中断状态，如果线程被中断，就退出线程。

:::

>扩展面试题2：说说interrupt方法？

:::danger 参考答案： 

`interrupt()` 方法用于通知线程停止，但不会直接终止线程，需要线程自行处理中断标志。

:::

>扩展面试题3：说说yield方法？

:::danger 参考答案： 

`yield()` 方法的目的是让当前线程让出 CPU 使用权，回到就绪状态。但是线程调度器可能会忽略。

:::

### 11.线程有几种状态？状态之间是如何转换的？

![684c7c28-6d59-4a6e-b428-eee36ccee964](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260223192053007.png)

:::note 参考答案： 

线程有6种状态。

分别为：`NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED`。

- `NEW`：代表线程被创建但未启动；
- `RUNNABLE`：代表线程处于就绪或正在运行状态，由操作系统调度；
- `BLOCKED`：代表线程被阻塞，等待获取锁；
- `WAITING`：代表线程等待其他线程的通知或中断；
- `TIMED_WAITING`：代表线程会等待一段时间，超时后自动恢复；
- `TERMINATED`：代表线程执行完毕，生命周期结束。

**状态流转：**

- 当线程被创建后，比如通过`new Thread()` ,它处于新建状态。
- 当调用线程的`start()`方法后，线程进入可运行状态。当线程获得CPU后就会运行。
- 当线程进入同步代码块需要获取锁时，锁被其他线程持有，线程进入阻塞状态。
- 当调用wait()方法时，会进入无线等待状态。
- 当调用sleep()方法/join()方法/wait(timeout)会进入限时等待状态。
- 当线程执行完毕，会进入终止态。

:::

### 12.线程间有哪些通信方式？

:::note 参考答案： 

线程之间传递信息的方式有多种，比如说使用 volatile 和 synchronized 关键字共享对象、使用 wait() 和 notify() 方法实现生产者-消费者模式、使用 Exchanger 进行数据交换、使用 Condition 实现线程间的协调等。

:::

### 13.了解虚拟线程吗？它的调度原理是什么？用的是Java的什么东西？

:::note 参考答案： 

虚拟线程是Java 21正式引入的轻量级线程，由JVM而非操作系统管理。它们解决了传统平台线程创建昂贵、数量受限的问题，实现了在阻塞式I/O场景下极高的并发量.其调度原理是通过调度器（Executor）将大量虚拟线程映射到少量操作系统线程（Carrier Threads）上，利用ForkJoinPool进行工作窃取。

1. 什么是虚拟线程（Virtual Threads）？

- **轻量级：** 虚拟线程的内存占用非常小（KB级别），且在用户空间中由JVM创建和销毁，不需要在操作系统中创建真正的内核线程。
- **高并发：** 可以轻松创建数百万个虚拟线程，不再受限于OS的线程数上限。
- **阻塞式：** 虚拟线程在执行阻塞操作（如网络I/O或数据库操作）时，会被JVM自动分离，释放出底层的内核线程去执行其他任务。 

2. 调度原理

虚拟线程的调度分为两个层面：

- **M:N映射：** 虚拟线程（M）映射到操作系统线程（N，通常小于M）。执行虚拟线程的OS线程被称为**“载体线程”（Carrier Thread）**。
- **调度器（Scheduler）：** Java运行时使用一个先进先出（FIFO）的模式的调度器。当虚拟线程被提交执行时，调度器将其调度到一个载体线程上。
- **上下文切换与挂起：** 当虚拟线程发生阻塞（例如在执行 `InputStream.read()` 时），JVM会进行挂起操作：
  1. 保存虚拟线程的栈帧（Context）到JVM堆内存中。
  2. 载体线程被释放，能够挂载其他的虚拟线程。
  3. 阻塞操作完成后，调度器将该虚拟线程再次挂载到空闲的载体线程上继续执行。
- **核心组件：** 默认使用 `ForkJoinPool` 作为调度器，并采用工作窃取（Work-Stealing）算法实现高效率的利用率。

3. 用的是Java的什么东西？

虚拟线程的核心设计基于以下几个组件：

### 14.如何控制某个方法允许的并发访问线程的数量？

:::note 参考答案： 

可以使用 `Semaphore` (信号量) 来实现。`Semaphore` 维护一组许可（permits），每个线程在访问方法前需要先获取一个许可，在访问结束后释放许可。通过初始化 `Semaphore` 时设置的许可数量，可以限制同时进入该方法的线程数量。

### 15.什么是线程安全？怎么保证线程安全？

:::note 参考答案： 

1.什么是线程安全？对线程安全的理解？

线程安全是指在并发环境下，多个线程访问共享资源时，程序能够正确地执行，而不会出现数据不一致的情况。

如果一段代码块或者一个方法被多个线程同时执行，还能够正确地处理共享数据，那么这段代码块或者这个方法就是线程安全的。

<br>

2.怎么保证线程安全？

可以从三个要素来确保线程安全：

- **原子性**：一个操作要么完全执行，要么完全不执行，不会出现中间状态。
- **可见性**：当一个线程修改了共享变量，其他线程能够立即看到变化。
- **有序性**：要确保线程不会因为死锁、饥饿、活锁等问题导致无法继续执行。

在Java中保证线程安全主要有以下几种方式：

- 使用synchronized 关键字对方法加锁，对代码块加锁；
- 使用ReentrantLock 并发重入锁对代码块进行加锁；
- 使用volatile 关键字保证变量的内存可见性；
- 对于简单的原子变量操作，可以使用Atomic 原子类；
- 对于线程独立的数据，可以使用ThreadLocal来为每个线程提供专属的变量副本；
- 对于需要并发容器的地方，可以使用ConcurrentHashMap、CopyOnWriteArrayList等线程安全的容器。

:::

>扩展面试题1：说一个线程安全的使用场景？

:::danger 参考答案： 

**单例模式**。在多线程环境下，如果多个线程同时尝试创建实例，单例类必须确保只创建一个实例，并提供一个全局访问点。

- 饿汉式是一种比较直接的实现方式，它通过在类加载时就立即初始化单例对象来保证线程安全。
- 懒汉式单例则在第一次使用时初始化单例对象，这种方式需要使用双重检查锁定来确保线程安全，volatile 关键字用来保证可见性，syncronized 关键字用来保证同步。

:::

>扩展面试题2：说一个线程不安全的例子？

:::danger 参考答案： 

 售票系统例子：

- **场景**: 一个售票系统有1000张票，多个用户线程同时购买。
- **不安全行为**: 如果系统不是线程安全的，多个线程可能会同时执行减1操作。例如，两个线程同时读到还有10张票，它们都以为自己成功购买了一张，各自执行减1操作，结果是系统依然记录还有9张票，而实际上卖出去了两张。
- **结果**: 这会导致数据不一致或“脏数据”，系统无法准确地知道剩余的票数。

:::

>扩展面试题3：线程安全的队列有哪些？

:::danger 参考答案： 

线程安全队列主要分为**阻塞队列**（`BlockingQueue`，适用于生产者-消费者场景）和**非阻塞队列**（基于CAS，适用于高并发场景）。

核心的线程安全队列包括：`LinkedBlockingQueue`、`ArrayBlockingQueue`、`ConcurrentLinkedQueue`、`PriorityBlockingQueue`、`DelayQueue` 和 `SynchronousQueue`。

非阻塞队列通常使用 CAS 等无锁算法，适合高并发读写场景；阻塞队列则提供阻塞操作，常用于生产者-消费者模式。 

:::

## 线程池

### 1.什么是线程池？有什么好处？

:::note 参考答案： 

线程池是用来管理和复用线程的工具，线程池的主要目的是为了避免频繁地创建和销毁线程。

使用线程池可以减少线程的创建和销毁开销。

:::

### 2.线程池的核心参数？线程池参数之间的关系？

:::note 参考答案： 

线程池有7个参数，分别为：核心线程数、最大线程数、等待队列、拒绝策略、线程工厂、非核心线程存活时间、非核心线程存活时间单位。

线程池参数之间的关系：

任务优先使用核心线程执行，满了进入等待队列，队列满了启用非核心线程备用，线程池达到最大线程数量后触发拒绝策略，非核心线程的空闲时间超过存活时间就被回收。

:::

### 3.线程池的拒绝策略有哪些？

:::note 参考答案： 

有4种拒绝策略，分别是：

- **中止策略**：默认的拒绝策略，会直接抛出RejectedExecutionException异常；
- **调用者运行策略**：将任务退回给提交任务的线程，让提交任务的线程自己来执行这个任务；
- **丢弃策略**：丢弃被拒绝的任务，不做任何处理也不抛出异常；
- **丢弃最旧任务策略**：会丢弃等待队列中等待最久的任务，然后尝试重新提交被拒绝的任务。

当然，我们也可以通过实现 `RejectedExecutionHandler 接口`来自定义淘汰策略。

:::

>扩展面试题：调用者运行策略这种方式有什么缺点？

:::danger 参考答案： 

核心缺点是**导致任务提交线程（通常是主线程或请求处理线程）在执行任务时被阻塞**，从而降低系统整体吞吐量、显著增加请求响应延迟，甚至因过载导致级联故障，不适用于响应时间敏感的场景。

- **阻塞提交线程，系统变慢：** 当线程池满时，该策略由调用者（即发起 `execute` 或 `submit` 的线程）亲自执行该任务。如果任务耗时较长，会导致调用者无法立即处理下一个请求，整个系统响应速度下降。
- **引发级联故障风险：** 若在高并发环境下使用，调用者在执行任务时，其自身处理其他请求的能力会丧失，导致队列继续堆积。如果调用者是Tomcat的HTTP线程，会导致前端请求大量堆积，甚至引发系统瘫痪。
- **丧失并发优势：** 虽然通过执行任务避免了丢弃任务，但该任务本身并没有利用线程池的后台线程进行异步处理，而是“退化”为同步执行，失去了使用线程池的意义。
- **不易监控与排查：** 由于任务是由调用者而非线程池线程处理的，可能会导致任务的处理日志、性能监控数据与预期的线程池状态不一致，增加排查难度。

:::

### 4.线程池的工作流程？

<img src="https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260326193935114.png" alt="c4408279-6978-4dfa-aa92-55e156aa30ce" style="zoom:50%;" />

:::note 参考答案： 

可以简单总结为：任务提交 → 核心线程执行 → 任务队列缓存 → 非核心线程执行 → 拒绝策略处理。

- 线程池通过 `submit()` 提交任务；
- 线程池会先创建核心线程来执行任务；
- 如果核心线程都在忙，任务会被放入任务队列中；
- 如果任务队列已满，且当前线程数量小于最大线程数，线程池会创建新的线程来处理任务；
- 如果线程池中的线程数量已经达到最大线程数，且任务队列已满，线程池会执行拒绝策略。

:::

### 5.线程池的种类有哪些？

:::note 参考答案： 

主要有4种，分别为：固定大小的线程池、缓存线程池、定时任务线程池、单线程线程池。

- newFixedThreadPool：固定大小的线程池。
  - 适合用于任务数量确定，且对线程数有明确要求的场景。例如，IO 密集型任务、数据库连接池等。
- newCachedThreadPool：缓存线程池。
  - 适用于短时间内任务量波动较大的场景。例如，短时间内有大量的文件处理任务或网络请求。
- newScheduledThreadPool：定时任务线程池。
  - 适用于需要定时执行任务的场景。例如，定时发送邮件、定时备份数据等。
- newSingleThreadExecutor：单线程线程池。
  - 适用于需要按顺序执行任务的场景。例如，日志记录、文件处理等。

:::

:::note 参考答案： 

常用的有5种，分别是：有界队列、无界队列、优先级队列、延迟队列和同步队列。

### 7.为什么不建议用 Executors 创建线程池？

:::note 参考答案： 

`Executors` 创建的线程池，可能出现**内存溢出 (OOM)** 的风险。

- FixedThreadPool 和 SingleThreadPool:允许的请求队列长度为 Integer.MAX_VALUE，可能会堆积大量的请求，从而导致 OOM。
- CachedThreadPool 和 ScheduledThreadPool:允许的创建线程数量为 Integer.MAX_VALUE，可能会创建大量的线程，从而导致 OOM。

:::

### 8.如何确定核心线程数量？

:::note 参考答案： 

首先，我会分析线程池中执行的任务类型是 CPU 密集型还是 IO 密集型。

1. 对于 CPU 密集型任务，目标是尽量减少线程上下文切换，以优化 CPU 使用率。一般来说，核心线程数设置为**CPU核心数+1**是较理想的选择。
2. 对于 IO 密集型任务，由于线程经常处于等待状态，等待 IO 操作完成，所以可以设置更多的线程来提高并发，比如说 **CPU 核心数的两倍**。

这个只是理论上的配置，可以先根据理论设置一个值，之后进行**压测**以及**结合实际业务需求和系统资源**进行调整，最终达到一个相对合理的值。

:::

### 9.线程池的提交 execute 和 submit 有什么区别？

:::note 参考答案： 

线程池主要通过 `execute()` 和 `submit()` 两种方法提交任务，区别如下：

- `execute()` 方法没有返回值，用于提交无返回值的Runnable任务，适用于不关心结果和异常的简单任务。
- `submit()`方法有返回值，可以提交Runnable或Callable任务，并返回一个Future对象，允许获取任务的执行结果和处理异常，适用于需要获取结果或处理异常的场景。

### 11.线程池怎么关闭？

:::note 参考答案： 

可以调用线程池的`shutdown`或`shutdownNow`方法来关闭线程池。

- shutdown 不会立即停止线程池，而是等待所有任务执行完毕后再关闭线程池
- shutdownNow 会尝试通过一系列动作来停止线程池，包括停止接收外部提交的任务、忽略队列里等待的任务、尝试将正在跑的任务 interrupt 中断。

但是，shutdownNow 不会真正终止正在运行的任务，只是给任务线程发送 interrupt 信号，任务是否能真正终止取决于线程是否响应 InterruptedException。

## Java内存模型

### 1.什么是Java内存模型（JMM）？

<img src="https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260326185810633.png" alt="a3e52c2d-e3c9-4e12-ab96-81823e9ccd92" style="zoom:50%;" />

:::note 参考答案： 

Java 内存模型是 JVM规范中定义的一个抽象模型，用来描述**多线程环境中共享变量的内存可见性**。

共享变量存储在`主内存`中，每个线程都有一个私有的`本地内存`，存储了共享变量的副本。

- 当一个线程更改了本地内存中共享变量的副本，它需要 JVM 刷新到主内存中，以确保其他线程可以看到这些更改。
- 当一个线程需要读取共享变量时，它会先从本地内存中读取。如果本地内存中的副本是过时的，JVM 会将主内存中的共享变量最新值刷新到本地内存中。

:::

### 2.volatile 了解吗？使用场景是什么？

:::note 参考答案： 

volatile是Java中的一个关键字，主要作用有两个：

- 第一，**保证可见性**，线程修改 volatile 变量后，其他线程能够立即看到最新值；
- 第二，**禁止指令重排**，volatile 变量的写入不会被重排序到它之前的代码。

:::

### 3.volatile 怎么保证可见性的？

:::note 参考答案： 

- 当线程对 volatile 变量进行写操作时，JVM 会在这个变量写入之后插入一个写屏障指令，这个指令会强制将本地内存中的变量值刷新到主内存中。
- 当线程对 volatile 变量进行读操作时，JVM 会插入一个读屏障指令，这个指令会强制让本地内存中的变量值失效，从而重新从主内存中读取最新的值。

:::

### 4.volatile 怎么保证有序性的？你了解的有哪些内存屏障？

:::note 参考答案： 

JVM 会在 volatile 变量的读写前后插入 “**内存屏障**”，以约束 CPU 和编译器的优化行为,从而保证有序性。

- **StoreStore 屏障**可以禁止普通写操作与 volatile 写操作的重排
- **StoreLoad 屏障**会禁止 volatile 写与 volatile 读重排
- **LoadLoad 屏障**会禁止 volatile 读与后续普通读操作重排
- **LoadStore 屏障**会禁止 volatile 读与后续普通写操作重排

:::

### 5.volatile能保证线程安全吗？

:::note 参考答案： 

`volatile` 不能保证线程安全，因为它仅提供变量的**可见性**和**有序性**（禁止指令重排），但不能保证**原子性**。

比如一个简单的自增操作 `i++`，实际上包含三个步骤：读取、修改、写回。

`volatile` 不能保证这三个步骤能被一个不可分割的整体执行，从而可能导致多个线程操作时出现数据竞争和不一致。 

:::

### 2.synchronized 是什么？有什么用？

### 4.synchronized 底层原理？

:::note 参考答案： 

 synchronized底层依赖 JVM 内部的 **Monitor 对象**来实现线程同步。对于同步代码块，会使用`monitorenter` 指令指向同步代码块的开始位置，`monitorexit` 指令则指明同步代码块的结束位置。

- `monitorenter`表示线程正在尝试获取 lock 对象的 Monitor；
- `monitorexit`表示线程执行完了同步代码块，正在释放锁。

对于方法，会使用`ACC_SYNCHRONIZED`标识来指明该方法是一个同步方法。

### 5.你对 Monitor 了解多少？

:::note 参考答案： 

Monitor 是 JVM 内置的同步机制，每个对象在内存中都有一个对象头——Mark Word，用于存储锁的状态，以及 Monitor 对象的指针。

synchronized 依赖对象头的 Mark Word 进行状态管理，支持无锁、偏向锁、轻量级锁，以及重量级锁。

synchronized 升级为重量级锁时，依赖于操作系统的互斥量——mutex 来实现，mutex 用于保证任何给定时间内，只有一个线程可以执行某一段特定的代码段。

:::

### 6.synchronized 怎么保证可见性？怎么保证有序性？

:::note 参考答案： 

保证可见性：

- 加锁时，线程必须从主内存读取最新数据。
- 释放锁时，线程必须将修改的数据刷回主内存，这样其他线程获取锁后，就能看到最新的数据。

保证有序性：

- synchronized 通过 JVM 指令 monitorenter 和 monitorexit，来确保加锁代码块内的指令不会被重排。

:::

### 7.synchronized 怎么实现可重入的？

:::note 参考答案： 

可重入意味着同一个线程可以多次获得同一个锁，而不会被阻塞。

synchronized 之所以支持可重入，是因为 Java 的对象头包含了一个 Mark Word，用于存储对象的状态，包括锁信息。

当一个线程获取对象锁时，JVM 会将该线程的 ID 写入 Mark Word，并将锁计数器设为 1。

如果一个线程尝试再次获取已经持有的锁，JVM 会检查 Mark Word 中的线程 ID。如果 ID 匹配，表示的是同一个线程，锁计数器递增。

当线程退出同步块时，锁计数器递减。如果计数器值为零，JVM 将锁标记为未持有状态，并清除线程 ID 信息。

:::

### 8.synchronized 锁升级过程？

大概流程是：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁。**

①、从无锁到偏向锁：

当一个线程首次访问同步代码时，如果此对象处于无锁状态且偏向锁未被禁用，JVM 会将该对象头的锁标记改为偏向锁状态，并记录当前线程 ID。此时，对象头中的 Mark Word 中存储了持有偏向锁的线程 ID。

如果另一个线程尝试获取这个已被偏向的锁，JVM 会检查当前持有偏向锁的线程是否活跃。如果持有偏向锁的线程不活跃，可以将锁偏向给新的线程；否则撤销偏向锁，升级为轻量级锁。

②、偏向锁的轻量级锁：

进行偏向锁撤销时，会遍历堆栈的所有锁记录，暂停拥有偏向锁的线程，并检查锁对象。如果这个过程中发现有其他线程试图获取这个锁，JVM 会撤销偏向锁，并将锁升级为轻量级锁。

当有两个或以上线程竞争同一个偏向锁时，偏向锁模式不再有效，此时偏向锁会被撤销，对象的锁状态会升级为轻量级锁。

③、轻量级锁到重量级锁：

轻量级锁通过自旋来等待锁释放。如果自旋超过预定次数（自旋次数是可调的，并且是自适应的，失败次数多自旋次数就少），表明锁竞争激烈。

当自旋多次失败，或者有线程在等待队列中等待相同的轻量级锁时，轻量级锁会升级为重量级锁。在这种情况下，JVM 会在操作系统层面创建一个互斥锁——Mutex，所有进一步尝试获取该锁的线程将会被阻塞，直到锁被释放。

:::note 参考答案： 

大概流程是：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**。

synchronized锁会根据**竞争的激烈程度，从无锁到偏向锁 、轻量级锁逐步升级到重量级锁。整个过程是不可逆的**。

- 对象刚创建时，是无锁状态。
- 如果一个对象一直被同一个线程访问，就会使用**偏向锁**。JVM 会在对象头里记录这个线程的 ID，以后再加锁时直接偏向这个线程，不用真正加锁。适合单线程反复进入同步块的场景。
- 当另一个线程来竞争偏向锁时，JVM 会把**锁升级为轻量级锁，**它通过 **CAS**来竞争锁，不会阻塞线程，而是通过自旋来获取锁。适用于那种**同步块执行速度快**，并且**锁竞争不激烈**的场景。
- 当多个线程自旋尝试获取轻量级锁失败，或者自旋次数达到一定阈值后，锁就会升级**为重量级锁**。底层会用操作系统的互斥量**mutex**，没有获得锁的线程会被**挂起**，进入阻塞状态。直到锁释放，才会被唤醒重新竞争锁。

总结来说，synchronized 的锁升级过程是：无锁 → 偏向锁 → 轻量级锁 → 重量级锁，用来在不同竞争场景下平衡性能和安全。无竞争时用偏向锁，让单个线程访问同步块几乎没有开销。轻度竞争时用轻量级锁，通过自旋来避免线程挂起，提高性能。激烈竞争时，用重量级锁，让没有获得锁的线程进入阻塞状态，避免 CPU 空转。

:::

### 9.ReentrantLock 是什么？实现原理？

ReentrantLock是基于 AQS 实现的 可重入排他锁，使用 CAS 尝试获取锁，失败的话，会进入 CLH 阻塞队列，支持公平锁、非公平锁，可以中断、超时等待。

内部通过一个计数器 state 来跟踪锁的状态和持有次数。当线程调用 `lock()` 方法获取锁时，ReentrantLock 会检查 state 的值，如果为 0，通过 CAS 修改为 1，表示成功加锁。否则根据当前线程的公平性策略，加入到等待队列中。

线程首次获取锁时，state 值设为 1；如果同一个线程再次获取锁时，state 加 1；每释放一次锁，state 减 1。

当线程调用 `unlock()` 方法时，ReentrantLock 会将持有锁的 state 减 1，如果 `state = 0`，则释放锁，并唤醒等待队列中的线程来竞争锁。

`new ReentrantLock()` 默认创建的是非公平锁 NonfairSync。在非公平锁模式下，锁可能会授予刚刚请求它的线程，而不考虑等待时间。当切换到公平锁模式下，锁会授予等待时间最长的线程。

:::note 参考答案： 

**1、ReentrantLock是什么？**

ReentrantLock是Lock接口下的一个实现类，是一个**可重入排他锁**，和synchronized关键字类似，但是更灵活、更强大。

<br>

**2、 ReentrantLock 的实现原理？**

ReentrantLock是基于 **AQS** 实现的**可重入排他锁**。使用 **CAS** 尝试获取锁，失败的话，会进入 **CLH 等待队列**，支持公平锁、非公平锁，可以中断、超时等待。

AQS内部通过一个**state变量**来跟踪锁的状态和持有次数。0表示没有线程持有，>0表示被线程持有，每次加锁就是`state+1`，释放锁就是`state-1` 。

- 当线程调用`lock()`方法获取锁时，线程会先尝试使用CAS操作，去修改 AQS 的state变量。

  - 如果state是0，说明锁是空闲的，CAS 成功，state变为 1，当前线程成功获取锁。

  - 如果state大于 0，说明锁被其他线程持有。这时会判断**持有锁的线程是不是自己**。如果是，就将 state加 1，实现锁的**可重入。**

- 如果 CAS 失败，并且持有锁的线程不是自己，那么该线程就会被封装成一个**Node节点**，加入到 AQS 的**等待队列**中。
  - 线程进入队列后，会**尝试自旋**去获取锁。如果自旋失败，就会被**挂起**，进入阻塞状态，等待被唤醒。

- 当线程调用`unlock()`方法释放锁时，会通过CAS把AQS的**state值减1**，如果 `state == 0`，说明锁完全释放了，就从等待队列里唤醒一个线程继续抢锁。

**总结来说，ReentrantLock基于AQS实现，核心是用 CAS修改 state 表示锁的占用情况，没抢到锁的线程进入等待队列，释放锁时唤醒队列里的线程。**

:::

### 10.synchronized 和 ReentrantLock 的区别？

### 13.什么是CAS？CAS的原理？

:::note 参考答案： 

**CAS的全称是 Compare And Swap（比较与交换）**，CAS 是一个原子操作，底层依赖于一条 CPU 的原子指令。

CAS 涉及三个操作数：V是要更新的变量，E是预期值，N是要写入的新值。

CAS 的执行过程可以概括为一句话：**当且仅当** **V 的值等于 E 时，才将 V 的值更新为 N**。

如果V和E不相等，就说明值被其他线程修改过，CAS 会操作失败并进行重试。

CAS的优点是无锁，不会阻塞线程，在并发量不大时，它的性能比加锁要高。

:::

### 1.CAS 有什么问题？怎么解决？

:::note 参考答案： 

CAS存在的问题：

- 存在**ABA问题**。ABA 问题指的是，一个值原来是 A，后来被改为 B，再后来又被改回 A，这时 CAS 会误认为这个值没有发生变化。从而更新成功，但实际上，这个值已经被其他线程修改过了。
- 自旋开销大。CAS 失败时会不断自旋重试，如果一直不成功，会给 CPU 带来非常大的执行开销。可以加一个自旋次数的限制，超过一定次数，就切换到 synchronized 挂起线程。
- 只能操作一个变量。涉及到多个变量同时更新，可以将多个变量封装为一个对象，使用 AtomicReference 进行 CAS 更新。

**如何解决ABA问题？**

可以使用版本号/时间戳的方式来解决 ABA 问题。

比如说，每次变量更新时，不仅更新变量的值，还更新一个版本号。CAS 操作时，不仅比较变量的值，还比较版本号。

比如Java原子类中的 `AtomicStampedReference `就增加了版本号，它会同时检查引用值和 stamp 是否都相等。

:::

### 1.AQS了解吗？

:::note 参考答案： 

AQS中文叫做**抽象队列同步器。**AQS 是一个用来构建锁和同步器的框架**，**像我们熟悉的ReentrantLock、CountDownLatch就是基于AQS的。

AQS 的核心思想是，如果被请求的共享资源处于空闲状态，则当前线程成功获取锁；否则，将当前线程加入到等待队列中，当其他线程释放锁时，通知等待队列中的第一个线程去竞争锁。

它的实现原理包括两个部分：**state变量**和 **CLH 等待队列。**state变量是用**volatile修饰**的，用来表示共享资源的状态。通过 **CAS**操作来保证对它的修改是原子性的。

CLH 等待队列是一个**双向链表**，保存等待的线程。每个等待的线程都会被封装成一个 `Node` 节点，添加到队列的尾部并自璇，等待前一个节点的线程释放锁。

AQS支持两种模式，**独占模式和共享模式**。独占模式一个线程拿到锁，其他线程必须等待，比如 ReentrantLock。共享模式可以同时有多个线程获取资源，比如CountDownLatch。

总结来说，**AQS 是一个基于 state + CLH 队列的同步框架，通过 CAS修改 state 来实现锁竞争，没拿到锁的线程进入队列挂起，释放资源时唤醒队列中的线程；它支持独占模式和共享模式，是 JUC 锁和同步工具的核心。**

:::

### 1.什么是悲观锁？什么是乐观锁？

:::note 参考答案： 

- **悲观锁**认为每次访问共享资源时都会发生冲突，所以在操作前一定要先加锁，防止其他线程修改数据。像 Java 中`synchronized`和`ReentrantLock`等独占锁就是悲观锁思想的实现。
- **乐观锁**认为冲突不会总是发生，所以在操作前不加锁，只在提交更新时才检查数据在操作期间是否有被其他线程修改过。如果发现数据被修改了，会重新读取数据并再次尝试更新。
- 悲观锁适用于**写操作非常频繁**的场景，也就是**并发冲突非常严重**的地方。它可以确保数据的一致性，但是性能比较低，还可能会存在死锁问题。
- 乐观锁适用于**读多写少**的场景，也就是**并发冲突不严重**的地方。它的优点是不会加锁，性能好，缺点是如果冲突频繁，会导致大量的重试，性能反而会下降。

:::

### 如何实现乐观锁？

乐观锁一般会使用**版本号机制**或 **CAS** **算法**实现。

- **版本号机制**，会在数据表中增加一个**version字段。**读取数据时，将**version值一起读出**，更新数据时，会将**version值+1以及将version值作为更新条件，**只有刚才读取到的 version 值和当前数据库中的 version 值相等时才会更新成功。如果更新失败，说明版本号已经被其他线程修改，数据发生了冲突，需要重新读取数据并再次尝试更新。
- **CAS****。CAS的全称是 Compare And Swap（比较与交换）**，CAS 是一个原子操作，底层依赖于一条 CPU 的原子指令。CAS 涉及三个操作数：V是要更新的变量，E是预期值，N是要写入的新值。

CAS 的执行过程可以概括为一句话：**当且仅当** **V 的值等于 E 时，才将 V 的值更新为 N**。

如果V和E不相等，就说明值被其他线程修改过，CAS 会操作失败并进行重试。

CAS的优点是无锁，不会阻塞线程，在并发量不大时，它的性能比加锁要高。

缺点是**存在ABA问题**。也就是说：假设变量的值从 A 变成了 B，然后又变回了 A。在 CAS 检查时，会发现当前值和预期值都是 A，认为没有发生过变化，从而更新成功。但实际上，这个值已经被其他线程修改过了。

### 1.什么是公平锁？什么是非公平锁？

:::note 参考答案： 

- 公平锁：每个线程获取锁的顺序是按照线程访问锁的先后顺序获取的，最前面的线程总是最先获取到锁。
- 非公平锁：每个线程获取锁的顺序是随机的，并不会遵循先来先得的规则，所有线程会竞争获取锁。

:::

### 1.什么是死锁？死锁的条件？如何避免死锁？死锁怎么排查？  

:::note 参考答案： 

**1、什么是死锁？**

死锁发生在多个线程相互等待对方释放锁时。比如说线程 1 持有锁 R1，等待锁 R2；线程 2 持有锁 R2，等待锁 R1。

**2、死锁的条件？**

死锁的发生需要有四个条件，分别是**互斥、持有并等待、不可抢占和循环等待**。

**互斥**指的是资源不能被多个线程共享，一次只能由一个线程使用。

**持有并等待**也就是一个线程已经持有一个资源，并且在等待获取其他线程持有的资源。

**不可抢占**指的是资源不能被强制从线程中夺走，必须等线程自己释放。

**循环等待**指的是存在一种线程等待链，比如线程 A 等待线程 B 持有的资源，线程 B 等待线程 C 持有的资源，线程 C又等待线程 A 持有的资源。

**3、如何避免死锁呢？**

第一，所有线程都按照固定的顺序来申请资源。例如，先申请 R1 再申请 R2。

第二，如果线程发现无法获取某个资源，可以先释放已经持有的资源，重新尝试申请。

**4、死锁问题怎么排查呢？**

首先可以使用`top` 、`ps` 等命令查看进程状态，看看是否有进程占用了过多的资源。

然后可以使用JDK 自带的`jps -l` 查看当前进程，然后使用 `jstack 进程号` 查看当前进程的线程堆栈信息，看看是否有线程在等待锁资源。

:::

## ThreadLocal

### 1.ThreadLocal 是什么? 有什么好处？

:::note 参考答案： 

ThreadLocal是一种用于实现线程局部变量的工具类。它允许每个线程都拥有自己的独立副本，从而实现线程隔离。

ThreadLocal的优点：

- ThreadLocal 通过为每个线程提供独立的变量副本，从根本上解决了共享变量带来的线程安全问题。
- ThreadLocal 可用于跨方法、跨类时传递上下文数据，不需要在方法间传递参数。

:::

### ThreadLocal的使用场景？

:::note 参考答案： 

- 在 Web 应用中，可以使用 ThreadLocal 存储用户会话信息，这样每个线程在处理用户请求时都能方便地访问当前用户的会话信息。
- 在数据库操作中，可以使用 ThreadLocal 存储数据库连接对象，每个线程有自己独立的数据库连接，从而避免了多线程竞争同一数据库连接的问题。
- 在格式化操作中，例如日期格式化，可以使用 ThreadLocal 存储 SimpleDateFormat 实例，避免多线程共享同一实例导致的线程安全问题。

:::

### 2.ThreadLocal 的实现原理？

:::note 参考答案： 

每个线程内部会维护一个ThreadLocalMap，ThreadLocalMap的每个Entry的 key 是ThreadLocal对象，value 就是 ThreadLocal对象调用`set`方法设置的值，也就是线程的局部变量，这样就相当于为每个线程维护了一个变量副本。

ThreadLocal的`get()`和`set()`方法本质上就是对当前线程的`ThreadLocalMap`进行存取操作。

:::

### 3.ThreadLocal 内存泄露是怎么回事？怎么解决？

:::note 参考答案： 

ThreadLocalMap 的Key是弱引用，但Value是强引用。

key为弱引用，意味着如果ThreadLocal 实例不再被任何强引用指向，垃圾回收器会在下次 GC 时回收该实例，导致ThreadLocalMap中对应的 key 变为`null` 。

value 是强引用，即使 `key`被 GC 回收，`value`仍然被`ThreadLocalMap.Entry`强引用存在，无法被 GC 回收。

如果线程持续存活（例如线程池中的线程），`ThreadLocalMap` 也会一直存在，导致 key 为 `null` 的 entry 无法被垃圾回收，即会造成内存泄漏。

<br>

解决办法：使用完 ThreadLocal 后，及时调用`remove()`方法释放内存空间。

:::

### 4.父线程能用 ThreadLocal 给子线程传值吗？

:::note 参考答案： 

不能。由于 ThreadLocal 的变量值存放在 Thread 里，而父子线程属于不同的Thread，因此父子线程的 ThreadLocal 值无法进行传递。

可以使用 `InheritableThreadLocal`来解决这个问题。

:::

### 5.你了解哪些 ThreadLocal 的改进方案？

:::note 参考答案： 

① Netty 中的 FastThreadLocal，它是 Netty 对 ThreadLocal 的优化，内部维护了一个索引常量 index，每次创建 FastThreadLocal 中都会自动+1，用来取代 hash 冲突带来的损耗，用空间换时间。

② 阿里的 TransmittableThreadLocal，不仅实现了子线程可以继承父线程 ThreadLocal 的功能，并且还可以跨线程池传递值。

:::

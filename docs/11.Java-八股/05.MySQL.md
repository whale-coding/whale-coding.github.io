## MySQL基础

### 1.说一下数据库的三大范式？

:::note

第一范式：要求数据库表的每一列都是不可分割的基本数据单元；

第二范式：要求表中的每一列都和主键直接相关；

第三范式：要求非主键列应该只依赖于主键列。

:::

### 2.MySQL 中 DELETE、DROP 和 TRUNCATE 的区别是什么？

:::note

- DROP 是物理删除，用来删除整张表，包括表结构，且不能回滚；
- DELETE 支持行级删除，可以带 WHERE 条件，可以回滚；
- TRUNCATE 用于清空表中的所有数据，但会保留表结构，不能回滚。

:::

### 3.MySQL 中 inner join、left join、right join 的区别是什么？

:::note

内连接可以用来找出两个表中共同的记录，相当于两个数据集的**交集**；

左连接和右连接是外连接，可以用来找出两个表中不同的记录，相当于两个数据集的**并集**，不同的是左外连接会保留左表中符合条件的所有记录，右连接外则刚好相反。

:::

### 4.MySQL 中 count(*)、count(1) 和 count(字段名) 有什么区别？

:::note

- 在 InnoDB 引擎中，`COUNT(1)` 和 `COUNT(*)` 没有区别，都是用来统计所有行，**包括 NULL**。
- 如果表有索引，`COUNT(*)` 会直接用索引统计，而不是全表扫描，而 `COUNT(1)` 也会被 MySQL 优化为 `COUNT(*)`。
- `COUNT(列名)` **只统计列名不为 NULL 的行数**。

:::

### 5.MySQL 中CHAR 和 VARCHAR 的区别是什么？

:::note

CHAR 是定长字符串，VARCHAR 是变长字符串。当定义一个`CHAR(10)`字段时，不管实际存储的字符长度是多少，都只会占用 10 个字符的空间。但是 `VARCHAR(10)`会根据传入的字符数动态占用空间。

CHAR适合固定长度的场景，比如存储国内手机号、身份证号等。

:::

### 6.什么是视图？

:::note

视图（View） **是一种虚拟表**，内容由一个 SQL 查询语句定义。

视图并不实际存储数据，而是基于一个或多个基础表（或其他视图）的查询结果动态生成。

使用视图的好处：简化复杂查询、增强安全性和逻辑独立性。

:::

## 存储引擎

### 1.SQL 查询语句的执行顺序了解吗？

![1280X1280](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304204811679.PNG)

:::note

先执行 FROM 确定主表，再执行 JOIN 连接，然后 WHERE 进行过滤，接着 GROUP BY 进行分组，HAVING 过滤聚合结果，SELECT 选择最终列，ORDER BY 排序，最后 LIMIT 限制返回行数。

:::

### 2.MySQL支持哪些存储引擎？

:::note

MySQL支持多种引擎，可以通过`show engines`命令查看MySQL支持的所有引擎。

常见的有InnoDB、MyISAM、Memory。

- InnoDB：InnoDB是MySQL的默认存储引擎，支持事务、行级锁、外键约束等特性。
  - 适用于高并发的读写操作，支持较好的数据完整性和并发控制。
- MyISAM：不支持事务、行级锁和外键。
  - 适用于大量读操作的场景。
- Memory：Memory引擎将数据存储在内存中，适用于对性能要求较高的读操作。但是服务器重启数据会丢失。

:::

### 3.InnoDB 和 MyISAM  存储引擎有什么区别？ 

:::note

- InnoDB 支持行级锁，MyISAM 不支持，只支持表级锁；
- InnoDB 支持事务，MyISAM 不支持；
- InnoDB 支持外键，MyISAM 不支持；
- 底层都是用B+Tree 作为索引结构，但是两者的实现方式不一样，InnoDB 是聚簇索引，索引和数据不分开，MyISAM 为非聚簇索引，索引和数据分开存储；
- InnoDB 有`redo log`日志机制，支持崩溃恢复，MyISAM 不支持。

:::

>1.执行 count(\*) 采用InnoDB和MyISAM哪个存储引擎性能更好？

:::danger

MyISAM的性能更好。

InnoDB不保存表的具体行数，执行select count(*)时需要全表扫描；

而MyISAM是用一个变量保存了整个表的行数，执行select count(*)时只需要读取该变量即可，速度更快。

:::

### MySQL中一条SQL查询语句是如何执行的？

![](https://oss.javaguide.cn/javaguide/13526879-3037b144ed09eb88.png)

:::note

首先客户端通过**连接器**与MySQL服务建立连接，连接器会对用户身份、权限进行校验；

执行查询语句的时候，在MySQL8.0以前会先**查询缓存**，没有命中缓存的话，SQL 语句就会经过**分析器**，分析器对SQL 语句进行**词法分析和语法分析**，词法分析解析这条SQL要干嘛，识别出关键字、表名、字段等，语法分析检查SQL语句是否符合语法规则。之后是进入**优化器**，优化器负责确定 SQL 语句的执行计划，包括选择使用哪些索引，决定关联表的顺序等。最后是**执行器**调用存储引擎的接口去执行SQL语句，然后从**存储引擎**返回数据。

:::

## 索引

### 1.什么是索引？索引的优缺点？

:::note

索引是一种用于快速查询和检索数据的数据结构。索引就像一本书的目录，能让 MySQL 快速定位数据，避免全表扫描。

索引的优点：可以大大加快数据的检索速度，提高查询效率；索引的数据结构一般是 B+ 树，查找效率是 O(log n)，比从头到尾全表扫描数据要快得多。

索引的缺点：创建和维护索引需要耗费一些时间、索引需要使用物理文件存储会耗费一定空间。

:::

>1.为什么用索引？索引为什么会加快查询速度？

:::danger 答案

索引可以加快数据库查询的速度和性能。

索引的数据结构一般是 B+ 树，查找效率是 O(log n)，比从头到尾全表扫描数据要快得多。

:::

### 2.索引的分类有哪些？

:::note

可以按照不同的角度来给索引分类：

- 按「数据结构」分类：B+ tree索引、Hash索引、全文索引。
- 按「物理存储」分类：聚簇索引（主键索引）、非聚簇索引（二级索引）。
- 按「字段特性」分类：主键索引、唯一索引、普通索引。
- 按「字段个数」分类：单列索引、联合索引。

:::

>1.B+树索引和 Hash 索引有什么区别？

:::danger 答案

- B+ 树索引支持范围查询、有序扫描，是 InnoDB 的默认索引结构；
- Hash 索引只支持等值查找，速度快但功能弱，常见于 Memory 引擎。

:::

### 3.什么是主键索引？什么是唯一索引？什么是普通索引？

:::note

- **主键索引**就是**由主键字段创建的唯一索引**，一张数据表只能有一个主键，并且主键不能为 null，不能重复。在 InnoDB 存储引擎中，主键索引也被叫做**聚簇索引**。
- **唯一索引**是建立在**UNIQUE字段**上的索引，一张表可以有多个唯一索引，索引列的值必须唯一，但是允许有空值。
  - MySQL 的唯一索引允许有空值，并且允许有多个 `NULL` 值。这是因为在数据库中，`NULL` 表示未知，而两个未知的值（即两个 `NULL`）不能被判定为相等或不相等，因此它们不会违反唯一约束。
- **普通索引**就是建立在普通字段上的索引，既不要求字段为主键，也不要求字段为 UNIQUE。

:::

>1.唯一索引和主键索引有什么区别？

:::danger 答案

- 每个表只能有一个主键索引，但可以有多个唯一索引；
- 主键索引不允许插入 NULL 值，唯一索引允许插入 NULL 值。

:::

>2.普通索引和唯一索引有什么区别？

:::danger 答案

- 普通索引仅用于加速查询，不限制字段值的唯一性；唯一索引的字段值具有唯一性。
- 普通索引适用于高频写入的字段、范围查询的字段；唯一索引适用于业务唯一性约束的字段、防止数据重复插入的字段。

:::

### 4.聚簇索引和非聚簇索引有什么区别？ 

>1.什么是聚簇索引？什么是非聚簇索引？

:::note

- **聚簇索引**是指索引与数据行一起存储，数据行物理顺序与索引顺序一致，每个表只能有一个。InnoDB存储引擎中，主键索引就是聚簇索引。
- **非聚簇索引**则将索引与数据行分开存储，索引的叶子节点包含指向实际数据行的指针或主键值，一个表可以有多个非聚簇索引。 普通索引、唯一索引都是非聚簇索引。

:::

>2.聚簇索引和非聚簇索引的区别？

:::note

- 聚簇索引的索引结构和数据是放在一起的，数据就存储在主键 B+ 树的**叶子节点**上；非聚簇索引的索引结构和数据是分开的，索引的叶子节点存的是**主键值**，不是数据本身。
- 一个表只能有一个聚簇索引，但是可以有多个非聚簇索引。
- 查主键时，聚簇索引效率高，不需要回表；查非主键列时，非聚簇索引可能需要回表，效率相对低一些。

:::

### 5.索引的底层原理？索引的结构是什么样的？

主键索引的 B+Tree 如图所示：

![8ceea6b6-06d9-47a8-a993-39db4d19629a](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304211333557.png)

:::note

MySQL 默认的存储引擎是 `InnoDB`，其索引的底层实现主要基于 `B+ 树`数据结构。

B+ Tree是一种多叉树，叶子节点才存放数据，非叶子节点只存放索引，而且每个叶子节点里的数据是**按照索引列的值排序存放**的（主键索引**按主键顺序存放，其他索引按照索引字段值排序顺序存放**）。

每一层父节点的索引值都会出现在下层子节点的索引值中，因此在叶子节点中，包含了所有的索引值信息，并且每一个叶子节点都有两个指针，分别指向下一个叶子节点和上一个叶子节点，形成一个双向链表。

当使用索引进行查询的时候，B+ Tree会**自顶向下逐层进行查找**。

 InnoDB支持聚簇索引，也叫主键索引和非聚簇索引，它们的B+树有所区别：

聚簇索引的B+树叶子节点**按主键顺序存放完整的数据行**；而在非聚簇索引的B+树叶子节点**按索引列的值排序，存放的是索引列的值和对应的主键值**。查询完整的数据行需要回表。

:::

### 6.什么是联合索引？什么是最左匹配原则？联合索引的B+树是什么样的？联合索引的实现原理？

联合索引(product_no, name) 的 B+Tree 示意图如下：

![a8fc31f9-e082-4fb3-8e29-b5c98279785f](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304211416564.png)

>1.什么是联合索引？

:::note

联合索引就是将多个字段组合成一个索引，联合索引必须遵守“最左前缀”原则，只有从第一个字段开始连续使用，索引才会生效。

:::

>2.什么是索引的最左匹配原则？

:::note

最左前缀匹配原则指的是**在使用联合索引时**，查询条件必须从索引的最左边的列开始**连续匹配**，中间**不能跳列**、**不能乱顺序，**索引才会生效。

:::

>3.联合索引的B+树是什么样的？

:::note

以联合索引(id, name) 的 B+Tree为例，

联合索引的非叶子节点用id和name两个字段的值作为B+ Tree的key值。当在使用联合索引查询数据时，先按id字段比较，在id相同的情况下再按name字段比较。

也就是说，联合索引查询的B+ Tree是先按id进行排序，然后在id相同的情况再按照name字段排序。

因此，使用联合索引时，存在最左匹配原则，如果不遵循，索引会失效。

:::

### 7.什么是覆盖索引？什么是回表？什么是索引下推？

>1.什么是覆盖索引？

:::note

如果一个索引覆盖所有需要查询的字段的值，我们就称之为覆盖索引。也就是查询所需的字段全部都在索引中，那么直接根据该索引，就可以查到数据了，而无需回表查询。

:::

>2.什么是回表？

:::note

非聚簇索引的 B+树的叶子节点存放的是主键值，而不是实际数据。

当使用非聚簇索引进行查询时，MySQL 需要先通过非聚簇索引找到主键值，然后再根据主键值回到聚簇索引中查找完整数据行，这个过程称为**回表**。

:::

>3.什么是索引下推？

:::note

索引下推是一种优化手段，**主要作用是减少回表次数，提高查询效率**。

在没有开启索引下推时，MySQL 会先通过非聚簇索引找到主键，再回表取出整行数据，然后判断 WHERE 条件是否满足。

开启索引下推后，MySQL 会尽量利用**索引中的字段在存储引擎层先做过滤**，只有满足条件的记录才会回表，**从而减少不必要的回表操作**。

索引下推仅适用于**二级索引**，在主键索引上并不会生效。特别适合在联合索引、范围查询等场景下使用。

:::

>扩展1：为什么MySQL会保留回表呢？

:::note

MySQL“回表”是为了**获取除主键以外的列数据**。这是因为聚簇索引的叶子节点包含了所有数据列，但二级索引的叶子节点只包含索引列和指向聚簇索引（主键）的指针。当查询条件仅包含二级索引列时，MySQL需要先通过二级索引找到对应行的主键值，然后再利用这个主键值进行一次“回表”查询，通过主键索引去查找完整的数据行。

:::

>扩展1：如何避免回表？

:::note

覆盖索引，让查询所需的所有字段都在索引中，无需回表。

:::

### 8.InnoDB 存储引擎会选择什么样的列作为索引？

:::note

- 如果有**主键**，默认会使用主键作为聚簇索引的索引键；
- 如果没有主键，就选择第一个**不包含 NULL 值的唯一列**作为聚簇索引的索引键；
- 在上面两个都没有的情况下，InnoDB+将自动生成一个**隐式自增id**列作为聚簇索引的索引键。

:::

### 9.MySQL索引为什么使用B+树？而不是二叉树、红黑树、B树、跳表？

>1.MySQL索引为什么使用B+树？

:::note

因为B+ 树是一种高度平衡的多路查找树，能有效降低磁盘的 IO 次数，并且支持有序遍历和范围查询。查询性能非常高，同时结构上也适合 MySQL 按照页为单位在磁盘上存储。

B+ 树具有更高的查询效率和更稳定的性能。它通过将所有数据存储在叶子节点，并使用叶子节点之间的链表连接，使得范围查询和全表扫描更加高效。同时，B+ 树的非叶子节点不存储数据，可以存储更多索引，使树更矮胖，减少磁盘 I/O 次数，降低了查询的磁盘读写代价。

:::

>2.为什么 MySQL 索引用的是 B+ 树而不是二叉树？

:::note

二叉树只有两个子节点，**分支少、树高很容易变高**，在数据量大的时候，查找路径就会很深，树就很高，

数据库索引需要频繁访问磁盘，**树越高，磁盘** **I/O** **次数越多，效率就越低**。

而 B+ 树是多路查找树，一个节点能有成百上千个子节点，**树高很低**，能大大减少磁盘访问次数，提升查询效率。再加上 B+ 树的数据都存在叶子节点，叶子之间还是链表连接的，做范围查询和排序非常快，**更适合数据库的典型查询场景**。

:::

>3.为什么 MySQL 索引用的是 B+ 树而不是红黑树？

:::note

红黑树是二叉树，树的高度相对较高，查询时磁盘 I/O 次数多，而 B+ 树是多路平衡树，节点分支多、树高低，可以有效减少磁盘访问次数。

:::

>4.为什么 MySQL 索引用的是 B+ 树而不是B树（B-树）？

:::note

- **B+** **Tree****只在****叶子节点****存储数据，非叶子节点仅存储索引，而****B树****的非叶子节点也要存储数据**。所以，**B+ Tree相比B树节点能容纳更多索引从而降低树的高度，减少磁盘的****IO****次数**。
- **B+树****叶子节点****通过****双向链表****连接，非常适合进行范围查询，而****B树****需要通过遍历整个树来实现范围查询，效率较低。**

:::

>5.为什么 MySQL 索引用的是 B+ 树而不是跳表？

:::note

B+树适合磁盘IO，跳表适合内存IO。

B+Tree一个节点是一个page，是一种多叉树结构，每个结点都是一个16k的数据页，能存放较多索引信息。

所以树的层数比较低， 三层左右就可以存储2kw左右的数据。

跳表一个node存放一个索引信息 ，所以树的层数比较高，磁盘IO次数多。

因此，B+ 树的**磁盘** **I/O** **效率更高。**

:::

>6.B+树为什么通常是2-3层？为什么不用1层？

:::note

B+树之所以通常是2到3层，**是因为其“高扇出”特性，使用较少的层数即可存储大量数据，从而减少了数据库查询所需的磁盘****IO****次数，提高了性能**。

它不会是1层，因为1层无法有效管理大量的数据；而采用多层且平衡的设计则保证了高效查找。 

:::

>7.B+树和B(B-)树的区别？

:::note

- B 树所有节点都存储数据，B+ 树只有叶子节点存数据，非叶子节点只存索引；
- B树叶子节点之间不连接，B+树叶子节点之间通过双向链表连接；
- B 树范围查询需要中序遍历，效率低，B+树直接遍历叶子链表，效率高

:::

### 10.查询数据时，到了B+树的叶子节点，之后的查找数据是如何做？

:::note

MySQL数据是按照页为单位存储在磁盘中的，一页数据大小是16KB。

**数据页中的记录按照【****主键****】顺序组成****单向链表**。单链表插入、删除快，但是检索慢。因此，数据页中有一个**页目录**，数据页中的页目录就是为了快速找到记录，起到一个记录的索引作用。

页目录的创建过程如下：

- 将所有的记录划分成几个组，这些记录包括最小记录和最大记录；
- 每个记录组的最后一条记录就是组内最大的那条记录，并且最后一条记录的头信息中会存储该组一共有多少条记录；
- 页目录用来存储每组最后一条记录的偏移量，这些偏移量会按照先后顺序存储起来，每组的地址偏移量也被称之为槽(slot)，每个槽相当于指针指向了不同组的最后一个记录。

页**目录就是由多个槽组成的，槽相当于分组记录的索引**。因为记录是按照**【****主键****值】**从小到大排序的，所以我们**通过槽查找记录时，可以使用****二分法****快速定位要查询的记录在哪个槽（哪个记录分组），定位到槽后，再遍历槽内的所有记录，找到对应的记录**，无需从最小记录开始遍历整个数据页中的记录链表。

:::

### 11.什么字段适合加索引？什么字段不适合加索引？

适合添加索引的情况：数据量大的表。不适合添加索引的情况：数据量很小的表。

>1.什么字段适合加索引？

:::note

- 主键字段、唯一键字段、外键字段；
- 经常在 `WHERE`、`ORDER BY`、`GROUP BY` 子句中出现的字段；
- 具有高区分度的字段；

:::

>2.什么字段不适合加索引？

:::note

- 需要频繁更新的字段；
- 重复值很多的列；
- 区分度比较低的列；
- 查询条件（WHERE、GROUP BY、ORDER BY）中用不到的字段；
- 使用左模糊匹配的字段，因为索引不会生效；

:::

### 12.索引在哪些情况下会失效？ 

:::note

- 联合索引违反了最左前缀原则，索引会失效；
- 对索引列使用函数或表达式会导致索引失效；
- LIKE 模糊查询以通配符开头会导致索引失效；
- 使用 OR 连接非索引列条件，会导致索引失效；
- 查询条件中使用不等号，会导致索引失效；
- 索引列上进行类型转换（隐式转换）；
- 使用 NOT IN、NOT EXISTS；

:::

### 13.建索引的时候要遵循哪些原则？

:::note

- 第一，选择合适的字段：频繁出现在 WHERE、JOIN、ORDER BY、GROUP BY 中的字段以及区分度高的字段适合建索引；
- 第二，要控制索引的数量，避免过度索引；
- 第三，联合索引的时候要遵循最左前缀原则；

:::

- index表示全表扫描索引文件；
- range表示检索指定范围的行
- const表示索引一次就得到了结果，一般是使用唯一索引或者是主键作为查询条件；
- system表示表中仅有一行数据

extra列会显示执行状态说明，

要确保此列不要出现using filesort、using temporary等使用临时表或外部文件的情况；

如果出现using index最好，它表示列数据仅仅使用了索引中的信息而没有回表查询。

### redo log的刷盘时机？

:::note

Redo log的刷盘（从内存Buffer刷到磁盘文件）主要时机包括：

- **事务提交时**：当执行事务提交时，`Redo log`会根据参数 `innodb_flush_log_at_trx_commit` 的设置决定是否立即刷盘。参数为1时会立即刷盘。
- **Redo Log Buffer 空间不足**：`Redo log` 的内存缓冲区（Buffer）是循环使用的。当 `Redo log buffer` 中已经被占用的空间超过一半时，后台线程会强制将这部分日志刷入磁盘。
- **后台线程每秒定时刷盘**：不管事务是否提交，InnoDB 都有一个后台线程，每隔 1 秒钟会将 `Redo log buffer` 中的数据写入到磁盘文件 `os cache`，再持久化到物理磁盘。
- **Checkpoint检查点机制**：InnoDB 引擎会定时执行 `checkpoint`。为了防止由于 `Redo log` 占用过多，此时会将 `Buffer Pool` 中已经刷盘对应的 `Redo log` 设置为可覆盖状态，此时对应的日志也会刷入磁盘。
- **正常关闭MySQL服务**：当 MySQL 服务执行正常关闭时（SHUTDOWN），也会将 `Redo log buffer` 中的所有数据全部刷入磁盘。 

:::

### redo log怎么保证持久性的？

:::note

它通过以下方式来保证持久性：

- **Write-ahead logging****（****WAL****）**：在事务提交之前，将事务所做的修改操作记录到redo log中，然后再将数据写入磁盘。这样即使在数据写入磁盘之前发生了宕机，系统也可以通过redo log中的记录来恢复数据。 
-  **Redo** **log****的顺序写入**：redo log采用追加写入的方式，将redo日志记录追加到文件末尾，而不是随机写入。这样可以减少磁盘的随机I/O操作，提高写入性能。 
- **Checkpoint****机制**：MySQL会定期将内存中的数据刷新到磁盘，同时将最新的LSN（Log Sequence Number）记录到磁盘中，这个LSN可以确保redo log中的操作是按顺序执行的。在恢复数据时，系统会根据LSN来确定从哪个位置开始应用redo log。

:::

### redo log和binlog的区别？

:::note

- redo log是InnoDB独有的日志，binlog是属于Mysql Server层的，与存储引擎无关；
- redo log属于物理日志，记录的是“对数据页做了什么修改”，binlog是逻辑日志，记录的是“数据库执行的逻辑SQL语句”；
- redo log日志是循环写，日志空间大小是固定的，满了之后会循环利用，binlog是追加写入，写满一个文件，就创建一个新的文件继续写，不会覆盖使用；
- redo log有crash-safe能力，用于掉电等故障恢复；binlog没有crash-safe能力，用于备份恢复、主从复制。

:::

### redo log 和 undo log 区别在哪？

:::note

redo log 和 undo log都属于 InnoDB 存储引擎的日志，它们的区别在于：

- redo log记录的是事务**修改后**的数据状态，记录的是**更新之后**的值，主要用于事务崩溃恢复，保证事务的持久性；
- undo log记录的是事务**修改前**的数据状态，记录的是**更新之前**的值，主要用于事务回滚，保证事务的原子性。

:::

### 有了binlog为什么还要undolog、redolog？

:::note

binlog 属于 Server 层，与存储引擎无关，无法直接操作物理数据页。而 redo log 和 undo log 是 InnoDB 存储引擎实现事务 ACID 的基石。

binlog 会记录整个 SQL 或行的变化；redo log 是为了恢复“已提交但未刷盘”的数据，undo log 是为了撤销未提交的事务。

:::

### 什么是两阶段提交？为什么需要两阶段提交？两阶段提交的流程？

:::note

**1、为什么需要两阶段提交？**

当我们同时使用了redo log 和 binlog，在事务提交后，redo log 和 binlog都要持久化到磁盘，但是这是两个独立的逻辑，可能出现半成功的状态，例如在将redo log刷入磁盘之后，MySQL突然宕机了，而binlog还没有来得及写入，这样就会造成两份日志之间的逻辑不一致。

为了保证redo log 和 binlog 中的数据一致性。就使用两阶段来保证一致性。

**2、什么是两阶段提交？**

两阶段提交就是将一个事务分成两个阶段来提交：prepare（准备） 阶段和 commit（提交） 阶段。

在 prepare 阶段，事务会先写 redo log 并将其标记为 prepare 状态，然后写 binlog；

在 commit 阶段，事务会将 redo log 标记为 commit 状态，并将 binlog 落盘。

这样，无论数据库在哪个时刻发生宕机，都可以根据 redo log 和 binlog 的状态来判断事务是否提交，并保证数据的一致性。

**3、两阶段提交的流程？**

- 在prepare阶段，MySQL先将数据修改写入redo log，并将其标记为prepare状态，表示事务还未提交。然后将对应的SQL语句写入binlog。
- 在commit阶段，MySQL将redo log标记为commit状态，表示事务已经提交。然后根据sync_binlog（是否同步binlog）参数的设置，决定是否将binlog刷入磁盘。

:::



## 事务

### 1.什么是事务？其特性是什么？

:::note

事务是逻辑上的一组操作，要么都执行，要么都不执行。

事务有ACID，即原子性、一致性、隔离性和持久性，这4个特性。

- 原子性：指的是一组操作要么全部执行成功，要么全部不执行；
- 一致性：指的是执行事务前后，数据要保持一致；
- 隔离性：指的是并发访问数据库时，多个并发事务之间是独立的，是相互隔离的，互不干扰；
- 持久性：指的是一个事务被提交之后，它对数据库中数据的改变是持久的。

在MySQL 中，我们可以使用`begin` 或 `start transaction` 命令开启一个事务，使用 `commit` 命令提交一个事务，使用 `rollback` 命令回滚一个事务。

:::

### 2.事务的4个特性具体是怎么实现的？

:::note

- 原子性主要通过 undo log 来实现；
- 持久性通过 redo log 来实现；
- 隔离性由 MVCC 和锁机制来实现；
- 一致性则由其他三大特性共同保证。

:::

### 3.并发事务会带来哪些问题?

:::note

并发事务可能会出现：**脏读**、**不可重复读**、**幻读**。

- **脏读**：如果一个事务读到了另一个未提交事务修改过的数据，就意味着发生了脏读现象。比如事务T1修改一个数据，事务T2随后又读取这个数据，如果T1回滚了这次修改，那么T2读取到的数据就是脏数据。
- **不可重复读**：在一个事务内多次读取同一个数据，如果出现前后两次读到的数据不一致的情况，就意味着发生了不可重复读现象。比如事务T1读取一个数据，事务T2对该数据做了修改，如果T1再次读取这个数据，此时读取的结果就和第一次读取的结果不同，就出现了不可重复读现象。
- **幻读**：在同一事务中，当事务对某一范围的数据进行多次查询，由于其他事务在该范围中插入或删除了数据，导致两次查询结果的行数不一致，就发生了幻读现象。比如事务T1读取某个范围的数据，事务T2 在这个范围内插入新的数据，T1再次读取这个范围内的数据，此时读取的结果和第一次读取的结果不同，就出现了幻读现象。

:::

>1.不可重复读和幻读有什么区别？

:::danger 答案

“不可重复读”和“幻读”都是读的过程中数据前后不一致，但是侧重点不一样：

- 不可重复读侧重于读取的数据被其他事务update导致不一致；
- 幻读侧重于读取的范围数据中，出现了insert或者delete操作，导致数据的数量不一致。

:::

### 4.事务有哪些隔离级别？

![4d18b3e6-be18-463a-a4f6-ae86b9988023](https://blog-whalecoding.oss-cn-beijing.aliyuncs.com/picgo/20260304213444511.png)

:::note

事务的隔离级别包括：**读未提交、读已提交(RC)、可重复读(RR)和可串行化**。

- **读未提交**，能够读取到没有被提交的数据，可能会导致脏读、幻读或不可重复读。
- **读已提交**，能够读取到那些已经提交的数据，可以阻止脏读但是幻读和不可重复读还是有可能发生。
- **可重复读**，指的是在一个事务内，对同一数据的多次读取结果都是一致的，除非数据是被本身事务自己所修改。可以防止脏读和不可重复读，但是可能出现幻读。
- **可串行化**，指的是所有的事务依次按照顺序逐个执行，是最高事务隔离级别，事务之间完全不可能产生干扰，可以防止脏读、不可重复读和幻读。

MySQL InnoDB 存储引擎的默认隔离级别是**可重复读（RR）**。

:::

### 可重复读隔离级别是怎么实现的？







### 6.介绍一下MVCC？MVCC的实现原理？

MVCC 指的是多版本并发控制，用来实现数据库的并发控制，提高数据库并发性能的同时保证事务的隔离性。它的基本原理是给每一个事务提供一个数据快照，每个事务在读数据时看到的是数据的一个快照，而不是正在被其他事务修改的数据版本。这样读操作就不会被写操作阻塞，实现了非阻塞读，也叫“快照读”。MVCC 通过维护多个版本的数据来实现读写分离，不需要加锁，从而提高数据库并发性能。

:::note

1、MVCC简介

MVCC 指的是多版本并发控制，用来实现数据库的并发控制，提高数据库并发性能的同时保证事务的隔离性。它的基本原理是给每一个事务提供一个数据快照，每个事务在读数据时看到的是数据的一个快照，而不是正在被其他事务修改的数据版本。这样读操作就不会被写操作阻塞，实现了非阻塞读，也叫“快照读”。MVCC 通过维护多个版本的数据来实现读写分离，不需要加锁，从而提高数据库并发性能。

2、MVCC的实现原理？

MVCC的底层实现主要包括三个部分：隐藏字段、`undo log`和`read view`。

每张表的每一行记录中，InnoDB 默认会维护两个隐藏字段：一个是 trx_id（事务id），记录最近一次修改这条记录的事务 ID；另一个是roll_pointer（回滚指针），指向这条记录之前的版本，也就是 undo log 的地址。

当一个事务对数据进行修改时，InnoDB 会把旧数据写入 undo log，并更新这两个隐藏字段，形成一个版本链，多个版本通过 roll_pointer 串联在一起。不同事务看到的版本可能不一样，从而实现了并发读写互不干扰。

为了判断一个事务是否能看到某条记录的某个版本，InnoDB会在执行快照读时生成一个 **Read View，**它记录了当前活跃事务的 ID 列表。系统通过判断版本的 trx_id 是否在这个视图中，来决定是否返回该版本的数据。如果不符合条件，就顺着版本链往回找，直到找到符合可见性规则的版本。

在不同隔离级别下 Read View 的生成时机也不同：

如果是rc（读已提交）隔离级别，每一次执行快照读时生成`read view`，如果是rr（可重复读）隔离级别仅在事务中第一次执行快照读时生成`read view`，后续复用。

:::

>1.MVCC中获取trx_id的时候是否需要回表？

:::danger 答案

在MVCC（多版本并发控制）中，**获取** `trx_id` **通常不需要回表**，因为 `trx_id` 是在聚簇索引（或二级索引）的叶子节点中直接存储的，数据行本身就包含了这个信息。 

:::

### 7.MVCC能否解决幻读问题？

:::note

在InnoDB的可重复读（RR）隔离级别下：

**MVCC****在“快照读”，如SELECT的情况下，可以通过为事务创建一致性快照来解决****幻读****问题**。

它的原理就是：当一个事务开始时，MVCC会创建一个“快照”，记录当前的数据版本。在此期间，该事务只能看到快照创建时的数据，无论其他事务如何修改数据，都不会影响该事务的读取结果，从而避免了幻读。

**MVCC****在“当前读”，如SELECT...FOR UPDATE/INSERT/UPDATE/DELETE的情况下，由于需要锁定行，MVCC本身无法完全解决****幻读****。**

具体来说，在进行插入、更新或者删除等操作时，数据库会采用锁定机制来保证数据一致性。在这种情况下，**MVCC****虽然提供了非阻塞的读，但**并不能阻止其他事务执行插入新行或者删除行的操作，因此会发生幻读。

要解决“当前读”下的幻读问题，InnoDB通过**临键锁(next key lock)**，临键锁是记录锁+间**隙**锁，锁定查询范围内的所有可能插入点，从而阻止其他事务插入或删除行，解决幻读问题。

:::



### 8.不可重复读和幻读分别是如何解决的？

:::note

MVCC能解决不可重复读问题，MVCC给每一个事务提供一个数据快照，每个事务在读数据时看到的是数据的一个快照，这样即使有其他的事务对表中的数据做了修改，也查不到。这样就保证了**在整个事务期间，看到的数据一致**，从而避免了不可重复读。

解决幻读需要依靠锁机制，幻读是指在一个事务中，多次执行某个范围查询时，发现新插入的记录“突然出现”了。MVCC 是解决不了这个问题的，因为它控制的是对**已有数据的读取一致性**，但是对于「有没有新的数据插入进来」是无能为力的。

InnoDB 是通过锁机制来解决幻读问题。在RR（可重复读）隔离级别下，对于当前读，InnoDB 会对满足条件的数据加**行锁，**同时对条件范围内的空隙加**间隙锁（Gap Lock）。**

这样就能防止其他事务在这个范围内插入新数据，也就防止了幻读。

:::



## 优化

### 1.什么是慢 SQL？如何分析定位慢SQL？

:::note

MySQL 中有一个叫 `long_query_time` 的参数，原则上执行时间超过该参数值的 SQL 就是慢 SQL，会被记录到慢查询日志中。

可以通过在配置文件中设置`slow_query_log=1`开启慢查询日志，以及配置`long_query_time`参数记录执行时间超过多少秒的查询，之后我们可以通过分析慢查询日志找到慢SQL。

然后我们可以使用`explain` 关键字分析SQL的执行计划，看看有没有用索引，大部分情况下，慢 SQL 的原因都是因为没有用到索引。

最后，根据分析结果，通过添加索引、优化查询条件、减少返回字段等方式进行优化。

:::

### 4.你平时对SQL语句做了哪些优化呢？

:::note

- 查询时指明字段名称而不是使用`SELECT *`；
- 使用索引以及避免索引失效的写法；
- 聚合查询时使用`UNION ALL`代替`UNION`；
- 避免使用 JOIN 关联太多的表；
- 表关联时优先使用`INNER JOIN`，以及在必须使用`LEFT JOIN`或`RIGHT JOIN`时，确保小表作为驱动表。

:::

## 锁

### 1.MySQL 中有哪几种锁？

:::note

MySQL 中有多种类型的锁，可以从不同维度来分类：

- 按锁粒度划分的话，有**表锁**、**行锁**
- 按照加锁机制划分的话，有**乐观锁**和**悲观锁**
- 按照兼容性划分的话，有**共享锁（S锁）**和**排他锁（X锁）**

:::

### 2.表锁和行锁有什么区别？

:::note

- 表锁锁定粒度是**整张表**，行锁锁定粒度是满足条件的**若干行**；
- 表锁的并发度比较低，开销小，加锁速度快，死锁的风险小；行锁的并发度高但是开销大，加锁速度慢，死锁的风险高；
- MyISAM存储引擎支持表锁，InnoDB支持行锁；

:::

### 3.InnoDB 有哪几类行锁？

:::note

 InnoDB 支持三种行锁：**记录锁**、**间隙锁**和**临键锁**。

- **记录锁**：锁定单行记录；
- **间隙锁**：锁定一个范围，不包括记录本身；
- **临键锁**：结合记录锁和间隙锁，锁定一个范围，包含记录本身，主要目的是为了解决幻读问题。

:::

### 4.什么是共享锁和排他锁？

:::note

不论是表级锁还是行级锁，都存在共享锁（S锁/读锁）和排他锁（X锁/写锁）。

- 共享锁，允许多个事务同时读取数据，但阻塞写操作；
- 排他锁，独占数据，会阻塞其他事务的读写。

兼容性方面，排他锁与任何的锁都不兼容，共享锁仅和共享锁兼容。

:::

### 5.什么是乐观锁和悲观锁？

:::note

- **悲观锁**认为数据被外界访问时必然会产生冲突，因此在数据处理过程中全程加锁，保证同一时间只有一个线程可以访问数据。MySQL 中的行锁和表锁都是悲观锁。
- **乐观锁**认为并发操作不会总发生冲突，因此不会在读取数据时加锁，而是在提交更新时才检查数据是否被其他事务修改过。

:::

### 6.什么是意向锁？有什么用？

:::note

意向锁是一种表级锁，主要作用是**协调行锁和表锁的关系，避免为了判断表是否存在行锁而去全表扫描**。

意向锁分为：意向共享锁（IS锁）、意向排他锁（IX锁）。

- 意向共享锁（IS锁）：事务在请求S锁前，要先获得IS锁
- 意向排他锁（IX锁）：事务在请求X锁前，要先获得IX锁

当事务需要添加行锁时，会先在表上添加意向锁。这样当要添加表锁的时候，可以通过查看表上的意向锁，快速判断是否有冲突，而无需逐行检查，从而提高加锁效率。

意向锁是由InnoDB在操作数据之前自动加的，不需要用户干预。

:::

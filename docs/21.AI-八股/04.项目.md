### 1.如何实现的多格式文档解析

项目支持多种格式的简历上传，包括 pdf、docx、markdown、txt 等，我们可以根据不同的格式使用不同的开源库来去进行解析，比如 POI 来去解析`.docx`，使用 pdfbox 来解析 pdf，但是这样太麻烦。我们这里是使用的 Apache Tika 来去完成多格式解析的，Apache Tika 是一个成熟的开源“内容分析工具箱”，支持上千种不同格式的文件，它提供了统一的接口来提取文本和元数据。

这样我们可以使用统一的接口来去处理不同多种格式的文档。

在解析完之后，还会对提取到的文本进行数据清洗，保证进行 AI 的文本质量，去除各类噪声。

比如，移除控制字符、移除图片文件名行和图片链接、移除分隔符行。

以及格式上的规范化，比如：统一换行符、压缩连续空行、压缩多余空白字符、行首尾空格裁剪等。

还有就是针对简历的特殊清洗，比如： 联系方式格式化、邮箱地址规范化以及对一些像身份证信息进行脱敏。

### 2.Hash的文件去重是怎么做的？

用户在上传简历之后，会先校验文件的MIME类型和大小，之后会检查上传的简历是否已存在。

这里的文件去重并不是像传统的基于文件名进行判断是否重复，因为这种方法存在明显缺陷：

- 相同内容的文件可能有不同的文件名
- 不同内容的文件可能有相同的文件名

我们的系统采用基于**文件内容哈希**的去重策略，通过计算文件的唯一指纹来判断文件是否重复。

基于SHA-256 哈希算法基于文件的内容计算得到一个十六进制哈希字符串。

之后再根据哈希字符串去数据库查询是否已经有存在的记录，如果不存在，说明不是重复简历，将其保存到数据库中；如果存在，说明是重复简历，将简历的历史分析结果给返回。

对于我们这种需要消耗 AI token的功能，做好文件去重，可以避免 AI 的冗余调用，节省成本。

 

>1.为什么用 SHA-256而不是 MD5？

- MD5 已被证明存在碰撞漏洞，不再推荐用于安全场景；
- SHA-256 哈希碰撞概率极低，更安全可靠；
- 对于文件去重场景，SHA-256 的计算开销是可接受的。

>2.大文件上传怎么优化？

分片上传，将大文件切分成多个小块进行上传，支持断点续传。



### 3.文件上传的完整流程是怎么样的？

八步上传流程：

1.文件验证：非空、大小限制（如10MB）、业务类型提示（简历）。

2.类型识别：用解析服务检测MIME（不信后缀），拒绝不支持类型。

3.去重：基于SHA-256的内容哈希查重，命中则返回历史分析结果。

4. 文本解析：Tika 解析 +二次清洗，保证进入 AI 的文本质量。

5.对象存储:上传到 RustFS/S3,拿到 fileKey / fileUrl。

6. 落库:保存 Resume 记录,状态置为PENDING

7.异步分析：发送 Redis Stream 任务（避免同步 5~30 秒阻塞）。

8.返回结果：同步返回ID、状态、下载链接等。

### 4.为什么文件分析要异步处理？

为什么简历分析要异步处理？

AI 分析简历通常需要 5-30秒，同步处理会导致：

- 用户等待时间长，体验差；
- HTTP 连接可能会超时（默认 30s）
- 服务器线程阻塞，影响并发

因此，采用异步处理。





MQ 异步处理的时候，业务队列处理超时、拒绝，或者消费者消费失败，我们配置的会进行 3 次重试，如果超过重试次数，就拒绝消息进入死信队列。

在死信队列里面记录异常日志，以及根据失败原因分类处理，或者是发送邮件，让人工介入。



为什么要使用死信队列？好处？

| 场景         | 普通队列处理                  | 死信队列处理               |
| :----------- | :---------------------------- | :------------------------- |
| **消息过期** | 消息在队列中超过TTL           | 自动转到死信，记录超时原因 |
| **队列满**   | 新消息被拒绝                  | 旧消息转死信，避免丢失     |
| **消费拒绝** | 消费者明确拒绝(requeue=false) | 进入死信，由专门处理器处理 |
| **重试超限** | 多次重试仍然失败              | 转死信，避免无限循环       |

死信队列的好处：

```
// 1. 避免消息丢失
// 普通队列消息被拒绝且requeue=false -> 直接丢弃
// 死信队列 -> 消息被保存到死信队列，可以后续处理

// 2. 异常隔离
// 普通队列 -> 处理正常业务
// 死信队列 -> 专门处理异常，不影响正常业务流程

// 3. 问题追溯
// 死信消息保留了原始消息和死亡原因

// 4. 分级处理
```

### 你设计的会话状态机具体是怎么流转的？为什么需要这三个状态？

我们的会话状态有三种，分别是CREATED：表示已创建会话

IN_PROGRESS：表示进行中

COMPLETED：表示会话完成。

我们以这个模拟面试来说明，

在简历分析完之后，用户点击模拟面试，可以开始面试，如果不存在未完成的面试会话，那么这个时候就会创建新的面试会话，这个时候的状态就是CREATED，同时会生成一个sessionId。

会话创建完之后，会请求获取当前问题，这个时候会话的状态会被设置为IN_PROGRESS。

当用户做到最后一题并提交答案的时候，这个时候会话的状态会被设置为COMPLETED，表示会话完成。同时会触发异步任务生成面试评估报告。

### "在 IN_PROGRESS 状态下，如果用户中途退出或者网络中断，你们怎么处理？"

对于中途退出或网络中断的情况，我们采用了'断点续面'机制。

**首先，会话持久化。**
用户在面试过程中的每一轮问答、答题进度、上下文信息，都会实时保存到Redis缓存中。即使用户关闭浏览器或者网络中断，这些数据也不会丢失。

**其次，状态保持。**
当用户退出时，我们不会将会话状态改为 COMPLETED，而是保持 IN_PROGRESS 状态。下次用户再次进入模拟面试模块时，系统会先查询是否存在 IN_PROGRESS 状态的会话。如果存在，直接恢复上次的面试进度，从上次中断的地方继续。

**第三，超时兜底。**
为了防止僵尸会话占用资源，我们设置了30分钟的超时机制。如果用户中断后30分钟内没有回来，系统会自动将会话状态从 IN_PROGRESS 改为 COMPLETED，并生成一个'因超时而结束'的简单报告。这样既释放了资源，也给用户一个交代。"

### 面试官追问3：**"COMPLETED 后触发异步生成报告，你们怎么保证最终一定能生成成功？"**

**你可以这样回答：**

"这个问题问到点子上了。我们通过三个层次来保证异步任务的可靠性：

**第一层：消息队列持久化。**
当会话状态变为 COMPLETED 时，我们会向 RabbitMQ 发送一条生成报告的消息。消息队列本身支持持久化，即使生成报告的消费者服务宕机重启，消息也不会丢失。

**第二层：死信队列兜底。**
如果生成报告失败（比如AI服务超时），消息会进入死信队列。死信队列有专门的处理程序，会根据失败原因决定是重试、还是转人工处理。比如AI服务超时，我们会自动重试3次；如果是业务数据问题，会发送告警通知开发人员。

**第三层：状态回查补偿。**
我们还设计了一个定时任务，每天凌晨会扫描前一天状态为 COMPLETED 但报告未生成的会话。如果发现漏网之鱼，会触发补偿机制重新生成报告。这个兜底策略确保数据最终一致性。

通过这三层保障，我们基本上做到了报告生成成功率99.99%。"

### Query 重写是怎么做的？为什么要进行改写？

我们做 Query Rewrite 的核心目的，是把用户的原始问题先转换成更适合检索的表达，再去做向量召回。实际实现上，系统会先对问题做一次标准化，然后通过专门的重写提示词让模型在不改变核心意图的前提下，把过短、口语化、指代不清的问题补成单句、检索友好的查询；如果重写失败、结果为空，或者开关关闭，就直接回退原问题，保证主流程稳定。这个流程在 [KnowledgeBaseQueryService.java:315](vscode-file://vscode-app/Applications/Visual Studio Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html) 里实现，提示词在 [knowledgebase-query-rewrite.st:1](vscode-file://vscode-app/Applications/Visual Studio Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html)，开关在 [application.yml:86](vscode-file://vscode-app/Applications/Visual Studio Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html) 配置。

为什么要做改写，主要是因为真实用户的问题经常很短、很口语化，甚至带代词，比如“这个怎么弄”“它为什么报错”，这种表达和知识库里的正式表述不一致，直接检索容易召回偏弱。改写之后，查询词会更贴近文档语言，能显著提升召回率和首轮命中率；同时我们要求改写不能引入新事实，所以它是“检索优化”，不是“答案生成”。如果面试官继续追问，你可以补一句：Query Rewrite 是为了减少查询和文档之间的语义和词面差异，ReRank 则是在召回之后进一步提精度，两者分别解决 Recall 和 Precision 的问题。







### 令牌桶限流， 是怎么做的？

首先会定义一个`@RateLimit` 注解，然后定义属性，比如：限流维度、在指定时间窗口内允许的最大请求数（令牌总数）、时间窗口大小、时间单位等等。

之后会定义切面类，使用`@Around`环绕通知拦截所有加了`@RateLimit`注解的方法。

核心逻辑分为三步：

限流会根据限流维度动态生成 Redis key，然后调用 lua 脚本执行原子限流，最后根据结果决定是否放行。

Lua 脚本中的逻辑是这样的：**第一阶段：预检查**
遍历所有维度key，检查每个维度的当前令牌数是否足够。如果任何一个维度不足，直接返回0。

**第二阶段：扣减**
只有所有维度都通过后，才真正执行扣减操作。这种'先检查后执行'的方式保证了数据的一致性。

同时脚本还会**自动回收过期令牌**：通过ZSET记录每次请求，当令牌超过时间窗口后，自动回收并重新可用。"



### RAG你有做过哪些优化？

关于RAG的话，一开始我们就做了很多尝试，就比如切块是给它按500个token硬切，结果就经常把一段话给它懒腰斩断，检索出来的东西就前言不搭后语，后来改成按PDF段落或者markdown标题来切，还加了100-200个token的重叠，就让这个上下文连贯一点。然后做完这一步，准确率大概提升了15%。

然后呢，不止用了这个向量检索，还加了关键词检索，搞了个混合召回，就像人名啊，专业术语啊这种，光靠语义就很容易漏，单关键词一抓就很准，另外召回太多片段给大模型就比较贵，所以我们加了重排序模型，就比如Cohere的，把前二三十个结果再打一次分，只挑TOP3位给大模型，这样就又省钱又节省干扰。

还有一个问题就是，用户的问题经常太短，比如说就问了怎么用，然后我们就先用一个小模型，把问题扩写一下，生成几个变体，再去查，覆盖就会更全。最后就是复杂问题，我们会从FAQ还有用户手册、技术手册几个地方同时查，把结果汇总起来，避免信息不全，我们每个改动都做了AB测试，然后最终的用户满意度就从60%提高到85%。

几个关键的点：

1、切换+overlap听着好，但如果你的文档是扫描PDF，根本就没有段落结构，那你刚刚说的那一招就失效了。标准的做法是这样的：用语义边界检测动态切分，同时监控向量库存的一个增长，别为了那15%的准确率，就把成本翻倍了。

2、混合检索是不能简单的把向量分和关键词分加起来的，它们的数值范围是不一样的，所以直接加它的方向就会偏，那工业级的解决方案，会用一个轻量级的排序模型，比如LambdaMART，用这个模型统一的去进行打分，而不是拍脑袋去调权重。

3、重排序模型，像Cohere一次要200毫秒，那如果用户同时来100个请求，那延迟是不是就直接就炸了，所以线上得要做策略，简单的问题直接用向量TOP5，只有复杂问题的时候才开rerank。

4、Query改写很容易改歪，比如把退款改成怎么样赚钱，反而会引入了噪声，那正确的做法就是改完之后，用向量算一下和原问题的相似度，低于0.8就直接舍弃。

最后，你说满意度从60%到了85%，这个数字得要拆开看，真正要看的是用户是不是真的解决问题了，有没有幻觉，是不是第一轮就回答对了，这些其实才是核心的指标。



### 混合检索是怎么做的？

我们这里的“混合检索”不是传统意义上简单的 BM25 + 向量双通道硬拼，而是一个更偏工程化的多阶段检索链路：先做 Query Rewrite，把用户原始问题改写成更适合知识库召回的查询；然后同时保留“改写后的 query”和“原始 query”作为候选入口去做向量召回；召回阶段再按问题长度动态调整 topK 和 minScore，短问题放宽召回、长问题收紧召回；如果开启 ReRank，就先放大候选集，再用模型精排，最后只把最相关的前 N 条上下文送给大模型。核心目标是同时提升召回率和精度，减少短句、口语化、歧义问题带来的漏召回和噪声命中。实现上可以看 [KnowledgeBaseQueryService.java:254](vscode-file://vscode-app/Applications/Visual Studio Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html) 和 [KnowledgeBaseVectorService.java:88](vscode-file://vscode-app/Applications/Visual Studio Code.app/Contents/Resources/app/out/vs/code/electron-browser/workbench/workbench.html)。

我们这样设计是因为 RAG 里最常见的问题不是“模型不会答”，而是“检索没把对的材料找回来”。Query Rewrite 解决的是查询表达和文档表达不一致的问题，动态召回解决的是不同长度问题的分布差异，ReRank 解决的是候选集里噪声太多的问题。三层一起做，本质上就是把 Recall 和 Precision 分开优化，而且每一层都有失败回退，不会影响主流程稳定性。



### Rerank怎么做的？

可以这样跟面试官说，核心要抓住一句话：Rerank 是 RAG 里的第二阶段精排，用来在召回出的候选结果里重新按“问题相关性”排序，提升最终送进大模型的上下文质量。

你可以直接用这个版本回答：

我们通常把检索分成两步：第一步是召回，目标是尽量多捞一些可能相关的候选；第二步是 Rerank，目标是在这批候选里再做更精细的语义匹配，把最相关的排到前面。因为向量召回更擅长“找得到”，但对细粒度语义区分不够强，比如同一主题下多个段落都相似，召回阶段很容易把噪声也带进来。Rerank 一般会用更强但更慢的模型，比如 cross-encoder 或基于大模型的打分方式，把 query 和每个候选逐一做相关性判断，最后只保留前 N 条给生成模型。这样做的价值是提高最终答案的准确性，减少无关上下文干扰，同时还能和召回解耦，方便做性能和效果上的平衡。

如果你想讲得更像“有实战经验”的表达，可以再补这几句：

1. Rerank 不是替代召回，而是在召回基础上做精排，所以一般会先扩大候选集，再排序截断。
2. 它特别适合长文档、知识库问答、结果相似度很接近的场景。
3. 代价是延迟和成本更高，所以线上通常会控制候选数量、超时、降级策略，保证主流程稳定。
4. 如果效果优先于时延，Rerank 很值得加；如果是极低延迟场景，就要谨慎控制它的使用范围。

面试时你还可以用一个更短的版本：

Rerank 的作用是把“召回到的候选”重新做一次语义精排，解决向量召回只能粗筛、不能精排的问题。它通常放在召回之后，先拿一批候选，再用更强模型判断 query 和文档的真实相关性，最后选前几条进入生成阶段。这样能明显提升上下文质量和回答准确率，代价是增加一点延迟，所以工程上一般会配合候选集放大、超时和降级策略一起用。

	养成习惯：每次生成完，花两分钟扫一遍关键逻辑。这两分钟能帮你省两小时 debug
	
6. 善用 MCP 和 Skills 扩展能力边界
	把你项目的 best practice 封装成 MCP 或 Skills，Opus 调用时就不会凭空瞎写
	这也是 ShipSwift 在做的事——不是教你写代码，是教 AI 写代码，db 设计，数据安全，auth，payment 等生产级模块全覆盖
	
7. 重置上下文的时机很关键
	如果你发现 Opus 开始输出不相关的内容，或者忘记了之前的约束，不要硬撑，果断重置，一个干净的上下文比一个冗长的对话有效 10 倍

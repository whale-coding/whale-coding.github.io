## Transformer

### Transformer的基本原理？



Transformer是一种完全基于Attention机制的序列建模架构，用来替代传统的RNN和CNN，主要解决了无法并行和长距离依赖建模困难的问题。



其核心是Self-Attention机制：输入序列中的每个位置会映射为Query、Key、Value，通过

softmax(QKᵀ / √dk)计算注意力权重，并对Value加权求和，从而实现任意位置之间的信息交互。



相比RNN需要逐步传递信息，Self-Attention使任意两个位置之间的信息传递路径长度为O(1)，更容易建模长距离依赖。



在此基础上，Transformer引入Multi-Head Attention，通过多个头并行学习不同子空间的语义关系，从而增强表达能力。



模型整体采用Encoder-Decoder结构：



- Encoder由Self-Attention和前馈网络组成
- Decoder包含Masked Self-Attention（保证自回归）以及Cross-Attention（与Encoder交互）





此外，由于模型本身不具备位置信息，引入了Positional Encoding（如sin/cos函数）来编码序列顺序。



工程上，每一层还包括残差连接和LayerNorm来稳定训练。



相比RNN，Transformer可以高效并行计算，并且更擅长捕捉长距离依赖，但其计算复杂度为O(n²)，在长序列场景下成本较高。

:::note









**简单来说，Transformer 是用 Attention 机制完全取代了 RNN/CNN 的序列建模架构**，解决了传统循环网络无法并行、长距离依赖衰减的问题。

Transformer 的整体结构**由 Encoder 和 Decoder 两个部分组成**，Encoder对输入序列编码，Decoder逐步生成输出 。

核心组件有三部分：

- **Self-Attention（自注意力）**。输入序列每个位置都会生成 **Q、K、V** 三个向量。每个词通过计算自己的 Q 与其他所有位置的 K 的相似度，得到注意力权重，再对所有位置的 V 加权求和。这样**每个词都能直接看到全局上下文**，距离再远也是 O(1)。
- **Multi-Head Attention（多头注意力）**。本质上是把Self-Attention过程**并行做多次**，每次用不同的投影矩阵，相当于从**不同语义子空间**观察关系，最后拼接起来。
- **位置编码（Positional Encoding）**。因为去掉了 RNN 的时序计算，模型本身不知道词的顺序，所以要加上正弦/余弦的位置编码注入位置信息。

Decoder 和 Encoder 很像，但有两个区别：

一是用了 **Masked Self-Attention**，保证预测第 t 个词时只能看到前面的 1 到 t-1 个词（自回归特性）；

二是中间有一个 Cross-Attention 层，让 Decoder 去关注 Encoder 的输出。

最后每层都有 **残差连接和 Layer Norm**，以及简单的 Feed-Forward 网络做非线性变换。

:::

### 什么是self-attention?

:::note

`self-attention`自注意力机制是一种**让序列中每个位置与序列中所有其他位置建立关联，并根据重要性动态加权的信息融合机制**。

在 Self-Attention 中，每个输入向量都会映射成`Q、K、V`三个向量，然后通过公式计算注意力：
$$
Attention(Q, K, V) = softmax(\frac{QK^T}{\sqrt{d_k}}) V
$$
:::

### GPT和BERT的区别？

:::note

- GPT是基于Transformer Decoder的自回归模型，通过Masked Self-Attention逐步预测下一个token，属于生成式模型，适合文本生成任务。
- BERT是基于Transformer Encoder的双向模型，通过Masked Language Model训练，可以同时利用上下文信息，更适合理解类任务，比如文本分类和命名实体识别。

:::





我们通过聊天框发送一条消息给AI模型，，然后AI模型生成一个回复。我们发的消息就叫User Prompt 也就是用户提示词，一般就是我们提出的问题或者想说的话。

但是现实生活中，当我们和不同人聊天时，即便是完全相同的话，对方也会根据自己的经验给出不同的答案。

比如，我说我肚子疼

我妈可能会问我要不要去医院看看

我爸可能会让我去厕所

我女朋友可能会说一句：滚一边去，老娘也疼

但是AI并没有这样的人设，所以它只能给出一个通用的四平八稳的回答：请就医。

显得非常无趣，因此我们希望给AI也加上人设，最直接的方法就是把人设信息和用户要说的话，打包成一条User Prompt发过去，比如你扮演我的女朋友，我说我肚子疼，然后AI就可能回复，滚一边去，老娘也疼。

但问题是你扮演我温柔的女朋友，这句话并不是我们真正想说的内容，于是人们干脆把人设信息单独拎出来，放到另外一个Prompt里面，这就是System Prompt 系统提示词。

System Prompt 主要用来描述AI的角色、性格、背景信息、语气等等。总之只要不是用户直接说出来的内容，都可以放进System Prompt 里面。每次用户发送User Prompt 的时候，系统会自动把System Prompt也一起发送给AI模型。

这样整个对话就显得更加自然了，

AI只是一个聊天机器人，你问一个问题，它最多给你答案或者告诉你怎么做，但实际动手的还是你自己，那么能不能让AI自己去完成任务呢？

比如我们想让AI自动帮我们管理电脑的文件，我们可以先写好一些文件的管理函数，比如说list_file用来列目录、read_file用来读文件，然后把这些函数以及它们的功能描述、使用方法注册到AI agent中，AI Agent会根据这些信息生成一个System Prompt ，告诉AI模型用户给了你哪些工具，它们都是干什么的，以及AI如果想要使用它们，应该返回什么样的格式，最后把这个System Prompt 连同用户的请求，比如说帮我找一找原神的安装目录一起发给AI，如果AI模型足够聪明，就会按照要求的格式，返回一个调用某个函数的信息，AI agent进行解析之后，就可以调用对应的函数了，然后再把结果丢回给AI，AI再根据函数调用的结果决定下一步应该做什么操作。

提供给AI调用的函数或者服务，就叫做Agent Tools

Function Calling：统一格式、规范描述，将每个Tool都用一个Json对象来定义。

然后这些对象也从System Prompt中被剥离了出来，单独放到了一个字段里面，

Function Calling也规定了AI使用工具时应该返回的格式。

AI Agent如何与Tools进行通信的，MCP是一个通信协议，专门用来规范Agent和Tool服务之间是怎么交互的，运行Tool的服务叫做MCP Server，调用它的Agent叫做MCP Client。

MCP规定了MCP Server如何和MCP Client通信，以及MCP Server要提供哪些接口

## Agent

### 1.什么是AI Agent ？

智能体（Agent）是一种能够自主规划、决策、执行任务的组件，核心是让大语言模型（LLM）根据任务需求，选择并调用工具，完成单靠模型自身无法解决的复杂问题。

- 没有Agent时，LLM只能基于自身训练数据回答问题，遇到需要实时数据、复杂计算、外部工具调用的场景就会卡壳。
- 有了Agent后，LLM就像一个“指挥官”，能思考任务步骤->选择合适工具->执行工具调用->根据结果调整策略，直到完成任务。

实现一个Agent有三大核心点：

- 工具
- 记忆
- 规划

当用户需求提交给Agent智能体后，Agent先做规划，思考完成这个需求，应该做什么事情；规划好之后，怎么去做呢，由于大语言模型只能回答问题，没有行动力，我们可以给Agent提供工具，这就相当于给大模型提供了行动力；在整个流程里面，还会保存短期的记忆，大模型也会根据前面保存的短期记忆从而去动态的调整后续的工具的调用。最终完成用户的需求。

核心特点：

- 目标驱动：围绕用户的具体任务目标展开工作。
- 工具调用能力：能连接外部工具，弥补LLM的局限性。
- 自主决策与迭代：不需要人工干预，能根据工具返回的结果，判断是否需要继续调用工具，或直接生成最终答案。

###  什么是ReAct？

ReAct的全称是Reasoning+Acting（推理+行动），是让Agent像人类一样【思考问题->制定策略->执行行动->验证结果】的关键逻辑。

简单来说：ReAct让Agent不再是“直接回答问题”，而是通过“自然语言思考过程”指导工具调用，一步步解决复杂问题，完美适配需要多步推理、工具协作的场景。

## Skills

### 什么是Skills？

Skills（技能）指的是供Agent调用的可复用能力模块,简单来说就是给Agent用的技能。

我们可以将"业务逻辑、SOP、工程最佳实践"进行模块化封装，封装成一个可复用的技能包。使Agent在任务执行过程中能够按需调用这些能力。

从本质上看，Skills扩展了大模型的能力边界，使模型不仅能进行语言理解和生成，还能够执行实际操作。

在实现上，Skills通常通过Function Calling等机制接入，由模型根据任务动态选择并调用。

Skills有三个特点：

- 按需加载（渐进式上下文披露）：Skills采用**延迟加载、按需激活**的机制，不会一次性全量注入上下文。Agent仅在任务匹配时才加载对应技能内容，有效降低Token开销，减少上下文干扰，提升模型决策效率。
- 即插即用：它是轻量级的技能插件，可以在不同的Agent之间复用，无需修改Agent的核心代码。
- 封装经验：Skills不仅是工具调用的封装，更重要的是**将专家经验、执行范式与完整解决方案结构化**。它让Agent掌握“如何系统性完成任务”，实现**工具使用**到**流程执行**的能力升级。

### Skills相比于MCP有什么优势？

Skills和MCP其实不在同一个层级。

Skills是面向Agent的能力抽象，强调“能做什么”；而MCP是一种工具接入协议，解决“怎么接入工具”。

相比MCP，Skills的优势主要在于：

第一，它的抽象层更高，不仅封装接口，还包含业务逻辑和使用时机；

第二，它更贴近业务，可以将SOP或多步流程封装成一个能力；

第三，Skills支持组合和编排，可以构建复杂任务；

第四，在Agent系统中，Skills可以作为动作空间，便于模型做决策。

而MCP的优势在于标准化和解耦，适合做底层工具接入。

可以理解为：MCP负责“连接工具”，而Skills负责“组织能力”。



### Skills和MCP的区别？

- MCP是**Agent与外部系统的标准化交互接口层**，它负责帮AI连接外部世界，解决的是Agent去哪拿数据、能调用哪些系统接口的问题，比如让Agent连接数据库、Github、飞书等这些外部工具。
- Skills是**行为规范层**，它负责帮AI固化专业流程，解决的是Agent如何正确执行任务的问题，教Agent按特定步骤、格式、标准完成工作。
- MCP解决怎么连、连得上：让Agent能够调用数据库、API、文件系统、第三方服务，聚焦连接能力。
- Skills解决怎么做、做的对：让Agent按步骤、按规则、按标准完成任务，聚焦执行能力。
- MCP可以理解为一个接口，提供标准化调用方式，不关系业务逻辑；Skills是一个解决问题的方案。







### skills的文件结构？

https://cloud.tencent.com/developer/article/2616510



我们可以将“业务逻辑、SOP、工程最佳实践与可执行脚本”统一封装在结构化目录中，通常以`SKILL.md`作为核心入口文件。

一般来说，一个完整的Skill，包含以下文件：

```apl
my-skill/  # skills的名字
 |———— SKILL.md  # 概览与导航（必须）
 |———— reference.md  # 参考文档（可选）
 |———— examples.md  # 示例（可选）
 |———— scripts/  # 脚本目录（可选）
 			   |———— helper.py  # python脚本	  
```

**重要提醒：**

**1、文件夹名称必须是小写字母+连字符，例如 hotspot-collector（不能有空格、大写）。**

**2、SKILL.md是唯一必需的，其他都是可选的。**

SKILL.md 是核心文件，它的结构是固定的分为两部分**：**

**1、YAML头部**（必需）：用---包裹，包含 name和description字段，这是 OpenCode用来识别 Skill 的名片。

**2、Markdown 主体**（必需）：详细的工作流程，输出格式要求，示例等。

```yaml
---
name: 你的skill名称
description: 简要描述该技能的功能以及何时该使用它
---
# 你的技能名称
## 指令 (Instructions)
为 OpenCode提供清晰、逐步的操作指南。
## 示例 (Examples)
展示使用该技能的具体代码或操作案例。
```

最最核心的，其实就是description这个字段了，就是描述Agent会在何时如何调用你这个skills。

这块一定要注意，别把一些Prompt的坏习惯带过来，一定要始终使用第三人称。

因为描述会被注入到系统提示中，不一致的视角会导致发现问题。

优秀的："处理Excel文件并生成报告"

不太行的："我可以帮助你处理Excel文件"

不太行的："你可以使用这个来处理Excel文件"

且，尽量包含你的触发关键词，同时整个将SKILL.md的正文，一定要保持在500行以内，这样效果菜最好。

## Harness Engineering

### 什么是Harness Engineering？

**Harness Engineering（约束工程）**简单说，这是一套用机械化的约束和反馈循环，让 AI Agent 的输出变得**可控、可审查、可迭代**的方法论。

所谓Harness Engineering，是通过强制性的工程约束和标准化的输入输出，将 AI 的不可预测性，收敛为确定性的交付。
